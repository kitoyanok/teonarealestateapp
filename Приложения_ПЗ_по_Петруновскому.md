# Приложение А (обязательное)

# Техническое задание. Требования к программным модулям

Приложение А в логике Петруновского является не полным ГОСТ-документом на десятки страниц, а компактным приложением с прикладными требованиями к разрабатываемым модулям. Для проекта `Тэона` его следует делать в следующем составе.

## А.1 Назначение разработки

Разрабатываемые программные модули предназначены для автоматизации работы риелтора при сопровождении клиентских заявок, связанных с покупкой недвижимости в городе Краснодаре. Основным пользователем системы является риелтор, который формирует карточку клиента, задает параметры поиска, инициирует автоматический поиск объектов, просматривает найденные объявления и формирует подборку для клиента.

Назначение разработки заключается в создании набора программных модулей, позволяющих сопровождать клиентскую заявку на основных этапах ее жизненного цикла: от ввода исходных параметров клиента до подготовки текста сообщения с подборкой подходящих объектов недвижимости.

## А.2 Требования к функциональным характеристикам

В состав программной системы должны входить следующие модули:

1. модуль авторизации риелтора;
2. модуль ведения карточек клиентов;
3. модуль хранения параметров поиска недвижимости;
4. модуль автоматического поиска объектов по внешним источникам;
5. модуль нормализации и фильтрации найденных объявлений;
6. модуль расчета процента соответствия объекта требованиям клиента;
7. модуль формирования подборки объектов;
8. модуль формирования текста сообщения;
9. модуль сводной аналитики по текущим заявкам;
10. модуль хранения результатов поиска и истории запусков.

Разрабатываемые модули должны обеспечивать возможность:

- входа в систему под преднастроенной учетной записью риелтора;
- создания новой карточки клиента;
- редактирования данных клиента;
- сохранения профиля поиска;
- автоматического запуска поиска после сохранения клиента;
- повторного запуска поиска по существующей карточке;
- отображения найденных объектов с краткими характеристиками;
- просмотра подробной карточки объекта;
- ручного добавления объекта в подборку;
- удаления объекта из подборки;
- формирования текста сообщения на основе подборки;
- фиксации факта отправки подборки клиенту;
- просмотра сводных показателей по активным заявкам.

## А.3 Требования к входным данным

Входными данными системы являются:

- логин и пароль риелтора;
- фамилия, имя и отчество клиента;
- контактный телефон клиента;
- электронная почта клиента;
- канал связи с клиентом;
- тип объекта недвижимости;
- бюджет;
- диапазон площади;
- диапазон комнатности;
- районы города или населенные пункты;
- срок сдачи объекта;
- тип отделки;
- параметры дома и участка;
- дополнительные текстовые комментарии риелтора.

Для модуля поиска дополнительными входными данными являются HTML-страницы и JSON-LD-блоки внешних сайтов недвижимости.

## А.4 Требования к выходным данным

Выходными данными системы являются:

- карточка клиента;
- сохраненный профиль поиска;
- список найденных объектов;
- краткое нормализованное описание объекта;
- процент совпадения объекта с параметрами клиента;
- причины совпадения и несоответствия;
- подборка объектов;
- текст сообщения для клиента;
- записи базы данных о результатах поиска;
- сводные показатели dashboard.

## А.5 Требования к надежности

Система должна обеспечивать сохранность данных после перезапуска. Для этого база данных PostgreSQL должна использовать постоянное хранилище через Docker volume.

При ошибке работы поискового сервиса система должна:

- сохранить карточку клиента;
- зафиксировать ошибку в журнале запусков;
- перевести заявку в состояние `error`;
- предоставить возможность повторного запуска поиска.

При отсутствии результатов поиска система должна:

- завершать поиск корректно;
- сохранять информацию о запуске;
- переводить карточку клиента в состояние `no_results`.

## А.6 Ограничения разработки

Поисковый модуль ориентирован только на объекты недвижимости города Краснодара. Объекты из других городов Краснодарского края не должны включаться в результирующую выдачу.

Добавление новых риелторов через интерфейс в текущей версии не реализуется. Работа выполняется только с заранее созданными учетными записями.

Отправка сообщений клиенту выполняется вне системы. Система формирует только текст подборки и позволяет зафиксировать факт отправки.

## А.7 Требования к пользовательскому интерфейсу

Пользовательский интерфейс должен быть реализован как веб-приложение и содержать:

- страницу входа;
- главную страницу со сводными показателями;
- страницу списка клиентов;
- форму создания клиента;
- карточку клиента;
- детальную карточку объекта недвижимости.

Интерфейс должен обеспечивать удобную навигацию, наглядное отображение состояния клиентской заявки и быстрый переход между основными действиями.

# Приложение Б (обязательное)

# Руководство по стилю

В референсе Петруновского приложение Б представляет собой не теоретический раздел, а прикладное описание визуальных правил интерфейса. Для проекта `Тэона` приложение Б следует делать как компактный style guide.

## Б.1 Общая визуальная концепция

Приложение `Тэона` должно быть выполнено в сдержанном деловом стиле. Интерфейс ориентирован на практическую ежедневную работу риелтора, поэтому визуальная подача должна быть спокойной, строгой и функциональной. В оформлении не допускается использование декоративных иллюстраций, ярких фоновых эффектов, сложных градиентных композиций и перегруженных карточных сеток.

Основными принципами визуальной организации интерфейса являются:

- простота композиции;
- ясная типографическая иерархия;
- минимализм в оформлении;
- акцент на читаемости данных;
- визуальное выделение основных действий.

## Б.2 Цветовая палитра

Основные цвета интерфейса должны соответствовать фактической реализации frontend:

- основной цвет текста — `#3d3d3d`;
- дополнительный серый текст — `#898989`;
- основной акцентный цвет — `#fd6000`;
- цвет акцентного состояния при наведении — `#e95700`;
- мягкий акцентный фон — `#fff1e8`;
- цвет границы акцентных элементов — `#ffd2b8`;
- основной фон страницы — `#fafafa`;
- фоновый цвет мягких поверхностей — `#f1f1f1`;
- основной цвет карточек и модальных окон — `#ffffff`;
- цвет стандартной границы — `#e7e7e7`.

Если в приложении используются статусные метки, рекомендуется применять:

- зеленый фон для успешных состояний;
- желтый фон для предупреждений;
- красный фон для состояний ошибки.

## Б.3 Типографика

Основной шрифт интерфейса — `Inter`.  
Допускается использование системного fallback-набора:

- `system-ui`;
- `-apple-system`;
- `BlinkMacSystemFont`;
- `Segoe UI`;
- `sans-serif`.

Основные требования к типографике:

- основной текст интерфейса — 14 px;
- крупные заголовки экранов — 32–40 px;
- подзаголовки разделов — 20–24 px;
- подписи и второстепенные пояснения — 12–14 px;
- межбуквенный интервал не уменьшается;
- перегруженные жирные начертания не используются без необходимости.

## Б.4 Оформление кнопок

Основное действие должно выделяться акцентной кнопкой оранжевого цвета. Для кнопок основного действия используются следующие правила:

- фон по умолчанию — `#fd6000`;
- цвет текста — белый;
- радиус скругления — 10 px;
- при наведении фон становится белым;
- при наведении появляется оранжевая обводка;
- второстепенные кнопки имеют белый фон и светло-серую границу.

В интерфейсе должны использоваться:

- кнопки основного действия;
- второстепенные кнопки;
- строковые кнопки внутри карточек объектов;
- кнопки удаления или отмены.

## Б.5 Оформление полей ввода

Поля ввода и списки выбора должны иметь:

- белый или очень светлый фон;
- тонкую серую границу;
- радиус скругления 10 px;
- внутренний отступ не менее 11–13 px;
- единый размер текста с остальным интерфейсом.

При ошибке ввода поле должно визуально отличаться от стандартного состояния и сопровождаться сообщением об ошибке.

## Б.6 Карточки и панели

Карточки и панели интерфейса должны использовать белый фон и четкую, но ненавязчивую границу. Карточки объектов недвижимости должны содержать:

- краткое название объекта;
- основные характеристики;
- цену;
- процент совпадения;
- действия по просмотру и добавлению в подборку.

Карточки клиента и панели подборки должны иметь логичное разделение на блоки:

- сведения о клиенте;
- параметры поиска;
- найденные объекты;
- подборка;
- история поисковых запусков.

## Б.7 Графики и обзорные блоки

Dashboard должен использовать простые обзорные плашки и линейные графики без декоративной перегрузки. Цвет графиков должен быть согласован с основной акцентной палитрой интерфейса. Подписи к графикам и KPI должны быть читаемыми и короткими.

## Б.8 Визуальные материалы приложения Б

Если приложение Б оформляется как у Петруновского с рисунками, рекомендуется использовать следующие подписи:

- `Рисунок Б.1 – Логотип системы «Тэона»`;
- `Рисунок Б.2 – Основная цветовая палитра интерфейса`;
- `Рисунок Б.3 – Основные состояния кнопок интерфейса`;
- `Рисунок Б.4 – Пример карточки объекта недвижимости`;
- `Рисунок Б.5 – Пример формы ввода данных клиента`;

# Программный код модулей

## В.1 Общий принцип отбора кода

В приложение В не следует помещать:

- весь проект целиком;
- повторяющийся boilerplate;
- служебные типы без прикладной ценности;
- второстепенные стили и мелкие утилиты.

В приложение В должны попадать только те фрагменты, которые раскрывают логику системы.

В отличие от полного исходного кода проекта, приложение содержит только те листинги, которые демонстрируют обработку клиентской заявки, работу авторизации, запуск поиска, фильтрацию и нормализацию объявлений, расчет процента совпадения, формирование подборки и подготовку сообщения для клиента.

# Руководство пользователя. Выполнение программы и сообщения пользователю

По референсу Петруновского приложение Д следует делать не как общий абстрактный “user guide”, а как компактное руководство, в котором есть две главные части:

2. сообщения пользователю.

Именно такую структуру рекомендуется сохранить для проекта `Тэона`.

## Д.1 Выполнение программы

Для начала работы пользователь запускает приложение `Тэона` через браузер после предварительного разворачивания системы. После открытия адреса `http://localhost:5002` на экране отображается страница входа.

Для входа в систему пользователь вводит логин и пароль одной из преднастроенных учетных записей риелторов. После успешной авторизации открывается главная страница со сводными показателями, включающими число клиентов в работе, количество найденных объектов, число объектов в подборках и количество подборок, готовых к отправке.

Для создания новой клиентской заявки пользователь переходит в раздел добавления клиента. В форме последовательно вводятся персональные данные клиента, канал связи и параметры поиска недвижимости. После заполнения обязательных полей пользователь выполняет сохранение карточки клиента.

После сохранения карточки система автоматически запускает поиск объектов недвижимости. В процессе поиска выполняются обращение к внешним источникам, фильтрация объектов по городу Краснодару, нормализация объявлений, расчет процента совпадения и сохранение результатов в базе данных.

После завершения поиска открывается карточка клиента. В карточке клиента отображаются:

- основные сведения о клиенте;
- параметры поискового профиля;
- найденные объекты недвижимости;
- текущая подборка;
- история запусков поиска.

Для просмотра подробной информации по объекту пользователь открывает детальную карточку объявления. В ней доступны цена, площадь, комнатность или характеристики дома, район, описание и ссылка на источник.

Для формирования подборки пользователь вручную добавляет в нее подходящие объекты из общего списка результатов. После этого в блоке подборки становится доступной функция генерации текста сообщения.

После формирования текста пользователь копирует его и отправляет клиенту через внешний канал связи. После фактической отправки пользователь выполняет подтверждение отправки в интерфейсе, что изменяет состояние клиентской заявки.

Перед завершением работы пользователь убеждается, что необходимые изменения сохранены, после чего закрывает вкладку браузера или завершает работу с системой стандартными средствами операционной системы.

# Приложение Е (обязательное)

# Руководство программиста. Характеристика программы

По референсу Петруновского приложение Е должно дополнять пользовательское руководство и раскрывать программу уже с точки зрения разработчика или специалиста, который будет разбираться в структуре решения. Для `Тэоны` приложение Е рекомендуется делать в следующем составе.

## Е.1 Характеристика программы

Программа `Тэона` представляет собой многомодульное веб-приложение, предназначенное для автоматизации работы риелтора при сопровождении заявок на покупку недвижимости. Архитектурно программный продукт состоит из клиентской части, серверной части, отдельного поискового сервиса и реляционной базы данных.

Клиентская часть реализована на `React + TypeScript + Vite` и отвечает за отображение пользовательского интерфейса. Серверная часть реализована на `Node.js + Express + TypeScript` и выполняет функции backend API. Поисковый сервис реализован на `Python + FastAPI` и используется для обхода внешних сайтов, нормализации объявлений и расчета процента совпадения. Хранение данных выполняется в `PostgreSQL`.

## Е.2 Состав модулей программы

В состав программы входят следующие основные модули:

1. web-интерфейс риелтора;
2. модуль авторизации;
3. модуль ведения клиентов;
4. модуль хранения параметров поиска;
5. модуль запуска поиска;
6. модуль нормализации и фильтрации объектов;
7. модуль расчета процента совпадения;
8. модуль формирования подборки;
9. модуль генерации текста сообщения;
10. модуль работы с базой данных;
11. модуль контейнерного развертывания.

## Е.3 Структура каталогов проекта

Программисту рекомендуется ориентироваться на следующую структуру каталогов:

- `apps/web` — frontend;
- `apps/api` — backend API;
- `apps/search-service` — Python-сервис поиска;
- `database` — SQL-инициализация базы данных;
- `deploy` — вспомогательные материалы развертывания;
- `scripts` — утилиты разработки;
- `docker-compose.yml` — общая контейнерная конфигурация.

## Е.4 Порядок запуска программы

Для запуска программы в production-режиме следует выполнить:

Действие №1. Создать файл `.env` на основе `.env.example`.  
Действие №2. Выполнить `npm install`.  
Действие №3. Выполнить `npm run prod:up`.  
Действие №4. Дождаться запуска сервисов `postgres`, `search-service`, `api`, `web`.

Для остановки программы используется команда:

`npm run prod:down`

Для просмотра логов:

`npm run prod:logs`

Для полного сброса базы данных:

`npm run prod:reset-db`

## Е.5 Взаимодействие модулей

В ходе работы программы взаимодействие модулей организовано следующим образом:

1. пользователь работает с web-клиентом;
2. web-клиент обращается к backend API;
3. backend API сохраняет данные клиента в PostgreSQL;
4. backend API передает профиль поиска в search-service;
5. search-service выполняет обход внешних сайтов;
6. search-service возвращает нормализованные результаты;
7. backend API сохраняет найденные объекты и результаты сопоставления;
8. web-клиент получает готовые данные и отображает их пользователю.

## Е.6 Характеристика базы данных

База данных `PostgreSQL` содержит следующие основные таблицы:

- `users`;
- `clients`;
- `client_search_profiles`;
- `properties`;
- `client_found_properties`;
- `shortlist_items`;
- `share_messages`;
- `search_runs`.

Указанные таблицы обеспечивают хранение учетных записей риелторов, клиентских заявок, параметров поиска, найденных объектов, составов подборок, сообщений и истории поисковых запусков.

## Е.7 Основные алгоритмы программы

Наиболее значимыми алгоритмами программы являются:

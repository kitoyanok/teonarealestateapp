## Введение

В рамках дипломного проекта разрабатывается веб-приложение «Тэона» — рабочее пространство риелтора для подбора недвижимости клиентам. Приложение позволяет вести список клиентов, фиксировать параметры поиска, получать подходящие объекты из открытых источников, формировать подборку и копировать готовый текст для отправки клиенту.

Актуальность разработки связана с тем, что риелтору необходимо быстро обрабатывать запросы клиентов и сравнивать предложения из разных источников. Ручной поиск объектов занимает значительное время, а использование единого интерфейса позволяет сократить количество повторяющихся действий и повысить удобство работы.

Целью проекта является создание прикладного веб-приложения, автоматизирующего часть рутинных операций риелтора: ввод параметров клиента, поиск объектов, отбор подходящих вариантов и подготовку подборки для отправки.

Для достижения поставленной цели необходимо решить следующие задачи:

- спроектировать архитектуру приложения;
- реализовать интерфейс рабочего места риелтора;
- реализовать backend API для хранения клиентов и результатов работы;
- реализовать сервис поиска по открытым источникам;
- организовать хранение данных в PostgreSQL;
- подготовить развертывание приложения в Docker;
- провести тестирование основных пользовательских сценариев.

## 1. Техническое задание

### 1.1 Основание для разработки

Основанием для разработки является необходимость создания веб-приложения для сопровождения процесса подбора недвижимости клиентам риелтора.

### 1.2 Назначение разработки

Приложение предназначено для автоматизации работы риелтора при обработке клиентских заявок на подбор квартир и домов.

### 1.3 Требования к функциональным характеристикам

Разрабатываемое приложение должно обеспечивать:

- авторизацию пользователя;
- создание и хранение клиентов;
- ввод параметров поиска недвижимости;
- выбор типа недвижимости: квартира или дом;
- поиск объектов по заданным параметрам;
- отображение найденных объектов;
- просмотр карточки объекта;
- добавление объектов в подборку клиента;
- формирование текста подборки;
- копирование телефона клиента и текста сообщения;
- хранение данных в PostgreSQL.

### 1.4 Требования к надежности

Система должна сохранять данные клиентов и результатов поиска после перезапуска приложения. При отказе отдельного источника поиска остальные источники должны продолжать работу.

### 1.5 Требования к составу и параметрам технических средств

Для запуска системы требуется персональный компьютер или сервер с установленным Docker и поддержкой контейнерного запуска приложения. Для локальной проверки базы данных может использоваться клиент PostgreSQL, например TablePlus.

### 1.6 Требования к информационной и программной совместимости

Система должна работать с PostgreSQL, современными веб-браузерами и запускаться в контейнерах Docker Compose.

## 2. Описание предметной области

Риелтор работает с несколькими клиентами одновременно. Для каждого клиента необходимо учитывать бюджет, тип недвижимости, площадь, комнатность, район и дополнительные требования. При ручной работе поиск подходящих объектов ведется по нескольким сайтам, а результаты приходится сохранять и сравнивать вручную.

Предметная область проекта включает:

- ведение клиентской базы;
- фиксацию параметров запроса;
- поиск недвижимости на публичных источниках;
- отбор подходящих объектов;
- формирование подборки для клиента.

## 3. Анализ аналогов

Аналогами являются CRM-системы для агентств недвижимости, а также отдельные сайты-агрегаторы новостроек и коттеджных поселков. Большинство аналогов либо не объединяют поиск и работу с клиентом в одном интерфейсе, либо требуют использования нескольких разрозненных сервисов.

Преимуществом разрабатываемого решения является объединение клиентской карточки, параметров поиска, результатов парсинга и подборки в едином интерфейсе.

## 4. Проектирование приложения

### 4.1 Архитектура

Архитектура системы состоит из четырех основных частей:

- frontend на React + Vite;
- backend API на Node.js + Express;
- search-service на Python + FastAPI;
- база данных PostgreSQL.

Frontend отвечает за пользовательский интерфейс. Backend API обрабатывает запросы, авторизует пользователя и сохраняет данные. Search-service выполняет поиск по открытым источникам и возвращает нормализованные карточки объектов. PostgreSQL хранит клиентов, параметры поиска, найденные объекты, подборки и историю запусков поиска.

### 4.2 Структура базы данных

В базе данных используются следующие основные таблицы:

- `users`
- `clients`
- `client_search_profiles`
- `properties`
- `client_found_properties`
- `shortlist_items`
- `share_messages`
- `search_runs`

### 4.3 Пользовательские сценарии

Основные сценарии использования:

1. Авторизация риелтора.
2. Создание клиента.
3. Заполнение параметров поиска.
4. Запуск поиска объектов.
5. Просмотр найденных объектов.
6. Добавление объектов в подборку.
7. Копирование текста подборки и телефона клиента.

### 4.4 Интерфейс

Интерфейс приложения включает:

- страницу входа;
- главную страницу с KPI;
- список клиентов;
- карточку клиента;
- карточки найденных объектов;
- модальное окно просмотра объекта;
- страницу аналитики, профиль и справочную страницу.

## 5. Реализация

### 5.1 Frontend

Frontend реализован на React с использованием Vite. Интерфейс организован как SPA-приложение с маршрутизацией между страницами клиентов, dashboard, аналитики и настроек.

### 5.2 Backend

Backend реализован на Node.js и Express. Для работы с PostgreSQL используется библиотека `pg`. Все основные операции выполняются через SQL-репозиторий.

### 5.3 Сервис поиска

Сервис поиска реализован на Python и FastAPI. Источники поиска заранее заданы в коде через адаптеры. Парсинг HTML выполняется через `httpx`, `BeautifulSoup4` и `lxml`.

### 5.4 База данных

В качестве основной базы данных используется PostgreSQL. Для локального и продакшен-запуска используется Docker Compose. Данные БД сохраняются в persistent volume и не теряются после перезапуска контейнеров.

## 6. Тестирование

В проекте предусмотрены:

- unit-тесты для matcher и нормализации карточек;
- ручной QA-чеклист проверки поиска по параметрам;
- проверка сборки frontend и backend.

Текущие документы по тестированию:

- `docs/SEARCH_QA_CHECKLIST.md`
- `docs/PARSING_SOURCES.md`

## 7. Развертывание

Для запуска приложения используется Docker Compose. В контейнерах поднимаются:

- PostgreSQL;
- Python search-service;
- Node.js API;
- frontend через Nginx.

Такой подход упрощает локальный запуск и перенос приложения на сервер.

# Введение

Современный рынок недвижимости характеризуется высокой динамикой, большим количеством предложений, разнообразием источников публикации объектов и необходимостью оперативного взаимодействия между риелтором и клиентом. При сопровождении сделок по покупке недвижимости специалисту требуется не только собрать информацию о потребностях клиента, но и в короткие сроки подобрать релевантные варианты объектов, сравнить их между собой, исключить заведомо неподходящие предложения и подготовить подборку для дальнейшего обсуждения с клиентом. На практике значительная часть этих действий выполняется вручную, что увеличивает временные затраты, повышает риск пропуска подходящих объектов и усложняет хранение истории взаимодействия с клиентом.

Особенно остро проблема проявляется в ситуации, когда объекты недвижимости размещаются на большом количестве разрозненных сайтов: сайтах застройщиков, агрегаторах, каталогах новостроек, реестрах жилых комплексов и специализированных площадках по продаже домов. В этом случае риелтору приходится вручную открывать множество источников, многократно повторять одни и те же поисковые действия, фиксировать найденные предложения в заметках или таблицах, а затем отдельно формировать сообщение для клиента. Такой подход не обеспечивает целостности данных и плохо масштабируется при увеличении числа клиентов.

Внутри профессиональной деятельности риелтора особую роль играет учет параметров клиента. Важно хранить бюджет, диапазон площади, район, требования к дому или квартире, число комнат, отделку, предпочтения по сроку сдачи, площадь участка, количество этажей, наличие коммуникаций и другие характеристики. Если данные о потребности клиента не систематизированы, повторный поиск становится трудоемким, а история работы по заявке теряется. Кроме того, отсутствие единого механизма хранения результатов поиска затрудняет повторное обращение к ранее найденным объектам и снижает качество сопровождения клиента.

Актуальность темы выпускной квалификационной работы обусловлена необходимостью автоматизации процессов поиска недвижимости по внешним источникам, хранения клиентских потребностей и сопровождения риелтора на этапе формирования подборки объектов. Разработка программных модулей такого типа позволяет сократить долю ручных операций, повысить скорость обработки клиентских заявок, улучшить качество отбора объектов и обеспечить централизованное хранение данных в одной системе.

Объектом автоматизации является деятельность риелтора, связанная с приемом клиентской заявки, фиксацией параметров потребности, поиском объектов недвижимости по внешним сайтам, формированием подборки и сопровождением клиента до этапа отправки подобранных вариантов.

Предметом разработки являются программные модули, реализующие автоматизированный поиск недвижимости из внешних источников и интеграцию результатов поиска с модулями управления потребностями клиента при покупке недвижимости.

Целью выпускной квалификационной работы является разработка модулей программной системы, обеспечивающих:

- хранение и обработку информации о клиентах риелтора;
- сохранение параметров поиска недвижимости;
- автоматизированный поиск объектов по внешним интернет-источникам;
- фильтрацию и ранжирование найденных предложений;
- формирование подборки объектов для клиента;
- подготовку текста сообщения для дальнейшей отправки клиенту.

Для достижения поставленной цели в работе решаются следующие задачи:

- проанализировать назначение разрабатываемой системы и сформулировать цели ее создания;

- спроектировать модель данных системы;
- описать и формализовать основные алгоритмы обработки данных;

- реализовать базу данных в среде СУБД;

- провести модульное и интеграционное тестирование;
- подготовить эксплуатационную документацию.

Практическая значимость работы заключается в том, что разработанное решение может использоваться как единое рабочее пространство риелтора для ведения клиентских заявок, запуска поиска и подготовки подборок по объектам недвижимости города Краснодара.

# 1 Назначение и цели разработки

Разрабатываемые программные модули предназначены для автоматизации работы риелтора при обработке клиентских заявок, связанных с покупкой недвижимости. Система ориентирована на специалистов, которые ежедневно работают с большим числом клиентов, анализируют параметры потребности и осуществляют подбор объектов из внешних интернет-источников. Основным пользователем программного продукта является риелтор, которому необходимо быстро фиксировать запрос клиента, запускать поиск, просматривать найденные объекты, выбирать лучшие варианты и формировать подборку для дальнейшего общения с клиентом.

Назначение разработки заключается в создании набора взаимосвязанных программных модулей, позволяющих сопровождать клиентскую заявку на основных этапах ее обработки: от первичного занесения данных о клиенте до подготовки итоговой подборки объектов. В отличие от ручного подхода, при котором информация о клиентах, найденных объектах и подборках часто хранится в разных местах, разрабатываемая система концентрирует все данные в единой базе и предоставляет риелтору единый интерфейс для работы.

Программные модули должны обеспечивать возможность ввода контактной информации клиента, указания параметров поиска и хранения результатов последующей обработки. В системе должна быть предусмотрена работа как с квартирами, так и с домами, поскольку набор критериев в этих случаях различается. Для квартир существенными являются бюджет, количество комнат, площадь, район, этажность, срок сдачи и отделка. Для домов дополнительно важны площадь дома, площадь земельного участка, название поселка, количество спален, этажность дома, материал и наличие коммуникаций.

Назначение модуля поиска недвижимости состоит в автоматизированном обращении к внешним источникам информации, извлечении карточек объектов, нормализации данных, исключении нерелевантных предложений и расчете степени соответствия объектов параметрам клиента. Реализуемый модуль не просто собирает ссылки на объявления, а преобразует полученные данные в единую внутреннюю структуру, пригодную для сохранения в базе и отображения в интерфейсе.

Назначение модуля управления подборкой состоит в том, чтобы предоставить риелтору инструмент отбора лучших объектов из общего списка результатов. После анализа найденных вариантов риелтор должен иметь возможность вручную добавить часть из них в подборку, удалить неподходящие позиции, просмотреть итоговый состав и подготовить текст сообщения для клиента. Тем самым обеспечивается удобный переход от этапа автоматического поиска к этапу персональной работы риелтора с клиентом.

Еще одной важной функцией системы является фиксация истории поиска и состояния клиентской заявки. Пользователь должен видеть, был ли поиск уже запущен, сколько объектов найдено, какие объекты сохранены в подборку, была ли подборка отправлена клиенту. Такой подход повышает прозрачность работы и позволяет быстро продолжить обработку заявки после паузы.

Целью разработки является повышение эффективности работы риелтора за счет автоматизации операций, которые ранее выполнялись вручную. К таким операциям относятся:

- повторяющийся поиск по нескольким источникам;
- анализ характеристик найденных объектов;
- фильтрация результатов по району, бюджету и другим параметрам;
- подготовка списка рекомендованных вариантов;
- формирование текста для отправки клиенту.

Достижение указанной цели позволяет получить следующие практические результаты:

- сокращение времени на первичный подбор недвижимости;
- снижение вероятности ошибки при ручной обработке объявлений;
- упрощение хранения клиентских заявок и результатов поиска;
- обеспечение повторного использования найденных данных;
- повышение удобства сопровождения клиента.

Таким образом, назначение разработки состоит в создании прикладного программного продукта, позволяющего автоматизировать поиск недвижимости из внешних источников и интегрировать результаты такого поиска с модулем управления клиентскими потребностями.

## 2.2 Проектирование модели данных

Для обеспечения хранения клиентских заявок, параметров поиска, найденных объектов недвижимости, результатов ранжирования и подборок была разработана реляционная модель данных. Выбор реляционного подхода обусловлен тем, что внутри системы необходимо поддерживать строгие связи между сущностями, обеспечивать целостность данных, хранить историю поиска и использовать единые справочные данные при повторных запросах.

База данных должна решать несколько задач одновременно. Во-первых, требуется хранить учетные записи пользователей системы, то есть риелторов, которые работают с клиентами. Во-вторых, требуется хранить карточки клиентов и их параметры поиска. В-третьих, необходимо хранить сами объекты недвижимости, найденные из внешних источников, поскольку один и тот же объект может быть найден несколько раз или быть релевантен нескольким клиентам. В-четвертых, нужно хранить результаты поиска именно в контексте конкретного клиента, а также отдельную подборку и историю запуска поиска.

В результате проектирования были выделены следующие основные сущности:

- пользователи системы;
- клиенты;
- профили поиска клиентов;
- объекты недвижимости;
- найденные объекты по клиенту;
- подборка объектов;
- сообщения для отправки;
- журнал запусков поиска.

### Текстовое описание сущностей

Сущность `users` хранит учетные записи риелторов. Для каждой записи сохраняются логин, хэш пароля, имя сотрудника, электронная почта, телефон и служебные даты создания и обновления записи. Таблица необходима для авторизации и разграничения доступа к клиентским карточкам.

Сущность `clients` представляет основную карточку клиента. В ней содержатся ФИО, телефон, электронная почта, предпочитаемый канал связи, служебный статус заявки, тип интересующей недвижимости и комментарий риелтора. Каждая запись клиента принадлежит конкретному пользователю системы.

Сущность `client_search_profiles` хранит параметры поиска недвижимости. Она содержит числовые диапазоны бюджета, площади, комнатности, этажности, а также массивы районов и названий поселков. Для домов дополнительно сохраняются параметры площади дома, площади участка, материала, количества спален и наличия коммуникаций. Такая структура позволяет гибко адаптировать профиль поиска под разные типы недвижимости.

Сущность `properties` представляет централизованное хранилище всех найденных объектов недвижимости. Здесь сохраняются источник, ссылка на объявление, заголовок, описание, район, адрес, площадь, комнатность, цена, изображения и другие характеристики. Централизованное хранение позволяет не дублировать один и тот же объект при повторных поисках.

Сущность `client_found_properties` связывает конкретного клиента с конкретным найденным объектом. Здесь дополнительно фиксируются процент совпадения, причины совпадения и причины несоответствия. Благодаря этой таблице один и тот же объект может быть связан с несколькими клиентами, но с разной оценкой релевантности.

Сущность `shortlist_items` хранит состав подборки. В нее помещаются объекты, которые риелтор вручную отобрал для клиента. Эта сущность необходима для отделения общего результата автоматического поиска от итоговой персональной подборки.

Сущность `share_messages` предназначена для хранения подготовленного текста сообщения с подборкой объектов. Это позволяет фиксировать факт формирования сообщения и хранить историю таких действий.

Сущность `search_runs` хранит информацию о каждом запуске поиска. В таблице фиксируются статус выполнения, количество найденных объектов, количество сохраненных результатов, возможная ошибка, а также время начала и окончания обработки.

## 2.3 Разработка алгоритмов обработки данных

С точки зрения обработки данных наиболее значимыми в системе являются алгоритмы, обеспечивающие автоматический поиск объектов недвижимости, нормализацию полученной информации, фильтрацию нерелевантных записей и расчет степени соответствия объектов требованиям клиента. Именно эти алгоритмы превращают разрозненные данные из внешних сайтов в структурированный результат, пригодный для дальнейшей работы риелтора.

В рамках проекта целесообразно выделить следующие основные алгоритмы:

- алгоритм поиска объектов недвижимости по профилю клиента;
- алгоритм нормализации объявления;
- алгоритм фильтрации объектов по городу Краснодару;
- алгоритм расчета процента совпадения объекта с параметрами клиента;
- алгоритм формирования подборки и текста сообщения.

### 2.3.1 Разработка алгоритма поиска объектов недвижимости по профилю клиента

Алгоритм поиска объектов недвижимости предназначен для автоматического формирования списка релевантных предложений после создания карточки клиента или при повторном запуске поиска. На вход алгоритма поступают контактные данные клиента, тип недвижимости и параметры его потребности. На выходе алгоритм формирует ранжированный список объектов, сохраненных в базе данных и связанных с конкретной заявкой клиента.

Последовательность работы алгоритма выглядит следующим образом. После создания карточки клиента backend-сервер обращается к модулю поиска и передает ему пакет данных, описывающий профиль клиента. Далее модуль поиска определяет, какие источники необходимо использовать: источники для квартир или источники для домов. После этого выполняется обращение к стартовым URL этих источников, загрузка HTML-страниц и попытка извлечения информации из JSON-LD и видимых карточек объявлений.

Затем каждая найденная карточка проходит этап нормализации. Из текста извлекаются цена, площадь, комнатность, район, адрес, этаж, тип отделки и другие характеристики. Если объект не имеет минимально достаточного набора данных, он исключается. После этого выполняется географическая фильтрация: в систему допускаются только объекты, относящиеся к городу Краснодару.

Далее оставшиеся объекты проходят обязательную проверку по бюджету и другим параметрам. Для валидных объектов рассчитывается процент совпадения, после чего результат сортируется по степени соответствия и возвращается backend-серверу. Backend сохраняет объекты в централизованную таблицу объектов, связывает их с клиентом и обновляет статус поиска.

### 2.3.2 Разработка алгоритма нормализации объявления

Алгоритм нормализации объявления предназначен для преобразования сырых данных из внешнего источника в единый формат объекта недвижимости. Внешние сайты могут возвращать заголовки, описания и изображения в разных форматах, поэтому без нормализации невозможно обеспечить единообразный вид результата в базе данных и интерфейсе.

На первом этапе алгоритм получает текст карточки и дополнительные элементы, такие как JSON-LD, изображения, адресные фрагменты и служебные поля. Из полученного текста извлекаются базовые характеристики: цена, площадь, количество комнат, район, адрес, тип отделки, этаж и этажность, а для домов — площадь участка, количество спален, материал и коммуникации.

Далее заголовок проверяется на наличие служебного или маркетингового мусора. Если сайт вернул некорректный заголовок, например общий заголовок страницы, вместо него формируется краткое структурированное название на основе реальных характеристик объекта. Аналогичная логика применяется к описанию: если описание содержит служебные слова, телефоны, бессмысленные фрагменты или большое количество шумовых данных, оно отбрасывается, а затем создается краткий запасной текст на основе основных характеристик объекта.

После этого выполняется фильтрация изображений. Из выдачи исключаются логотипы, иконки, служебные изображения и технические SVG-файлы. В результате алгоритм формирует чистый нормализованный объект, содержащий только полезную информацию.

Наличие такого алгоритма особенно важно потому, что многие сайты публикуют объекты не только Краснодара, но и других городов Краснодарского края.

Алгоритм получает текст объявления, адресные фрагменты и URL-адрес страницы. На первом этапе он проверяет наличие прямых указаний на город Краснодар. Затем анализируются названия районов и микрорайонов города. Если объект содержит явные указания на другие города края, например Сочи, Новороссийск, Геленджик или Анапу, он исключается из дальнейшей обработки. Если объект содержит указание на Краснодар или на один из районов Краснодара, он признается допустимым и проходит в следующий этап.

Использование такого алгоритма позволяет исключить попадание в выдачу заведомо нерелевантных объектов и повысить доверие к результатам поиска.

### 2.3.4 Разработка алгоритма расчета процента совпадения

После прохождения обязательных фильтров каждый объект недвижимости должен быть оценен с точки зрения соответствия параметрам клиента. Для этого используется алгоритм расчета процента совпадения. Его назначение состоит в том, чтобы помочь риелтору быстрее выявить наиболее релевантные предложения.

Для квартир алгоритм учитывает следующие параметры:

- соответствие бюджета;
- соответствие комнатности;
- соответствие площади;
- совпадение района;
- соответствие сроку сдачи;
- соответствие отделке.

Для домов учитываются:

- соответствие бюджета;
- соответствие площади дома;
- соответствие площади участка;
- совпадение локации;
- наличие требуемых коммуникаций;
- соответствие числу этажей и спален.

Каждому критерию присваивается определенный вес. При выполнении условия объект получает соответствующую долю баллов. Если критерий нарушен, система фиксирует причину несоответствия. В результате формируются не только итоговый процент совпадения, но и списки причин совпадения и несоответствия. Это делает оценку прозрачной для пользователя.

### 2.3.5 Разработка алгоритма формирования подборки и текста сообщения

Заключительным алгоритмом, имеющим важное прикладное значение, является алгоритм формирования подборки и текста сообщения для клиента. После того как автоматический поиск завершен, риелтор должен вручную отобрать наиболее подходящие объекты. Выбранные объекты сохраняются в отдельной подборке, а на их основе система формирует человекочитаемое сообщение.

## 2.4 Проектирование пользовательского интерфейса

Пользовательский интерфейс системы должен быть ориентирован на практическую работу риелтора, а значит, обеспечивать быстрый доступ к основным действиям, минимизировать число лишних переходов и позволять быстро оценивать состояние клиентской заявки. При проектировании интерфейса учитывалось, что пользователь работает с приложением в браузере и регулярно переключается между обзорными экранами и детальными карточками клиентов.

Интерфейс системы реализован как набор основных экранных форм, логически связанных между собой. Вход в систему осуществляется через отдельную страницу авторизации. После успешного входа пользователь попадает на главную страницу с общей сводкой. Далее ему доступны раздел списка клиентов, форма создания нового клиента, карточка клиента, а также вспомогательные страницы профиля, настроек и справочной информации.

Страница входа должна выполнять две задачи: подтверждать доступ пользователя к системе и быстро объяснять назначение приложения. Поэтому на ней целесообразно разместить краткое описание возможностей системы, а также форму ввода логина и пароля.

Главная страница должна предоставлять сводную информацию по текущей работе. На ней требуется отобразить число клиентов в работе, число найденных объектов, число объектов в подборках и число подборок, готовых к отправке. Дополнительно рекомендуется разместить график активности и список клиентов, требующих внимания.

Страница списка клиентов предназначена для поиска и выбора нужной карточки. На ней должны быть фильтры, строка поиска и таблица со статусами клиентов. Это позволяет риелтору быстро ориентироваться в текущем объеме заявок.

Форма создания клиента является одной из наиболее важных экранных форм. В ней необходимо обеспечить ввод персональных данных клиента и параметров поиска недвижимости. Так как набор параметров для квартир и домов различается, в интерфейсе должно быть предусмотрено переключение между режимами ввода.

Карточка клиента является центральным экраном системы. На ней должны отображаться:

- основные сведения о клиенте;
- параметры поиска;
- число найденных объектов;
- состав подборки;
- список найденных объектов;
- история запусков поиска.

Внутри карточки клиента должна быть реализована возможность открытия подробной карточки объявления в модальном окне или выдвижной панели. Это дает возможность изучить объект подробнее, не теряя контекст работы с клиентом.

С точки зрения визуального проектирования интерфейс должен быть достаточно строгим и рабочим, без лишних декоративных элементов. Важные действия, такие как запуск поиска, добавление в подборку и формирование текста сообщения, должны быть выделены достаточно заметно. При этом второстепенные действия должны оставаться доступными, но не перегружать пользователя.

## 3.1 Реализация базы данных в среде СУБД

Для практической реализации базы данных в рамках проекта была выбрана реляционная система управления базами данных PostgreSQL. Выбор данной СУБД обусловлен ее широким распространением, поддержкой транзакций, надежной работой с ограничениями целостности, удобством хранения структурированных данных и возможностью дальнейшего масштабирования. Для разрабатываемой системы PostgreSQL является удачным выбором, поскольку позволяет хранить как строго структурированные таблицы, так и отдельные JSON-поля, а также поддерживает массивы строк, что удобно для списков районов и коммуникаций.

Физическая структура базы данных создается SQL-скриптом инициализации, расположенным в проекте. Данный скрипт формирует таблицы пользователей, клиентов, профилей поиска, найденных объектов, подборок, истории запуска поиска и подготовленных сообщений. Кроме создания таблиц в нем задаются внешние ключи, ограничения уникальности и индексы, ускоряющие выполнение типовых запросов.

Отдельного внимания заслуживает то, что база данных подключена к системе через Docker Compose и использует постоянный том хранения. Это позволяет обеспечить сохранность данных после перезапуска контейнеров. Таким образом, база данных функционирует не как временное хранилище, а как постоянный уровень хранения прикладной информации.

При реализации таблицы `users` были предусмотрены поля авторизации и базовой идентификации риелтора. Для хранения карточек клиентов была реализована таблица `clients`, связанная с таблицей пользователей отношением «один ко многим». Для параметров поиска была создана отдельная таблица `client_search_profiles`, что позволило отделить контактные данные клиента от его поисковой потребности. Такой подход улучшает читаемость модели и упрощает дальнейшую модификацию структуры.

Таблица `properties` была реализована как единое хранилище найденных объектов недвижимости. Данный подход выгоден тем, что исключает повторное дублирование карточек объекта при нескольких поисковых запусках. Для связи конкретного клиента с конкретным найденным объектом используется таблица `client_found_properties`, в которой дополнительно сохраняются значение процента совпадения и объясняющие причины.

Для хранения пользовательской подборки была реализована таблица `shortlist_items`, а для хранения текстов, формируемых системой перед отправкой клиенту, — таблица `share_messages`. История запусков поиска хранится в таблице `search_runs`. Это позволяет проследить, когда выполнялся поиск, успешно ли он завершился и сколько объектов было найдено и сохранено.

Для обеспечения производительности в схеме были предусмотрены индексы по полям, наиболее часто используемым при фильтрации и соединении данных. К таким полям относятся идентификатор риелтора у клиентов, статус клиентской заявки, ссылка на объект недвижимости, идентификатор клиента в таблицах найденных объектов, подборок и запусков поиска.

## 3.2 Программная реализация модулей системы в среде программирования

Программная реализация системы выполнена по модульному принципу. Архитектурно решение разделено на три основные части: клиентскую часть, серверную часть и отдельный сервис поиска. Такой подход позволяет разграничить ответственность между модулями, упростить сопровождение системы и обеспечить независимое развитие каждого слоя.

Клиентская часть реализована на React с использованием TypeScript и сборщика Vite. Она отвечает за отображение интерфейса риелтора, маршрутизацию между страницами, отправку запросов на backend и отрисовку результатов. Серверная часть реализована на Node.js с использованием Express и TypeScript. Ее задачи заключаются в обработке запросов frontend, выполнении бизнес-логики, работе с PostgreSQL и запуске внешнего поискового модуля. Модуль поиска реализован на Python с использованием FastAPI и библиотек для парсинга HTML.

### Реализация клиентской части

Клиентская часть включает в себя страницы входа, dashboard, списка клиентов, создания клиента, карточки клиента, страницы профиля, настроек и помощи. Основная маршрутизация приложения организована в корневом модуле frontend. Для защищенных страниц применяется проверка факта авторизации пользователя.

В форме создания клиента реализована логика ввода параметров поиска недвижимости. В зависимости от выбранного типа объекта — квартира или дом — форма отображает разный набор полей. После отправки формы frontend формирует структурированный запрос и передает его backend-серверу.

Особое значение имеет экран карточки клиента. Именно здесь объединяются результаты поиска, подборка и история запусков. Для каждого найденного объекта в интерфейсе отображаются краткие характеристики, цена, степень соответствия и действия риелтора. При необходимости пользователь может открыть подробную информацию об объекте и вручную включить его в подборку.

### Реализация серверной части

Серверная часть реализует авторизацию, работу с клиентскими карточками, хранение параметров поиска, получение dashboard-статистики и формирование подборки. Для проверки доступа используется JWT-сессия, передаваемая через защищенную cookie. После успешного входа backend определяет пользователя по логину и сверяет хэш пароля.

Работа с базой данных организована через SQL-слой, где реализованы запросы создания, чтения, изменения и удаления данных. Такой подход обеспечивает прозрачный контроль над структурой запросов и связями между таблицами.

При создании клиента backend не ограничивается простым сохранением карточки. Он также формирует профиль поиска и сразу же запускает модуль поиска. После получения результатов выполняется сохранение объектов недвижимости, а затем фиксируется связь между клиентом и найденными объектами.

### Реализация модуля поиска

Поисковый модуль работает как отдельный HTTP-сервис. Backend передает в него структурированный запрос, содержащий параметры поиска клиента. После этого сервис определяет нужный набор источников, выполняет обход сайтов, извлекает объявления из JSON-LD и HTML-структур, фильтрует результаты и рассчитывает степень соответствия.

Для нормализации данных используется базовый адаптер, в котором сосредоточена общая логика очистки заголовков, описаний, изображений и адресных данных. Отдельное внимание уделено фильтрации по географии, так как проект должен отбирать только объекты города Краснодара. После нормализации выполняется расчет степени соответствия объекту требованиям клиента.

# 4 Тестирование программных модулей

## 4.1 Модульное тестирование

Для проверки корректности отдельных участков логики было проведено модульное тестирование. Наиболее подходящим объектом для автоматизированной проверки в рамках данного проекта является модуль поиска, так как именно в нем содержится логика нормализации данных и расчета степени соответствия объектов требованиям клиента.

Модульные тесты реализованы на Python с использованием стандартного механизма `unittest`. Они покрывают наиболее значимые сценарии:

- корректный расчет совпадения для квартиры;
- корректный расчет совпадения для дома;
- замена мусорного заголовка на нормализованный;
- отбрасывание служебных изображений;
- защита от ложной комнатности;

## 5.1 Руководство пользователя

Разрабатываемая система предназначена для практического использования риелтором, поэтому важной частью пояснительной записки является описание порядка работы с приложением. Руководство пользователя должно объяснять, как войти в систему, как создать карточку клиента, как запустить поиск и как сформировать подборку.

### Текст для раздела

Для начала работы пользователь должен открыть приложение в браузере и перейти на страницу входа. После этого необходимо ввести логин и пароль одной из преднастроенных учетных записей риелторов. После успешного входа система перенаправляет пользователя на главную страницу, где отображаются сводные показатели текущей работы.

Для создания новой клиентской заявки необходимо перейти в раздел добавления клиента. На данной странице пользователь вводит контактные данные клиента и задает параметры поиска недвижимости. После сохранения карточки автоматически запускается поиск объектов, а пользователь перенаправляется в карточку клиента.

В карточке клиента пользователь может просматривать найденные объекты, открывать подробную информацию по каждому из них и вручную добавлять наиболее подходящие объекты в подборку. После формирования подборки доступна функция генерации текста сообщения. Полученный текст пользователь может скопировать и отправить клиенту в выбранном канале связи.

После отправки сообщения пользователь может отметить подборку как отправленную. Это позволяет системе корректно отразить состояние клиентской заявки и учитывать его в общей статистике.

# Введение

Современный рынок недвижимости характеризуется большим объемом предложений и высокой скоростью их обновления. В процессе работы с клиентом риелтору требуется фиксировать параметры потребности, анализировать несколько внешних источников, отбирать подходящие объекты и подготавливать итоговую подборку. При ручной организации такой работы возрастают временные затраты, усложняется повторная обработка заявки и повышается риск потери информации.

Актуальность темы выпускной квалификационной работы обусловлена необходимостью автоматизации поиска недвижимости по внешним источникам и централизованного хранения клиентских заявок. Использование специализированной программной системы позволяет сократить объем ручных операций, ускорить подбор объектов и повысить удобство сопровождения клиента.

Объектом автоматизации является деятельность риелтора, связанная с приемом клиентской заявки, фиксацией параметров поиска, подбором объектов недвижимости и формированием итоговой подборки. Предметом разработки являются программные модули, обеспечивающие поиск недвижимости из внешних источников и интеграцию результатов поиска с модулем управления потребностями клиента.

Целью выпускной квалификационной работы является разработка программных модулей системы, предназначенной для хранения клиентских заявок, автоматизированного поиска объектов недвижимости, сохранения результатов и формирования подборки для клиента. Практическая значимость работы заключается в создании прикладного программного решения, которое может использоваться как единое рабочее пространство риелтора при сопровождении заявок на покупку недвижимости в городе Краснодаре.

# 1 Назначение и цели разработки

Разрабатываемые программные модули предназначены для автоматизации работы риелтора при обработке клиентских заявок, связанных с покупкой недвижимости. Основным пользователем системы является риелтор, который вводит данные клиента, задает параметры поиска, просматривает найденные объекты и формирует подборку для дальнейшей отправки.

Назначение разработки состоит в создании программного решения, объединяющего хранение клиентских заявок, учет параметров поиска и автоматизированный подбор объектов из внешних источников. Система должна обеспечивать единое информационное пространство для сопровождения клиентской заявки.

Программа должна поддерживать ввод и хранение клиентских данных, запуск поиска по внешним сайтам, сохранение результатов, отображение найденных объектов и формирование подборки. Целью разработки является повышение эффективности работы риелтора за счет сокращения ручных операций и упрощения дальнейшей работы с результатами поиска.

Указанные алгоритмы обеспечивают преобразование данных из внешних источников в структурированный результат, пригодный для дальнейшей работы риелтора. - алгоритм нормализации объявления;
- алгоритм фильтрации объектов по городу Краснодару;

### 2.6.1 Разработка алгоритма поиска объектов недвижимости по профилю клиента

Алгоритм поиска объектов недвижимости предназначен для автоматического формирования списка релевантных предложений после создания карточки клиента или при повторном запуске поиска. На вход алгоритма поступают контактные данные клиента, тип недвижимости и параметры его потребности. На выходе алгоритм формирует ранжированный список объектов, сохраненных в базе данных и связанных с конкретной заявкой клиента.

Последовательность работы алгоритма выглядит следующим образом. После создания карточки клиента backend-сервер обращается к модулю поиска и передает ему пакет данных, описывающий профиль клиента. Далее модуль поиска определяет, какие источники необходимо использовать: источники для квартир или источники для домов. После этого выполняется обращение к стартовым URL этих источников, загрузка HTML-страниц и попытка извлечения информации из JSON-LD и видимых карточек объявлений.

Затем каждая найденная карточка проходит этап нормализации. Из текста извлекаются цена, площадь, комнатность, район, адрес, этаж, тип отделки и другие характеристики. Если объект не имеет минимально достаточного набора данных, он исключается. После этого выполняется географическая фильтрация: в систему допускаются только объекты, относящиеся к городу Краснодару.

Далее оставшиеся объекты проходят обязательную проверку по бюджету и другим параметрам. Для валидных объектов рассчитывается процент совпадения, после чего результат сортируется по степени соответствия и возвращается backend-серверу. Backend сохраняет объекты в централизованную таблицу объектов, связывает их с клиентом и обновляет статус поиска.

### 2.6.2 Разработка алгоритма нормализации объявления

Алгоритм нормализации объявления предназначен для преобразования сырых данных из внешнего источника в единый формат объекта недвижимости. Внешние сайты могут возвращать заголовки, описания и изображения в разных форматах, поэтому без нормализации невозможно обеспечить единообразный вид результата в базе данных и интерфейсе.

На первом этапе алгоритм получает текст карточки и дополнительные элементы, такие как JSON-LD, изображения, адресные фрагменты и служебные поля. Из полученного текста извлекаются базовые характеристики: цена, площадь, количество комнат, район, адрес, тип отделки, этаж и этажность, а для домов — площадь участка, количество спален, материал и коммуникации.

Для части количественных характеристик может использоваться расчетный способ. Например, удельная стоимость объекта определяется по формуле

`C_{м²} = C / S`,                                                     (1)

где `C_{м²}` — стоимость одного квадратного метра, `C` — полная стоимость объекта, `S` — общая площадь объекта. Использование формулы (1) позволяет получить сопоставимый показатель даже в тех случаях, когда источник не передает стоимость одного квадратного метра в явном виде.

Далее заголовок проверяется на наличие служебного или маркетингового мусора. Если сайт вернул некорректный заголовок, например общий заголовок страницы, вместо него формируется краткое структурированное название на основе реальных характеристик объекта. Аналогичная логика применяется к описанию: если описание содержит служебные слова, телефоны, бессмысленные фрагменты или большое количество шумовых данных, оно отбрасывается, а затем создается краткий запасной текст на основе основных характеристик объекта.

После этого выполняется фильтрация изображений. Из выдачи исключаются логотипы, иконки, служебные изображения и технические SVG-файлы. В результате алгоритм формирует чистый нормализованный объект, содержащий только полезную информацию.

### 2.6.4 Разработка алгоритма расчета процента совпадения

После прохождения обязательных фильтров каждый объект недвижимости должен быть оценен с точки зрения соответствия параметрам клиента. Для этого используется алгоритм расчета процента совпадения. Его назначение состоит в том, чтобы помочь риелтору быстрее выявить наиболее релевантные предложения.

Для квартир алгоритм учитывает следующие параметры:

- соответствие бюджета;
- соответствие комнатности;
- соответствие площади;
- совпадение района;
- соответствие сроку сдачи;
- соответствие отделке.

Для домов учитываются:

- соответствие бюджета;
- соответствие площади дома;
- соответствие площади участка;
- совпадение локации;
- наличие требуемых коммуникаций;
- соответствие числу этажей и спален.

Каждому критерию присваивается определенный вес. При выполнении условия объект получает соответствующую долю баллов. Если критерий нарушен, система фиксирует причину несоответствия. В результате формируются итоговый процент совпадения, а также перечни причин совпадения и несоответствия.

Формализованно итоговый процент совпадения может быть представлен в виде взвешенной оценки

`M = (Σ(w_i * k_i) / Σw_i) * 100%`,                                   (2)

где `M` — итоговый процент совпадения объекта требованиям клиента, `w_i` — вес `i`-го критерия, `k_i` — коэффициент выполнения `i`-го критерия, принимающий значение в диапазоне от `0` до `1`, `n` — количество оцениваемых критериев.

Для частных критериев, допускающих долевую оценку, может использоваться дополнительная формула. Например, для оценки совпадения по коммуникациям у домов применяется выражение

`k_{комм} = n_{совп} / n_{треб}`,                                     (3)

где `k_{комм}` — коэффициент совпадения по коммуникациям, `n_{совп}` — число коммуникаций, фактически найденных у объекта, `n_{треб}` — число коммуникаций, указанных в запросе клиента.

## 2.7 Проектирование интерфейса пользователя

Пользовательский интерфейс системы должен обеспечивать быстрый доступ к основным действиям, минимальное число лишних переходов и наглядное отображение состояния клиентской заявки. При проектировании интерфейса учитывалась работа пользователя в браузере и необходимость регулярного перехода между обзорными экранами и детальными карточками клиентов.

Интерфейс системы реализован как набор основных экранных форм, логически связанных между собой. Вход в систему осуществляется через отдельную страницу авторизации. После успешного входа пользователь попадает на главную страницу с общей сводкой. Далее ему доступны раздел списка клиентов, форма создания нового клиента, карточка клиента, а также вспомогательные страницы профиля, настроек и справочной информации.

Страница входа должна обеспечивать контролируемый доступ к системе и содержать форму ввода учетных данных.

Главная страница должна предоставлять сводную информацию по текущей работе. На ней требуется отобразить число клиентов в работе, число найденных объектов, число объектов в подборках и число подборок, готовых к отправке. Дополнительно рекомендуется разместить график активности и список клиентов, требующих внимания.

Страница списка клиентов предназначена для поиска и выбора нужной карточки. На ней размещаются фильтры, строка поиска и таблица со статусами клиентов.

Форма создания клиента является одной из наиболее важных экранных форм. В ней необходимо обеспечить ввод персональных данных клиента и параметров поиска недвижимости. Так как набор параметров для квартир и домов различается, в интерфейсе должно быть предусмотрено переключение между режимами ввода.

Карточка клиента является центральным экраном системы. На ней должны отображаться:

- основные сведения о клиенте;
- параметры поиска;
- число найденных объектов;
- состав подборки;
- список найденных объектов;
- история запусков поиска.

Внутри карточки клиента должна быть предусмотрена возможность открытия подробной карточки объявления без потери контекста работы с заявкой.

С точки зрения визуального проектирования интерфейс должен быть достаточно строгим и рабочим, без лишних декоративных элементов. Важные действия, такие как запуск поиска, добавление в подборку и формирование текста сообщения, должны быть выделены достаточно заметно. При этом второстепенные действия должны оставаться доступными, но не перегружать пользователя.

# Приложение А

**Техническое задание. Требования к системе**

## А.1 Основание для разработки

Основанием для разработки программной системы `Тэона` является задание на выполнение выпускной квалификационной работы по теме разработки модулей поиска недвижимости из внешних источников и их интеграции с модулями управления потребностями клиента при покупке или продаже недвижимости.

Разработка выполняется как прикладной программный проект, ориентированный на автоматизацию рабочих действий риелтора при сопровождении заявок на покупку недвижимости в городе Краснодаре.

## А.2 Назначение разработки

Назначение программной системы состоит в объединении в одном приложении следующих функций:

- ведение клиентских карточек;
- хранение параметров поиска недвижимости;
- автоматический поиск объектов по внешним интернет-источникам;
- сохранение найденных объектов в базе данных;
- формирование подборки объектов для клиента;
- подготовка текста сообщения для последующей отправки.

Система предназначена для практического использования риелтором и должна обеспечивать сокращение объема ручной работы при подборе недвижимости.

## А.3 Функциональные требования

Разрабатываемая система должна обеспечивать выполнение следующих функций:

1. авторизация пользователя по логину и паролю;
2. отображение списка клиентов риелтора;
3. создание новой карточки клиента;
4. редактирование данных клиента и параметров поиска;
5. запуск поиска объектов недвижимости по заданному профилю;
6. фильтрация найденных объектов по географическому признаку;
7. расчет процента соответствия объекта параметрам клиента;
8. сохранение найденных объектов и результатов поиска в базе данных;
9. отображение найденных объектов в карточке клиента;
10. ручное добавление объектов в подборку;
11. удаление объектов из подборки;
12. формирование текста сообщения по подборке;
13. фиксация факта отправки подборки клиенту;
14. отображение сводных показателей по текущим заявкам.

## А.4 Требования к входным данным

Входными данными системы являются:

- логин и пароль риелтора;
- персональные данные клиента:
  - фамилия, имя и отчество;
  - телефон;
  - электронная почта;
  - предпочтительный канал связи;
- параметры поиска квартиры:
  - бюджет;
  - диапазон площади;
  - диапазон комнатности;
  - районы города;
  - срок сдачи;
  - тип отделки;
  - этажность;
- параметры поиска дома:
  - бюджет;
  - площадь дома;
  - площадь участка;
  - район или населенный пункт;
  - количество этажей;
  - количество спален;
  - материал дома;
  - наличие коммуникаций.

Дополнительными входными данными являются HTML-страницы и JSON-LD-блоки внешних сайтов недвижимости, используемые модулем поиска при извлечении объявлений.

## А.5 Требования к выходным данным

Выходными данными системы являются:

- список найденных объектов недвижимости;
- карточка объекта с нормализованными характеристиками;
- процент совпадения объекта с параметрами клиента;
- список причин совпадения и несоответствия;
- состав подборки клиента;
- текст сообщения, подготовленный для отправки;
- сводные показатели dashboard;
- записи базы данных, отражающие историю поиска и работы с заявкой.

## А.6 Требования к надежности и ограничениям

Система должна обеспечивать сохранность данных при перезапуске сервисов. Для выполнения этого требования база данных PostgreSQL должна использовать постоянное хранилище.

При ошибке запуска внешнего модуля поиска система должна:

- сохранять карточку клиента;
- фиксировать ошибку в журнале запусков поиска;
- переводить заявку в состояние `error`;
- позволять повторный запуск поиска после устранения причины ошибки.

При отсутствии найденных объектов система должна:

- фиксировать завершение поиска;
- сохранять информацию о запуске;
- переводить клиентскую заявку в состояние `no_results`.

Система ориентирована на поиск недвижимости в городе Краснодаре, поэтому найденные объекты других городов Краснодарского края не должны попадать в итоговую выдачу.

## А.7 Требования к программным и техническим средствам

Для работы системы требуется следующее программное окружение:

- Docker;
- Docker Compose;
- современный веб-браузер;
- локальная или серверная среда, поддерживающая запуск контейнеров.

Состав программных модулей:

- frontend-модуль на `React + TypeScript + Vite`;
- backend API на `Node.js + Express + TypeScript`;
- search-service на `Python + FastAPI`;
- реляционная база данных `PostgreSQL`;
- связующее контейнерное окружение `Docker Compose`.

## А.8 Требования к пользовательскому интерфейсу

Пользовательский интерфейс должен быть реализован в виде веб-приложения и обеспечивать:

- страницу входа в систему;
- страницу со сводными показателями;
- страницу списка клиентов;
- форму создания клиента;
- карточку клиента с найденными объектами;
- просмотр детальной информации по объекту недвижимости.

Интерфейс должен обеспечивать понятную навигацию между экранами, наглядное отображение основных действий и удобную работу риелтора с клиентской карточкой.

## 1 Характеристика программы

Программная система `Тэона` реализована как web-приложение с разделением на клиентскую часть, backend API, отдельный поисковый сервис и реляционную базу данных `PostgreSQL`. Основной проект организован по модульному принципу и включает каталоги `apps/web`, `apps/api` и `apps/search-service`.

В программе выделены следующие функциональные направления: модуль авторизации риелтора, модуль создания и сопровождения клиентских заявок, модуль автоматического поиска объектов недвижимости, модуль формирования подборки и модуль сводной аналитики по заявкам.

Модуль создания и сопровождения заявок отвечает за ввод данных клиента, сохранение параметров поиска и управление состоянием карточки. Поисковый модуль выполняет загрузку объявлений из внешних источников, фильтрацию по Краснодару, нормализацию характеристик и расчет процента совпадения. Модуль подборки обеспечивает выбор подходящих объектов и подготовку текста сообщения для клиента. Модуль аналитики обеспечивает расчет сводных показателей для главной страницы системы.

## 2 Обращение к программе

Для обращения к программе необходимо открыть корень проекта `realestate` в среде `Visual Studio Code`. После открытия проекта следует проверить наличие файла `.env`, доступность `Docker Desktop` и возможность подключения к базе данных `PostgreSQL`.

Подготовка к запуску выполняется в следующем порядке:

- открыть корневой каталог проекта в `Visual Studio Code`;
- при необходимости установить Node.js-зависимости командой `npm install`;
- запустить `Docker Desktop`;
- выполнить команду `npm run prod:up`;
- дождаться запуска контейнеров `postgres`, `search-service`, `api` и `web`;
- открыть в браузере адрес `http://localhost:5002`.

Для просмотра содержимого базы данных рекомендуется использовать `TablePlus`. Для этого необходимо подключиться к локальному экземпляру `PostgreSQL`, используя параметры из файла `docker-compose.yml`.

Для остановки программного продукта используется команда `npm run prod:down`. Для просмотра журналов контейнеров используется команда `npm run prod:logs`. Для полного сброса базы данных используется команда `npm run prod:reset-db`.

Модульные тесты запускаются командой `cd apps/search-service && .venv/bin/python -m unittest discover -s tests -v`. Указанные тесты проверяют корректность нормализации объявлений и расчета процента совпадения без необходимости ручного открытия интерфейса.

## 6. Построение диаграммы деятельности

### Название рисунка

`Рисунок 6 – Диаграмма деятельности подбора объектов недвижимости по профилю клиента`

### Логика потока

Поток должен идти сверху вниз.

### Стартовый блок

- `Начало`

### Основные действия

1. Риелтор открывает форму создания клиента.
2. Риелтор вводит данные клиента.
3. Система сохраняет карточку клиента.
4. Система сохраняет профиль поиска.
5. Backend запускает модуль поиска.
6. Search-service выбирает источники по типу недвижимости.
7. Search-service загружает страницы источников.
8. Search-service извлекает карточки объектов.
9. Система проверяет наличие цены и ссылки.
10. Система проверяет принадлежность объекта к Краснодару.
11. Система нормализует характеристики объекта.
12. Система проверяет обязательные фильтры клиента.
13. Система рассчитывает процент совпадения.
14. Backend сохраняет результаты в БД.
15. Backend возвращает результат frontend.
16. Риелтор просматривает найденные объекты.
17. `Конец`

### Где должны быть ромбы-условия

Ромб №1:

- `Карточка объекта извлечена корректно?`

Если нет:

- возврат к обработке следующего объекта.

Ромб №2:

- `Объект относится к Краснодару?`

Если нет:

- отклонить объект.

Ромб №3:

- `Объект проходит обязательные фильтры?`

Если нет:

- отклонить объект.

Ромб №4:

- `Найден хотя бы один объект?`

Если нет:

- установить состояние `no_results`.

Если да:

- показать результаты.

### Поясняющий текст к рисунку

На рисунке 6 представлена диаграмма деятельности подбора объектов недвижимости по профилю клиента. Диаграмма отражает последовательность действий от момента ввода параметров клиента до получения и сохранения найденных объявлений. Особое значение в алгоритме имеют этапы географической фильтрации, нормализации данных и расчета процента соответствия объектов.

## 7. Построение диаграммы состояний

### Название рисунка

`Рисунок 7 – Диаграмма состояний клиентской заявки`

### Объект, для которого строится диаграмма

- `Клиентская заявка`

### Состояния

- `new` – новая заявка.
- `searching` – идет поиск объектов.
- `found` – найдены объекты.
- `no_results` – подходящие объекты не найдены.
- `shortlist_ready` – подборка сформирована.
- `sent` – подборка отправлена клиенту.
- `error` – произошла ошибка поиска или обработки.

### Переходы между состояниями

- `new -> searching`
  - при сохранении клиента и запуске поиска.

- `searching -> found`
  - если система получила и сохранила хотя бы один объект.

- `searching -> no_results`
  - если поиск завершился без результатов.

- `searching -> error`
  - если модуль поиска вернул ошибку или недоступен.

- `found -> shortlist_ready`
  - если риелтор добавил хотя бы один объект в подборку.

- `shortlist_ready -> found`
  - если риелтор удалил из подборки все объекты, но найденные результаты остались.

- `shortlist_ready -> sent`
  - если риелтор сформировал сообщение и отметил подборку как отправленную.

### Поясняющий текст к рисунку

На рисунке 7 представлена диаграмма состояний клиентской заявки. Она отражает жизненный цикл карточки клиента внутри системы: от момента создания до формирования подборки и фиксации факта отправки. Использование диаграммы состояний позволяет показать, какие состояния являются допустимыми и при каких действиях пользователя или внутренних событиях системы выполняется переход между ними.

## 9. Построение фрагментов ER-диаграммы базы данных

### Названия рисунков

`Рисунок 9 – Фрагмент ER-диаграммы базы данных: пользователи, клиенты, поисковые профили и результаты поиска`

`Рисунок 10 – Фрагмент ER-диаграммы базы данных: объекты недвижимости, подборки, сообщения и история поиска`

### Сущности, которые должны быть на схемах

- `users`
- `clients`
- `client_search_profiles`
- `properties`
- `client_found_properties`
- `shortlist_items`
- `share_messages`
- `search_runs`

### Как именно разнести связи по фрагментам

#### Фрагмент 1: `users`, `clients`, `client_search_profiles`, `client_found_properties`

Связь:

- один пользователь системы может вести много клиентов;
- одна клиентская заявка имеет один профиль поиска;
- одна клиентская заявка может иметь много результатов поиска.

Графически:

- `users (1) ---- (M) clients`
- `clients (1) ---- (1) client_search_profiles`
- `clients (1) ---- (M) client_found_properties`

#### Фрагмент 2: `properties`, `shortlist_items`, `share_messages`, `search_runs`

Связи:

- один объект недвижимости может участвовать во множестве результатов поиска;
- один клиент может иметь много элементов подборки;
- один клиент может иметь много подготовленных сообщений;
- один клиент может иметь много запусков поиска;
- один объект может быть помещен в подборку разных клиентов, но связь с клиентом хранится в таблице `shortlist_items`;
- таблица `search_runs` хранит историю повторных запусков поиска по карточке клиента.

### Рекомендуемый порядок расположения сущностей на рисунках

Сверху:

- `users`

Под ним:

- `clients`

Справа от `clients`:

- `client_search_profiles`

Отдельным вторым рисунком:

- `properties`
- `shortlist_items`
- `share_messages`
- `search_runs`
- `client_found_properties`

Такое разбиение повышает читаемость модели данных, поскольку пользовательский контур и контур хранения результатов не смешиваются в пределах одного перегруженного рисунка.

## 10. Табличное описание базы данных

В референсной пояснительной записке модель данных поддерживается не только рисунками, но и табличным описанием сущностей. В рассматриваемой работе после блока фрагментов ER-диаграммы рекомендуется привести общую таблицу с характеристикой сущностей и, при необходимости, отдельные таблицы по полям.

### Таблица 4 – Характеристика сущностей и связей между сущностями ER-диаграммы

| Сущность | Назначение | Основной ключ | Внешние ключи | Основные связи |
|---|---|---|---|---|
| `users` | Хранение учетных записей риелторов | `id` | — | 1:M с `clients` |
| `clients` | Хранение карточек клиентов | `id` | `realtor_id -> users.id` | 1:1 с `client_search_profiles`, 1:M с `client_found_properties`, `shortlist_items`, `share_messages`, `search_runs` |
| `client_search_profiles` | Хранение критериев поиска | `id`, `client_id` | `client_id -> clients.id` | 1:1 с `clients` |
| `properties` | Хранение нормализованных объектов недвижимости | `id` | — | 1:M с `client_found_properties`, 1:M с `shortlist_items` |
| `client_found_properties` | Хранение результатов поиска по клиенту | `id` | `client_id -> clients.id`, `property_id -> properties.id` | M:1 с `clients`, M:1 с `properties` |
| `shortlist_items` | Хранение объектов, отобранных в подборку | `id` | `client_id -> clients.id`, `property_id -> properties.id` | M:1 с `clients`, M:1 с `properties` |
| `share_messages` | Хранение сформированных сообщений | `id` | `client_id -> clients.id` | M:1 с `clients` |
| `search_runs` | Хранение истории запусков поиска | `id` | `client_id -> clients.id` | M:1 с `clients` |

При оформлении блок-схем рекомендуется использовать следующий состав обозначений:

- овал или скругленный блок – начало и конец;
- прямоугольник – действие;
- ромб – условие;
- стрелки – переходы между этапами;
- при длинных схемах – разрыв линии или переход на вторую часть рисунка.

### Схемы, которые допускается разбивать на две части

Если рисунок получается слишком длинным, его целесообразно разбить на две части:

- алгоритм поиска объектов;
- алгоритм расчета процента совпадения.

### Как подписывать части рисунка

Например:

- `Рисунок 11 – Блок-схема алгоритма поиска объектов недвижимости. Часть 1`
- `Рисунок 11 – Блок-схема алгоритма поиска объектов недвижимости. Часть 2`

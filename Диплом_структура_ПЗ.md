# Введение

Современный рынок недвижимости характеризуется большим объемом предложений и высокой скоростью их обновления. В процессе работы с клиентом риелтору требуется фиксировать параметры потребности, анализировать несколько внешних источников, отбирать подходящие объекты и подготавливать итоговую подборку. При ручной организации такой работы возрастают временные затраты, усложняется повторная обработка заявки и повышается риск потери информации.

Актуальность темы выпускной квалификационной работы обусловлена необходимостью автоматизации поиска недвижимости по внешним источникам и централизованного хранения клиентских заявок. Использование специализированной программной системы позволяет сократить объем ручных операций, ускорить подбор объектов и повысить удобство сопровождения клиента.

Объектом автоматизации является деятельность риелтора, связанная с приемом клиентской заявки, фиксацией параметров поиска, подбором объектов недвижимости и формированием итоговой подборки. Предметом разработки являются программные модули, обеспечивающие поиск недвижимости из внешних источников и интеграцию результатов поиска с модулем управления потребностями клиента.

Целью выпускной квалификационной работы является разработка программных модулей системы, предназначенной для хранения клиентских заявок, автоматизированного поиска объектов недвижимости, сохранения результатов и формирования подборки для клиента. Практическая значимость работы заключается в создании прикладного программного решения, которое может использоваться как единое рабочее пространство риелтора при сопровождении заявок на покупку недвижимости в городе Краснодаре.

# 1 Назначение и цели разработки

Разрабатываемые программные модули предназначены для автоматизации работы риелтора при обработке клиентских заявок, связанных с покупкой недвижимости. Основным пользователем системы является риелтор, который вводит данные клиента, задает параметры поиска, просматривает найденные объекты и формирует подборку для дальнейшей отправки.

Назначение разработки состоит в создании программного решения, объединяющего хранение клиентских заявок, учет параметров поиска и автоматизированный подбор объектов из внешних источников. Система должна обеспечивать единое информационное пространство для сопровождения клиентской заявки.

Программа должна поддерживать ввод и хранение клиентских данных, запуск поиска по внешним сайтам, сохранение результатов, отображение найденных объектов и формирование подборки. Целью разработки является повышение эффективности работы риелтора за счет сокращения ручных операций и упрощения дальнейшей работы с результатами поиска.

## 2.6 Разработка алгоритмов обработки данных

Наиболее значимыми для системы являются алгоритмы автоматического поиска объектов недвижимости, нормализации полученной информации, фильтрации нерелевантных записей и расчета степени соответствия объектов требованиям клиента. Указанные алгоритмы обеспечивают преобразование данных из внешних источников в структурированный результат, пригодный для дальнейшей работы риелтора. В рамках проекта целесообразно выделить следующие основные алгоритмы:

- алгоритм поиска объектов недвижимости по профилю клиента;
- алгоритм нормализации объявления;
- алгоритм фильтрации объектов по городу Краснодару;
- алгоритм расчета процента совпадения объекта с параметрами клиента;

### 2.6.1 Разработка алгоритма поиска объектов недвижимости по профилю клиента

Алгоритм поиска объектов недвижимости предназначен для автоматического формирования списка релевантных предложений после создания карточки клиента или при повторном запуске поиска. На вход алгоритма поступают контактные данные клиента, тип недвижимости и параметры его потребности. На выходе алгоритм формирует ранжированный список объектов, сохраненных в базе данных и связанных с конкретной заявкой клиента.

Последовательность работы алгоритма выглядит следующим образом. После создания карточки клиента backend-сервер обращается к модулю поиска и передает ему пакет данных, описывающий профиль клиента. Далее модуль поиска определяет, какие источники необходимо использовать: источники для квартир или источники для домов. После этого выполняется обращение к стартовым URL этих источников, загрузка HTML-страниц и попытка извлечения информации из JSON-LD и видимых карточек объявлений.

Затем каждая найденная карточка проходит этап нормализации. Из текста извлекаются цена, площадь, комнатность, район, адрес, этаж, тип отделки и другие характеристики. Если объект не имеет минимально достаточного набора данных, он исключается. После этого выполняется географическая фильтрация: в систему допускаются только объекты, относящиеся к городу Краснодару.

Далее оставшиеся объекты проходят обязательную проверку по бюджету и другим параметрам. Для валидных объектов рассчитывается процент совпадения, после чего результат сортируется по степени соответствия и возвращается backend-серверу. Backend сохраняет объекты в централизованную таблицу объектов, связывает их с клиентом и обновляет статус поиска.

### 2.6.2 Разработка алгоритма нормализации объявления

Алгоритм нормализации объявления предназначен для преобразования сырых данных из внешнего источника в единый формат объекта недвижимости. Внешние сайты могут возвращать заголовки, описания и изображения в разных форматах, поэтому без нормализации невозможно обеспечить единообразный вид результата в базе данных и интерфейсе.

На первом этапе алгоритм получает текст карточки и дополнительные элементы, такие как JSON-LD, изображения, адресные фрагменты и служебные поля. Из полученного текста извлекаются базовые характеристики: цена, площадь, количество комнат, район, адрес, тип отделки, этаж и этажность, а для домов — площадь участка, количество спален, материал и коммуникации.

Для части количественных характеристик может использоваться расчетный способ. Например, удельная стоимость объекта определяется по формуле

`C_{м²} = C / S`,                                                     (1)

где `C_{м²}` — стоимость одного квадратного метра, `C` — полная стоимость объекта, `S` — общая площадь объекта. Использование формулы (1) позволяет получить сопоставимый показатель даже в тех случаях, когда источник не передает стоимость одного квадратного метра в явном виде.

Далее заголовок проверяется на наличие служебного или маркетингового мусора. Если сайт вернул некорректный заголовок, например общий заголовок страницы, вместо него формируется краткое структурированное название на основе реальных характеристик объекта. Аналогичная логика применяется к описанию: если описание содержит служебные слова, телефоны, бессмысленные фрагменты или большое количество шумовых данных, оно отбрасывается, а затем создается краткий запасной текст на основе основных характеристик объекта.

После этого выполняется фильтрация изображений. Из выдачи исключаются логотипы, иконки, служебные изображения и технические SVG-файлы. В результате алгоритм формирует чистый нормализованный объект, содержащий только полезную информацию.

### 2.6.4 Разработка алгоритма расчета процента совпадения

После прохождения обязательных фильтров каждый объект недвижимости должен быть оценен с точки зрения соответствия параметрам клиента. Для этого используется алгоритм расчета процента совпадения. Его назначение состоит в том, чтобы помочь риелтору быстрее выявить наиболее релевантные предложения.

Для квартир алгоритм учитывает следующие параметры:

- соответствие бюджета;
- соответствие комнатности;
- соответствие площади;
- совпадение района;
- соответствие сроку сдачи;
- соответствие отделке.

Для домов учитываются:

- соответствие бюджета;
- соответствие площади дома;
- соответствие площади участка;
- совпадение локации;
- наличие требуемых коммуникаций;
- соответствие числу этажей и спален.

Каждому критерию присваивается определенный вес. При выполнении условия объект получает соответствующую долю баллов. Если критерий нарушен, система фиксирует причину несоответствия. В результате формируются итоговый процент совпадения, а также перечни причин совпадения и несоответствия.

Формализованно итоговый процент совпадения может быть представлен в виде взвешенной оценки

`M = (Σ(w_i * k_i) / Σw_i) * 100%`,                                   (2)

где `M` — итоговый процент совпадения объекта требованиям клиента, `w_i` — вес `i`-го критерия, `k_i` — коэффициент выполнения `i`-го критерия, принимающий значение в диапазоне от `0` до `1`, `n` — количество оцениваемых критериев.

Для частных критериев, допускающих долевую оценку, может использоваться дополнительная формула. Например, для оценки совпадения по коммуникациям у домов применяется выражение

`k_{комм} = n_{совп} / n_{треб}`,                                     (3)

где `k_{комм}` — коэффициент совпадения по коммуникациям, `n_{совп}` — число коммуникаций, фактически найденных у объекта, `n_{треб}` — число коммуникаций, указанных в запросе клиента.

## 2.7 Проектирование интерфейса пользователя

Пользовательский интерфейс системы должен обеспечивать быстрый доступ к основным действиям, минимальное число лишних переходов и наглядное отображение состояния клиентской заявки. При проектировании интерфейса учитывалась работа пользователя в браузере и необходимость регулярного перехода между обзорными экранами и детальными карточками клиентов.

Интерфейс системы реализован как набор основных экранных форм, логически связанных между собой. Вход в систему осуществляется через отдельную страницу авторизации. После успешного входа пользователь попадает на главную страницу с общей сводкой. Далее ему доступны раздел списка клиентов, форма создания нового клиента, карточка клиента, а также вспомогательные страницы профиля, настроек и справочной информации.

Страница входа должна обеспечивать контролируемый доступ к системе и содержать форму ввода учетных данных.

Главная страница должна предоставлять сводную информацию по текущей работе. На ней требуется отобразить число клиентов в работе, число найденных объектов, число объектов в подборках и число подборок, готовых к отправке. Дополнительно рекомендуется разместить график активности и список клиентов, требующих внимания.

Страница списка клиентов предназначена для поиска и выбора нужной карточки. На ней размещаются фильтры, строка поиска и таблица со статусами клиентов.

Форма создания клиента является одной из наиболее важных экранных форм. В ней необходимо обеспечить ввод персональных данных клиента и параметров поиска недвижимости. Так как набор параметров для квартир и домов различается, в интерфейсе должно быть предусмотрено переключение между режимами ввода.

Карточка клиента является центральным экраном системы. На ней должны отображаться:

- основные сведения о клиенте;
- параметры поиска;
- число найденных объектов;
- состав подборки;
- список найденных объектов;
- текущий статус клиентской заявки.

Внутри карточки клиента должна быть предусмотрена возможность открытия подробной карточки объявления без потери контекста работы с заявкой.

С точки зрения визуального проектирования интерфейс должен быть достаточно строгим и рабочим, без лишних декоративных элементов. Важные действия, такие как запуск поиска, добавление в подборку и формирование текста сообщения, должны быть выделены достаточно заметно. При этом второстепенные действия должны оставаться доступными, но не перегружать пользователя.

| Повторный запуск поиска | Запуск поиска из карточки клиента | Обновляется история поиска, найденные объекты сохраняются в базе |
| Добавление объекта в подборку | Выбор найденного объекта в карточке клиента | Объект попадает в подборку клиента |
| Формирование текста сообщения | Генерация текста по собранной подборке | Система формирует готовое сообщение для отправки |
| Отметка об отправке | Подтверждение факта отправки клиенту | Статус заявки изменяется на соответствующее состояние |

Подробные тест-кейсы и результаты ручной проверки приведены в приложении Г. Проведенное интеграционное тестирование подтверждает корректность взаимодействия между всеми программными модулями системы.

## 3.5 Разработка эксплуатационной документации

### 3.5.1 Руководство пользователя

Руководство пользователя должно описывать порядок входа в систему, создание карточки клиента, запуск поиска, просмотр найденных объектов и формирование подборки. Основные сведения для пользователя приведены в приложении Д.

Пользователь начинает работу с входа в систему по логину и паролю преднастроенной учетной записи риелтора. После авторизации открывается главная страница, содержащая сводные показатели текущей работы.

Далее пользователь переходит к созданию клиентской заявки, указывает контактные данные и параметры поиска недвижимости. После сохранения карточки система выполняет автоматический запуск поиска и отображает найденные результаты в карточке клиента.

По итогам просмотра результатов пользователь может выбрать подходящие объекты, сформировать подборку и подготовить текст сообщения для дальнейшей отправки клиенту. После отправки подборка может быть отмечена как отправленная.

В рамках проекта для развертывания системы используются `Docker Desktop`, `Docker Compose`, `PostgreSQL` и средство просмотра базы данных `TablePlus`.

# Заключение

В ходе выполнения выпускной квалификационной работы были разработаны программные модули системы `Тэона`, предназначенные для автоматизации поиска недвижимости из внешних источников и интеграции результатов такого поиска с модулями управления потребностями клиента. В рамках работы была сформирована архитектура решения, включающая пользовательский интерфейс, backend API, отдельный модуль поиска и реляционную базу данных.

В результате проектирования были определены основные сценарии использования системы, разработана модель данных, отражающая структуру клиентских заявок и результатов поиска, а также формализованы ключевые алгоритмы обработки данных. Особое внимание было уделено алгоритмам поиска объектов по внешним источникам, нормализации объявлений, фильтрации объектов по географии и расчету степени соответствия объекта требованиям клиента.

В рамках программной реализации были созданы:

- frontend-модуль для работы риелтора в браузере;
- backend API-модуль для бизнес-логики и работы с PostgreSQL;
- Python-модуль поиска для обработки внешних сайтов недвижимости;
- схема базы данных и инфраструктура развертывания через Docker Compose.

Проведенное модульное и интеграционное тестирование подтвердило работоспособность основных сценариев системы и корректность наиболее значимых алгоритмов. Подготовленная эксплуатационная документация обеспечивает возможность запуска, использования и сопровождения программного продукта.

Практическая ценность выполненной работы заключается в том, что разработанная система позволяет сократить время ручного поиска недвижимости, централизовать хранение клиентских данных и обеспечить удобное формирование подборки объектов для клиента. Тем самым проект решает важную прикладную задачу автоматизации работы риелтора и может рассматриваться как основа для дальнейшего развития программного решения в области сопровождения сделок с недвижимостью.

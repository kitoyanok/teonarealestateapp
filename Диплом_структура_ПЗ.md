# Введение

Современный рынок недвижимости характеризуется большим объемом предложений и высокой скоростью их обновления. В процессе работы с клиентом риелтору требуется фиксировать параметры потребности, анализировать несколько внешних источников, отбирать подходящие объекты и подготавливать итоговую подборку. При ручной организации такой работы возрастают временные затраты, усложняется повторная обработка заявки и повышается риск потери информации.

Актуальность темы выпускной квалификационной работы обусловлена необходимостью автоматизации поиска недвижимости по внешним источникам и централизованного хранения клиентских заявок. Использование специализированной программной системы позволяет сократить объем ручных операций, ускорить подбор объектов и повысить удобство сопровождения клиента.

Объектом автоматизации является деятельность риелтора, связанная с приемом клиентской заявки, фиксацией параметров поиска, подбором объектов недвижимости и формированием итоговой подборки. Предметом разработки являются программные модули, обеспечивающие поиск недвижимости из внешних источников и интеграцию результатов поиска с модулем управления потребностями клиента.

Целью выпускной квалификационной работы является разработка программных модулей системы, предназначенной для хранения клиентских заявок, автоматизированного поиска объектов недвижимости, сохранения результатов и формирования подборки для клиента. Практическая значимость работы заключается в создании прикладного программного решения, которое может использоваться как единое рабочее пространство риелтора при сопровождении заявок на покупку недвижимости в городе Краснодаре.

# 1 Назначение и цели разработки

Разрабатываемые программные модули предназначены для автоматизации работы риелтора при обработке клиентских заявок, связанных с покупкой недвижимости. Основным пользователем системы является риелтор, который вводит данные клиента, задает параметры поиска, просматривает найденные объекты и формирует подборку для дальнейшей отправки.

Назначение разработки состоит в создании программного решения, объединяющего хранение клиентских заявок, учет параметров поиска и автоматизированный подбор объектов из внешних источников. Система должна обеспечивать единое информационное пространство для сопровождения клиентской заявки.

Программа должна поддерживать ввод и хранение клиентских данных, запуск поиска по внешним сайтам, сохранение результатов, отображение найденных объектов и формирование подборки. Целью разработки является повышение эффективности работы риелтора за счет сокращения ручных операций и упрощения дальнейшей работы с результатами поиска.

Сущность `clients` представляет основную карточку клиента. В ней содержатся ФИО, телефон, электронная почта, предпочитаемый канал связи, служебный статус заявки, тип интересующей недвижимости и комментарий риелтора. Каждая запись клиента принадлежит конкретному пользователю системы.

Для домов дополнительно сохраняются параметры площади дома, площади участка, материала, количества спален и наличия коммуникаций. Сущность `properties` представляет централизованное хранилище всех найденных объектов недвижимости. Здесь сохраняются источник, ссылка на объявление, заголовок, описание, район, адрес, площадь, комнатность, цена, изображения и другие характеристики. Централизованное хранение позволяет не дублировать один и тот же объект при повторных поисках.

Сущность `client_found_properties` связывает конкретного клиента с конкретным найденным объектом. Здесь дополнительно фиксируются процент совпадения, причины совпадения и причины несоответствия. Сущность `share_messages` предназначена для хранения подготовленного текста сообщения с подборкой объектов. Это позволяет фиксировать факт формирования сообщения и хранить историю таких действий.

Сущность `search_runs` хранит информацию о каждом запуске поиска. В таблице фиксируются статус выполнения, количество найденных объектов, количество сохраненных результатов, возможная ошибка, а также время начала и окончания обработки.

## 2.6 Разработка алгоритмов обработки данных

Наиболее значимыми для системы являются алгоритмы автоматического поиска объектов недвижимости, нормализации полученной информации, фильтрации нерелевантных записей и расчета степени соответствия объектов требованиям клиента. Указанные алгоритмы обеспечивают преобразование данных из внешних источников в структурированный результат, пригодный для дальнейшей работы риелтора. Описание блок-схем алгоритмов рекомендуется оформлять в соответствии с ГОСТ 19.701-90 [6].

В рамках проекта целесообразно выделить следующие основные алгоритмы:

- алгоритм поиска объектов недвижимости по профилю клиента;
- алгоритм нормализации объявления;
- алгоритм фильтрации объектов по городу Краснодару;
- алгоритм расчета процента совпадения объекта с параметрами клиента;
- алгоритм формирования подборки и текста сообщения.

### 2.6.1 Разработка алгоритма поиска объектов недвижимости по профилю клиента

Алгоритм поиска объектов недвижимости предназначен для автоматического формирования списка релевантных предложений после создания карточки клиента или при повторном запуске поиска. На вход алгоритма поступают контактные данные клиента, тип недвижимости и параметры его потребности. На выходе алгоритм формирует ранжированный список объектов, сохраненных в базе данных и связанных с конкретной заявкой клиента.

Последовательность работы алгоритма выглядит следующим образом. После создания карточки клиента backend-сервер обращается к модулю поиска и передает ему пакет данных, описывающий профиль клиента. Далее модуль поиска определяет, какие источники необходимо использовать: источники для квартир или источники для домов. После этого выполняется обращение к стартовым URL этих источников, загрузка HTML-страниц и попытка извлечения информации из JSON-LD и видимых карточек объявлений.

Затем каждая найденная карточка проходит этап нормализации. Из текста извлекаются цена, площадь, комнатность, район, адрес, этаж, тип отделки и другие характеристики. Если объект не имеет минимально достаточного набора данных, он исключается. После этого выполняется географическая фильтрация: в систему допускаются только объекты, относящиеся к городу Краснодару.

Далее оставшиеся объекты проходят обязательную проверку по бюджету и другим параметрам. Для валидных объектов рассчитывается процент совпадения, после чего результат сортируется по степени соответствия и возвращается backend-серверу. Backend сохраняет объекты в централизованную таблицу объектов, связывает их с клиентом и обновляет статус поиска.

### 2.6.2 Разработка алгоритма нормализации объявления

Алгоритм нормализации объявления предназначен для преобразования сырых данных из внешнего источника в единый формат объекта недвижимости. Внешние сайты могут возвращать заголовки, описания и изображения в разных форматах, поэтому без нормализации невозможно обеспечить единообразный вид результата в базе данных и интерфейсе.

На первом этапе алгоритм получает текст карточки и дополнительные элементы, такие как JSON-LD, изображения, адресные фрагменты и служебные поля. Из полученного текста извлекаются базовые характеристики: цена, площадь, количество комнат, район, адрес, тип отделки, этаж и этажность, а для домов — площадь участка, количество спален, материал и коммуникации.

Далее заголовок проверяется на наличие служебного или маркетингового мусора. Если сайт вернул некорректный заголовок, например общий заголовок страницы, вместо него формируется краткое структурированное название на основе реальных характеристик объекта. Аналогичная логика применяется к описанию: если описание содержит служебные слова, телефоны, бессмысленные фрагменты или большое количество шумовых данных, оно отбрасывается, а затем создается краткий запасной текст на основе основных характеристик объекта.

После этого выполняется фильтрация изображений. Из выдачи исключаются логотипы, иконки, служебные изображения и технические SVG-файлы. В результате алгоритм формирует чистый нормализованный объект, содержащий только полезную информацию.

Алгоритм получает текст объявления, адресные фрагменты и URL-адрес страницы. На первом этапе он проверяет наличие прямых указаний на город Краснодар. Если объект содержит указание на Краснодар или на один из районов Краснодара, он признается допустимым и проходит в следующий этап.

Использование такого алгоритма позволяет исключить попадание в выдачу заведомо нерелевантных объектов и повысить доверие к результатам поиска.

### 2.6.4 Разработка алгоритма расчета процента совпадения

После прохождения обязательных фильтров каждый объект недвижимости должен быть оценен с точки зрения соответствия параметрам клиента. Для этого используется алгоритм расчета процента совпадения. Его назначение состоит в том, чтобы помочь риелтору быстрее выявить наиболее релевантные предложения.

Для квартир алгоритм учитывает следующие параметры:

- соответствие бюджета;
- соответствие комнатности;
- соответствие площади;
- совпадение района;
- соответствие сроку сдачи;
- соответствие отделке.

Для домов учитываются:

- соответствие бюджета;
- соответствие площади дома;
- соответствие площади участка;
- совпадение локации;
- наличие требуемых коммуникаций;
- соответствие числу этажей и спален.

Каждому критерию присваивается определенный вес. При выполнении условия объект получает соответствующую долю баллов. Если критерий нарушен, система фиксирует причину несоответствия. В результате формируются итоговый процент совпадения, а также перечни причин совпадения и несоответствия.

### Состав элементов блок-схемы

- выбор режима оценки: квартира или дом;
- сравнение по каждому критерию;
- начисление баллов;
- фиксация причин совпадения;
- фиксация причин несоответствия;
- ограничение результата в диапазоне 0–100;
- возврат готовой оценки.

### 2.6.5 Разработка алгоритма формирования подборки и текста сообщения

Заключительным алгоритмом, имеющим важное прикладное значение, является алгоритм формирования подборки и текста сообщения для клиента. После того как автоматический поиск завершен, риелтор должен вручную отобрать наиболее подходящие объекты. Выбранные объекты сохраняются в отдельной подборке, а на их основе система формирует человекочитаемое сообщение.

Алгоритм начинает работу с просмотра найденных объектов и ручного выбора подходящих позиций. Далее для каждого выбранного объекта подготавливается краткое описание, включающее название, стоимость, площадь, район и ссылку на источник. После объединения этих элементов формируется итоговый текст, который риелтор может скопировать и отправить клиенту в удобном канале связи.

Такой алгоритм соединяет автоматическую обработку данных и ручную экспертную работу риелтора, что делает систему удобной в реальном рабочем процессе.

Интерфейс системы реализован как набор основных экранных форм, логически связанных между собой. Вход в систему осуществляется через отдельную страницу авторизации. # 3 Реализация

## 3.1 Обоснование выбора средств разработки и методов разработки

Выбор средств разработки определялся прикладным характером программного продукта, требованием к воспроизводимому развертыванию системы и необходимостью разделения пользовательской, серверной и поисковой логики. В качестве основы клиентской части был выбран стек React, TypeScript и Vite [7; 8; 9], обеспечивающий разработку интерактивного браузерного интерфейса.

Для серверной части были выбраны Node.js и Express [10; 11]. Данное решение позволяет реализовать прикладной API, маршруты работы с клиентскими заявками, авторизацию пользователя и взаимодействие с базой данных в рамках единого серверного слоя. Для модуля поиска был выбран Python с использованием FastAPI и библиотек HTML-парсинга [13; 14], поскольку эта среда хорошо подходит для обработки внешних интернет-источников, нормализации текстовых данных и реализации алгоритмов сопоставления.

В качестве СУБД была выбрана PostgreSQL [12], а для контейнеризации и локального разворачивания решения использованы Docker и Docker Compose [15; 16]. Такой набор средств разработки обеспечивает модульность архитектуры, повторяемость запуска и удобство дальнейшего сопровождения программной системы.

## 3.2 Разработка базы данных в среде СУБД

Для практической реализации базы данных в рамках проекта была выбрана реляционная система управления базами данных PostgreSQL. Выбор данной СУБД обусловлен поддержкой транзакций, надежной работой с ограничениями целостности, удобством хранения структурированных данных и возможностью дальнейшего масштабирования [12].

Физическая структура базы данных создается SQL-скриптом инициализации, расположенным в проекте. Данный скрипт формирует таблицы пользователей, клиентов, профилей поиска, найденных объектов, подборок, истории запуска поиска и подготовленных сообщений. Кроме создания таблиц в нем задаются внешние ключи, ограничения уникальности и индексы, ускоряющие выполнение типовых запросов.

Отдельного внимания заслуживает то, что база данных подключена к системе через Docker Compose и использует постоянный том хранения [15; 16]. Это позволяет обеспечить сохранность данных после перезапуска контейнеров. Таким образом, база данных функционирует не как временное хранилище, а как постоянный уровень хранения прикладной информации.

При реализации таблицы `users` были предусмотрены поля авторизации и базовой идентификации риелтора. Для хранения карточек клиентов была реализована таблица `clients`, связанная с таблицей пользователей отношением «один ко многим». Для параметров поиска была создана отдельная таблица `client_search_profiles`, что позволило отделить контактные данные клиента от его поисковой потребности. Такой подход улучшает читаемость модели и упрощает дальнейшую модификацию структуры.

Таблица `properties` была реализована как единое хранилище найденных объектов недвижимости. Данный подход выгоден тем, что исключает повторное дублирование карточек объекта при нескольких поисковых запусках. Для связи конкретного клиента с конкретным найденным объектом используется таблица `client_found_properties`, в которой дополнительно сохраняются значение процента совпадения и объясняющие причины.

Для хранения пользовательской подборки была реализована таблица `shortlist_items`, а для хранения текстов, формируемых системой перед отправкой клиенту, — таблица `share_messages`. История запусков поиска хранится в таблице `search_runs`. Это позволяет проследить, когда выполнялся поиск, успешно ли он завершился и сколько объектов было найдено и сохранено.

Для обеспечения производительности в схеме были предусмотрены индексы по полям, наиболее часто используемым при фильтрации и соединении данных. К таким полям относятся идентификатор риелтора у клиентов, статус клиентской заявки, ссылка на объект недвижимости, идентификатор клиента в таблицах найденных объектов, подборок и запусков поиска.

## 3.3 Программная реализация модулей в среде программирования

Программная реализация системы выполнена по модульному принципу. Архитектурно решение разделено на три основные части: клиентскую часть, серверную часть и отдельный сервис поиска. Такой подход позволяет разграничить ответственность между модулями, упростить сопровождение системы и обеспечить независимое развитие каждого слоя.

Клиентская часть реализована на React с использованием TypeScript и сборщика Vite [7; 8; 9]. Она отвечает за отображение интерфейса риелтора, маршрутизацию между страницами, отправку запросов на backend и отрисовку результатов. Серверная часть реализована на Node.js с использованием Express и TypeScript [10; 11]. Ее задачи заключаются в обработке запросов frontend, выполнении бизнес-логики, работе с PostgreSQL и запуске внешнего поискового модуля. Модуль поиска реализован на Python с использованием FastAPI и библиотек для парсинга HTML [13; 14].

### Реализация клиентской части

Клиентская часть включает в себя страницы входа, dashboard, списка клиентов, создания клиента, карточки клиента, страницы профиля, настроек и помощи. Основная маршрутизация приложения организована в корневом модуле frontend. Для защищенных страниц применяется проверка факта авторизации пользователя.

В форме создания клиента реализована логика ввода параметров поиска недвижимости. В зависимости от выбранного типа объекта — квартира или дом — форма отображает разный набор полей. После отправки формы frontend формирует структурированный запрос и передает его backend-серверу.

Особое значение имеет экран карточки клиента. Именно здесь объединяются результаты поиска, подборка и история запусков. Для каждого найденного объекта в интерфейсе отображаются краткие характеристики, цена, степень соответствия и действия риелтора. При необходимости пользователь может открыть подробную информацию об объекте и вручную включить его в подборку.

### Реализация серверной части

Серверная часть реализует авторизацию, работу с клиентскими карточками, хранение параметров поиска, получение dashboard-статистики и формирование подборки. Для проверки доступа используется JWT-сессия, передаваемая через защищенную cookie. После успешного входа backend определяет пользователя по логину и сверяет хэш пароля.

Работа с базой данных организована через SQL-слой, где реализованы запросы создания, чтения, изменения и удаления данных. Такой подход обеспечивает прозрачный контроль над структурой запросов и связями между таблицами.

При создании клиента backend не ограничивается простым сохранением карточки. Он также формирует профиль поиска и сразу же запускает модуль поиска. После получения результатов выполняется сохранение объектов недвижимости, а затем фиксируется связь между клиентом и найденными объектами.

### Реализация модуля поиска

Поисковый модуль работает как отдельный HTTP-сервис. Backend передает в него структурированный запрос, содержащий параметры поиска клиента. После этого сервис определяет нужный набор источников, выполняет обход сайтов, извлекает объявления из JSON-LD и HTML-структур, фильтрует результаты и рассчитывает степень соответствия.

Для нормализации данных используется базовый адаптер, в котором сосредоточена общая логика очистки заголовков, описаний, изображений и адресных данных. Отдельное внимание уделено фильтрации по географии, так как проект должен отбирать только объекты города Краснодара. После нормализации выполняется расчет степени соответствия объекту требованиям клиента.

### 3.5.1 Руководство пользователя

Разрабатываемая система предназначена для практического использования риелтором, поэтому важной частью пояснительной записки является описание порядка работы с приложением. Руководство пользователя должно объяснять, как войти в систему, как создать карточку клиента, как запустить поиск и как сформировать подборку.

### Текст для раздела

Для начала работы пользователь должен открыть приложение в браузере и перейти на страницу входа. После этого необходимо ввести логин и пароль одной из преднастроенных учетных записей риелторов. После успешного входа система перенаправляет пользователя на главную страницу, где отображаются сводные показатели текущей работы.

В карточке клиента пользователь может просматривать найденные объекты, открывать подробную информацию по каждому из них и вручную добавлять наиболее подходящие объекты в подборку. После отправки сообщения пользователь может отметить подборку как отправленную. Это позволяет системе корректно отразить состояние клиентской заявки и учитывать его в общей статистике.

```ts
export async function runClientSearch(clientId: string, realtorId: string) {
  const client = (await getClientForSearch(clientId, realtorId)) as SearchClient | null;
  if (!client || !client.searchProfile) {
    throw new Error("Client search profile not found");
  }

  const searchRun = (await createSearchRun(client.id, client.propertyType)) as SearchRunRef;
  await updateClientStatus(client.id, "searching");

  try {
    const response = await fetch(`${config.searchServiceUrl}/search`, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify(buildSearchRequest(client, client.searchProfile))
    });

    if (!response.ok) {
      throw new Error(`Search service returned ${response.status}`);
    }

    const payload = (await response.json()) as SearchResponse;
    let saved = 0;
    for (const item of payload.items.filter((property) => property.sourceUrl)) {
      const property = (await upsertProperty(cleanItem(item))) as PropertyRef;
      await upsertClientFoundProperty(client.id, property.id, item);
      saved += 1;
    }

    const status = saved > 0 ? "found" : "no_results";
    await finishSearchRun(searchRun.id, payload.totalFound, saved);
    await updateClientStatus(client.id, status);
    return { totalFound: payload.totalFound, totalSaved: saved, status };
  } catch (error) {
    const message = error instanceof Error ? error.message : "Unknown search error";
    await failSearchRun(searchRun.id, message);
    await updateClientStatus(client.id, "error");
    return { totalFound: 0, totalSaved: 0, status: "error", error: message };
  }
}
```

Листинг Б.3 показывает связь между карточкой клиента, поисковым сервисом и уровнем хранения данных. Именно этот фрагмент реализует основной прикладной цикл автоматизированного подбора объектов.

# Приложение Д

**Основные разделы руководства пользователя**

## Д.1 Назначение программы

Программная система `Тэона` предназначена для автоматизации работы риелтора при обработке клиентских заявок на покупку недвижимости. Система позволяет хранить данные клиента, сохранять параметры поиска, автоматически подбирать объекты по внешним источникам, формировать подборку и подготавливать текст сообщения для последующей отправки клиенту.

## Д.2 Условия выполнения программы

Для работы программы необходимо:

- развернуть систему через Docker Compose;
- убедиться в доступности web-интерфейса, backend API и search-service;
- использовать один из поддерживаемых современных браузеров;
- иметь данные для входа под одной из заранее подготовленных учетных записей риелторов.

После запуска приложение доступно по адресу `http://localhost:5002`.

## Д.3 Порядок запуска программы

Для запуска программного продукта необходимо выполнить следующие действия:

Действие №1. Скопировать файл `.env.example` в `.env`.  
Действие №2. Установить зависимости проекта командой `npm install`.  
Действие №3. Выполнить команду `npm run prod:up`.  
Действие №4. Дождаться запуска контейнеров `postgres`, `search-service`, `api` и `web`.  
Действие №5. Открыть в браузере адрес `http://localhost:5002`.

Для остановки системы используется команда `npm run prod:down`.

## Д.4 Вход пользователя в систему

После открытия web-приложения пользователь переходит на страницу авторизации. Для входа необходимо ввести логин и пароль одной из заранее созданных учетных записей риелторов. В учебной версии системы используются следующие логины:

- `ivan.nikitin`;
- `kirill.nabiev`;
- `ilya.berezin`;
- `marina.nikiforova`;
- `nikita.zubach`.

Общий пароль для входа: `Teona2026!`.

После успешной авторизации система перенаправляет пользователя на главную страницу со сводными показателями.

## Д.5 Создание карточки клиента

Для создания новой карточки клиента пользователь должен перейти в раздел добавления клиента и заполнить форму.

В форме указываются:

- ФИО клиента;
- телефон;
- электронная почта;
- канал связи;
- тип недвижимости;
- параметры поиска.

После заполнения формы выполняются следующие действия:

Действие №1. Пользователь нажимает кнопку сохранения.  
Действие №2. Система создает карточку клиента и сохраняет профиль поиска.  
Действие №3. Система автоматически запускает поиск объектов.  
Действие №4. После завершения поиска пользователь перенаправляется в карточку клиента.

## Д.6 Работа с найденными объектами

В карточке клиента отображается список найденных объектов недвижимости. Для каждого объекта выводятся:

- краткое название;
- стоимость;
- площадь;
- комнатность или характеристики дома;
- район;
- процент совпадения.

Пользователь может открыть подробную карточку объявления, чтобы просмотреть расширенное описание, ссылку на источник и дополнительные признаки соответствия параметрам клиента.

## Д.7 Формирование подборки

После просмотра результатов поиска пользователь может вручную отобрать наиболее подходящие объекты.

Для формирования подборки выполняются следующие действия:

Действие №1. Открыть карточку клиента.  
Действие №2. Просмотреть найденные объекты.  
Действие №3. Нажать кнопку добавления объекта в подборку для нужных объявлений.  
Действие №4. Перейти к блоку подборки и убедиться, что выбранные объекты сохранены.

После добавления объектов система изменяет внутреннее состояние заявки и позволяет перейти к формированию текста сообщения.

## Д.8 Формирование текста сообщения

При наличии в подборке хотя бы одного объекта пользователь может сформировать текст сообщения.

Действие №1. Нажать кнопку формирования текста.  
Действие №2. Дождаться генерации сообщения.  
Действие №3. Скопировать текст и использовать его для отправки клиенту в выбранном канале связи.

Сформированный текст содержит:

- обращение к клиенту;
- нумерованный перечень объектов;
- основные характеристики вариантов;
- ссылки на источники объявлений.

## Д.9 Подтверждение отправки подборки

После фактической отправки сообщения клиенту пользователь должен отметить подборку как отправленную.

Действие №1. Выполнить отправку сообщения вне системы.  
Действие №2. Нажать кнопку подтверждения отправки.  
Действие №3. Убедиться, что статус клиентской заявки изменился.

Фиксация данного действия необходима для корректного отображения статистики и текущего состояния работы с заявкой.

## Д.10 Возможные ошибки пользователя

В процессе работы с системой возможны следующие типовые ситуации:

- не заполнены обязательные поля формы клиента;
- введен неверный логин или пароль;
- поиск не дал результатов;
- модуль поиска временно недоступен;
- пользователь пытается сформировать сообщение при пустой подборке.

В указанных случаях система должна выводить понятное сообщение об ошибке и не допускать нарушения целостности данных.

## 7. Построение диаграммы состояний

### Название рисунка

`Рисунок 7 – Диаграмма состояний клиентской заявки`

### Объект, для которого строится диаграмма

- `Клиентская заявка`

### Состояния

- `new` – новая заявка.
- `searching` – идет поиск объектов.
- `found` – найдены объекты.
- `no_results` – подходящие объекты не найдены.
- `shortlist_ready` – подборка сформирована.
- `sent` – подборка отправлена клиенту.
- `error` – произошла ошибка поиска или обработки.

### Переходы между состояниями

- `new -> searching`
  - при сохранении клиента и запуске поиска.

- `searching -> found`
  - если система получила и сохранила хотя бы один объект.

- `searching -> no_results`
  - если поиск завершился без результатов.

- `searching -> error`
  - если модуль поиска вернул ошибку или недоступен.

- `found -> shortlist_ready`
  - если риелтор добавил хотя бы один объект в подборку.

- `shortlist_ready -> found`
  - если риелтор удалил из подборки все объекты, но найденные результаты остались.

- `shortlist_ready -> sent`
  - если риелтор сформировал сообщение и отметил подборку как отправленную.

### Поясняющий текст к рисунку

На рисунке 7 представлена диаграмма состояний клиентской заявки. Она отражает жизненный цикл карточки клиента внутри системы: от момента создания до формирования подборки и фиксации факта отправки. Использование диаграммы состояний позволяет показать, какие состояния являются допустимыми и при каких действиях пользователя или внутренних событиях системы выполняется переход между ними.

### Как именно разнести связи по фрагментам

#### Фрагмент 1: `users`, `clients`, `client_search_profiles`

Связь:

- один пользователь системы может вести много клиентов.

Графически:

- `users (1) ---- (M) clients`

#### Фрагмент 2: `properties`, `client_found_properties`

Связи:

- один объект недвижимости может участвовать во множестве результатов поиска;
- одна клиентская заявка может иметь много найденных объектов;
- таблица `client_found_properties` хранит связь между клиентом и объектом, а также прикладные атрибуты релевантности.

#### Фрагмент 3: `shortlist_items`, `share_messages`, `search_runs`

Связи:

- один клиент может иметь много элементов подборки;
- один клиент может иметь много подготовленных сообщений;
- один клиент может иметь много запусков поиска;
- один объект может быть помещен в подборку разных клиентов, но связь с клиентом хранится в таблице `shortlist_items`.

### Рекомендуемый порядок расположения сущностей на рисунках

Сверху:

- `users`

Под ним:

- `clients`

Справа от `clients`:

- `client_search_profiles`

Отдельным вторым рисунком:

- `properties`
- `client_found_properties`

Отдельным третьим рисунком ниже `clients`:

- `shortlist_items`
- `share_messages`
- `search_runs`

Такое разбиение повышает читаемость модели данных, поскольку пользовательский контур, контур найденных объектов и контур служебной истории не смешиваются в пределах одного перегруженного рисунка.

Диаграмма отражает логику ручного формирования подборки объектов и подготовки итогового сообщения для клиента на основе автоматически найденных результатов.

### Под use case диаграммой аналитики и контроля заявок

Диаграмма отражает обзорный контур работы риелтора и показывает, как сводные показатели системы используются для контроля текущей загрузки и перехода к заявкам, требующим внимания.

### Под диаграммой деятельности

Диаграмма деятельности описывает полный поток обработки клиентской заявки при автоматическом поиске недвижимости по внешним источникам, включая этапы фильтрации, нормализации и сохранения результатов.

### Под диаграммой состояний

Диаграмма состояний отображает жизненный цикл клиентской заявки и позволяет зафиксировать допустимые переходы между состояниями в зависимости от действий пользователя и результатов работы системы.

### Под диаграммой компонентов

Диаграмма компонентов показывает архитектурную структуру программной системы и раскрывает взаимодействие между пользовательским интерфейсом, серверной логикой, модулем поиска, базой данных и внешними источниками данных.

### Под фрагментами ER-диаграммы

Фрагменты ER-диаграммы отражают логическую модель данных системы и позволяют раздельно показать пользовательский контур, контур найденных объектов и контур служебной истории без перегрузки одного рисунка большим числом сущностей.

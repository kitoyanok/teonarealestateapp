# Введение

Современный рынок недвижимости характеризуется большим объемом предложений и высокой скоростью их обновления. В процессе работы с клиентом риелтору требуется фиксировать параметры потребности, анализировать несколько внешних источников, отбирать подходящие объекты и подготавливать итоговую подборку. При ручной организации такой работы возрастают временные затраты, усложняется повторная обработка заявки и повышается риск потери информации.

Актуальность темы выпускной квалификационной работы обусловлена необходимостью автоматизации поиска недвижимости по внешним источникам и централизованного хранения клиентских заявок. Использование специализированной программной системы позволяет сократить объем ручных операций, ускорить подбор объектов и повысить удобство сопровождения клиента.

Объектом автоматизации является деятельность риелтора, связанная с приемом клиентской заявки, фиксацией параметров поиска, подбором объектов недвижимости и формированием итоговой подборки. Предметом разработки являются программные модули, обеспечивающие поиск недвижимости из внешних источников и интеграцию результатов поиска с модулем управления потребностями клиента.

Целью выпускной квалификационной работы является разработка программных модулей системы, предназначенной для хранения клиентских заявок, автоматизированного поиска объектов недвижимости, сохранения результатов и формирования подборки для клиента. Практическая значимость работы заключается в создании прикладного программного решения, которое может использоваться как единое рабочее пространство риелтора при сопровождении заявок на покупку недвижимости в городе Краснодаре.

# 1 Назначение и цели разработки

Разрабатываемые программные модули предназначены для автоматизации работы риелтора при сопровождении клиентских заявок на покупку недвижимости. Основным пользователем системы является риелтор, который фиксирует данные клиента, задает параметры поиска, просматривает найденные объекты и формирует итоговую подборку для последующей отправки.

Назначение разработки заключается в создании программного решения, объединяющего хранение карточек клиентов, учет параметров поиска, автоматизированный поиск объявлений по внешним интернет-источникам, сохранение результатов и подготовку сообщения для клиента. Использование единой системы должно сократить объем ручных операций, упростить повторную обработку заявок и повысить удобство ежедневной работы риелтора.

Программа должна обеспечивать ввод и хранение клиентских данных, запуск поиска по внешним сайтам недвижимости, фильтрацию результатов по городу Краснодару, отображение найденных объектов, ручное формирование подборки и подготовку текстового сообщения. Целью разработки является повышение эффективности работы риелтора за счет сокращения времени на поиск, централизации сведений о клиентах и сохранения всех промежуточных результатов в едином информационном пространстве.

## 2.6 Разработка алгоритмов обработки данных

Наиболее значимыми для системы являются алгоритмы автоматического поиска объектов недвижимости, нормализации полученной информации, фильтрации нерелевантных записей и расчета степени соответствия объектов требованиям клиента. Указанные алгоритмы обеспечивают преобразование данных из внешних источников в структурированный результат, пригодный для дальнейшей работы риелтора. Описание блок-схем алгоритмов выполняется в соответствии с ГОСТ 19.701-90 [6].

В рамках проекта целесообразно выделить следующие основные алгоритмы:

- алгоритм поиска объектов недвижимости по профилю клиента;
- алгоритм нормализации объявления;
- алгоритм фильтрации объектов по городу Краснодару;
- алгоритм расчета процента совпадения объекта с параметрами клиента;
- алгоритм формирования подборки и текста сообщения.

### 2.6.1 Разработка алгоритма поиска объектов недвижимости по профилю клиента

Алгоритм поиска объектов недвижимости предназначен для автоматического формирования списка релевантных предложений после создания карточки клиента или при повторном запуске поиска. На вход алгоритма поступают тип недвижимости и параметры потребности клиента, сохраненные в профиле поиска. На выходе алгоритм формирует ранжированный список объектов, сохраненных в базе данных и связанных с конкретной заявкой клиента.

Последовательность работы алгоритма выглядит следующим образом. После создания карточки клиента backend-сервер обращается к поисковому сервису и передает ему пакет данных, описывающий профиль клиента. Далее поисковый сервис определяет, какие источники необходимо использовать: источники для квартир или источники для домов. После этого выполняется формирование набора стартовых URL, загрузка HTML-страниц и попытка извлечения информации из JSON-LD и видимых карточек объявлений.

Затем каждая найденная карточка проходит этап предварительной проверки. Если объект не имеет минимально достаточного набора данных, не содержит цены или не имеет исходной ссылки, он исключается. После этого выполняется географическая фильтрация: в систему допускаются только объекты, относящиеся к городу Краснодару или его районам. Только после прохождения обязательных проверок объект передается в модуль нормализации.

Далее оставшиеся объекты проходят обязательную проверку по бюджету и другим параметрам. Для валидных объектов рассчитывается процент совпадения, а также формируются причины совпадения и несоответствия. Backend сохраняет объекты в централизованную таблицу `properties`, связывает их с клиентом через таблицу `client_found_properties`, фиксирует запуск поиска в таблице `search_runs` и обновляет состояние заявки.

### 2.6.2 Разработка алгоритма нормализации объявления

Алгоритм нормализации объявления предназначен для преобразования сырых данных из внешнего источника в единый формат объекта недвижимости. Внешние сайты возвращают заголовки, описания, адресные элементы и изображения в разных форматах, поэтому без нормализации невозможно обеспечить единообразный вид результата в базе данных и интерфейсе.

На первом этапе алгоритм получает текст карточки и дополнительные элементы, такие как JSON-LD, изображения, адресные фрагменты и служебные поля. Из полученного текста извлекаются базовые характеристики: цена, площадь, количество комнат, район, адрес, тип отделки, этаж и этажность, а для домов — площадь участка, количество спален, материал стен и коммуникации.

Для части количественных характеристик может использоваться расчетный способ. Например, удельная стоимость объекта определяется по формуле

`C_{м²} = C / S`,                                                     (1)

где `C_{м²}` — стоимость одного квадратного метра, `C` — полная стоимость объекта, `S` — общая площадь объекта. Использование формулы (1) позволяет получить сопоставимый показатель даже в тех случаях, когда источник не передает стоимость одного квадратного метра в явном виде.

Далее заголовок проверяется на наличие служебного или маркетингового мусора. Если сайт вернул некорректный заголовок, например общий заголовок страницы или рекламный блок, вместо него формируется краткое структурированное название на основе реальных характеристик объекта. Аналогичная логика применяется к описанию: если описание содержит служебные слова, телефоны, бессмысленные фрагменты или большое количество шумовых данных, оно отбрасывается, а затем создается краткий запасной текст на основе основных характеристик объекта.

После этого выполняется фильтрация изображений. Из выдачи исключаются логотипы, иконки, служебные изображения и технические SVG-файлы. В результате алгоритм формирует чистый нормализованный объект, содержащий только полезную информацию и пригодный для отображения в карточке клиента и детальном окне объекта.

### 2.6.4 Разработка алгоритма расчета процента совпадения

После прохождения обязательных фильтров каждый объект недвижимости должен быть оценен с точки зрения соответствия параметрам клиента. Для этого используется алгоритм расчета процента совпадения. Его назначение состоит в том, чтобы помочь риелтору быстрее выявить наиболее релевантные предложения.

Для квартир алгоритм учитывает следующие параметры:

- соответствие бюджета;
- соответствие комнатности;
- соответствие площади;
- совпадение района;
- соответствие сроку сдачи;
- соответствие отделке.

Для домов учитываются:

- соответствие бюджета;
- соответствие площади дома;
- соответствие площади участка;
- совпадение локации;
- наличие требуемых коммуникаций;
- соответствие числу этажей и спален.

Каждому критерию присваивается определенный вес. При выполнении условия объект получает соответствующую долю баллов. Для части критериев допускается частичное совпадение. Если критерий нарушен, система фиксирует причину несоответствия. В результате формируются итоговый процент совпадения, а также перечни причин совпадения и несоответствия.

Формализованно итоговый процент совпадения может быть представлен в виде взвешенной оценки

`M = (Σ(w_i * k_i) / Σw_i) * 100%`,                                   (2)

где `M` — итоговый процент совпадения объекта требованиям клиента, `w_i` — вес `i`-го критерия, `k_i` — коэффициент выполнения `i`-го критерия, принимающий значение в диапазоне от `0` до `1`, `n` — количество оцениваемых критериев.

Для частных критериев, допускающих долевую оценку, может использоваться дополнительная формула. Например, для оценки совпадения по коммуникациям у домов применяется выражение

`k_{комм} = n_{совп} / n_{треб}`,                                     (3)

где `k_{комм}` — коэффициент совпадения по коммуникациям, `n_{совп}` — число коммуникаций, фактически найденных у объекта, `n_{треб}` — число коммуникаций, указанных в запросе клиента.

`Рисунок 14 – Блок-схема алгоритма расчета процента совпадения объекта требованиям клиента`

### Состав элементов блок-схемы

- выбор режима оценки: квартира или дом;
- сравнение по каждому критерию;
- начисление баллов;
- фиксация причин совпадения;
- фиксация причин несоответствия;

### 2.6.5 Разработка алгоритма формирования подборки и текста сообщения

Заключительным алгоритмом, имеющим важное прикладное значение, является алгоритм формирования подборки и текста сообщения для клиента. После того как автоматический поиск завершен, риелтор вручную отбирает наиболее подходящие объекты. Выбранные объекты сохраняются в отдельной подборке, а на их основе система формирует человекочитаемое сообщение.

Алгоритм начинает работу с просмотра найденных объектов и ручного выбора подходящих позиций. Далее проверяется, что подборка не пуста и что для клиента задан канал связи. После этого для каждого выбранного объекта подготавливается краткое описание, включающее название, стоимость, площадь, район и ссылку на источник. После объединения этих элементов формируется итоговый текст, который риелтор может скопировать и отправить клиенту в удобном канале связи. Сформированный текст сохраняется в таблице `share_messages`, что позволяет фиксировать результат подготовки сообщения.

Такой алгоритм соединяет автоматическую обработку данных и ручную экспертную работу риелтора, что делает систему удобной в реальном рабочем процессе.

## 2.7 Проектирование интерфейса пользователя

Пользовательский интерфейс системы должен обеспечивать быстрый доступ к основным действиям, минимальное число лишних переходов и наглядное отображение состояния клиентской заявки. При проектировании интерфейса учитывалась работа пользователя в браузере и необходимость регулярного перехода между обзорными экранами и детальными карточками клиентов.

Интерфейс системы реализован как набор взаимосвязанных экранных форм. Вход в систему осуществляется через отдельную страницу авторизации. После успешного входа пользователь переходит на главную страницу с общей сводкой. Далее ему доступны раздел списка клиентов, форма создания нового клиента, карточка клиента и вспомогательные экраны профиля, настроек и справочной информации.

Страница входа должна обеспечивать контролируемый доступ к системе и содержать форму ввода учетных данных. Главная страница должна предоставлять сводную информацию по текущей работе: число клиентов в работе, число найденных объектов, число объектов в подборках и число подборок, готовых к отправке. Дополнительно на главной странице отображаются график активности и список клиентов, требующих внимания.

Страница списка клиентов предназначена для поиска и выбора нужной карточки. На ней размещаются строка поиска, фильтр по статусам и список карточек клиентов с краткой информацией о текущей заявке. Форма создания клиента является одной из наиболее важных экранных форм. В ней необходимо обеспечить ввод персональных данных клиента и параметров поиска недвижимости. Так как набор параметров для квартир и домов различается, в интерфейсе должно быть предусмотрено переключение между режимами ввода.

Карточка клиента является центральным экраном системы. На ней одновременно должны отображаться основные сведения о клиенте, параметры поиска, текущий статус заявки, число найденных объектов, состав подборки и список найденных объектов. Внутри карточки клиента должна быть предусмотрена возможность открытия подробной карточки объявления без потери контекста работы с заявкой.

С точки зрения визуального проектирования интерфейс должен быть достаточно строгим и рабочим, без лишних декоративных элементов. Важные действия, такие как запуск поиска, добавление в подборку и формирование текста сообщения, должны быть выделены достаточно заметно. ## 3.5 Разработка эксплуатационной документации

### 3.5.1 Руководство пользователя

Руководство пользователя описывает порядок входа в систему, создание карточки клиента, запуск поиска, просмотр найденных объектов и формирование подборки. Основные сведения для пользователя приведены в приложении Д.

Пользователь начинает работу с входа в систему по логину и паролю преднастроенной учетной записи риелтора. После авторизации открывается главная страница, содержащая сводные показатели текущей работы.

Далее пользователь переходит к созданию клиентской заявки, указывает контактные данные и параметры поиска недвижимости. После сохранения карточки система выполняет автоматический запуск поиска и отображает найденные результаты в карточке клиента.

По итогам просмотра результатов пользователь может выбрать подходящие объекты, сформировать подборку и подготовить текст сообщения для дальнейшей отправки клиенту. После отправки подборка может быть отмечена как отправленная.

# Заключение

В ходе выполнения выпускной квалификационной работы были разработаны программные модули системы `Тэона`, предназначенные для автоматизации поиска недвижимости из внешних источников и интеграции результатов такого поиска с модулями управления потребностями клиента. В рамках работы была сформирована архитектура решения, включающая пользовательский интерфейс, backend API, отдельный модуль поиска и реляционную базу данных.

В результате проектирования были определены основные сценарии использования системы, разработана модель данных, отражающая структуру клиентских заявок и результатов поиска, а также формализованы ключевые алгоритмы обработки данных. Особое внимание было уделено алгоритмам поиска объектов по внешним источникам, нормализации объявлений, фильтрации объектов по географии и расчету степени соответствия объекта требованиям клиента.

В рамках программной реализации были созданы:

- frontend-модуль для работы риелтора в браузере;
- backend API-модуль для бизнес-логики и работы с PostgreSQL;
- Python-модуль поиска для обработки внешних сайтов недвижимости;
- схема базы данных и инфраструктура развертывания через Docker Compose.

Проведенное модульное и интеграционное тестирование подтвердило работоспособность основных сценариев системы и корректность наиболее значимых алгоритмов. Подготовленная эксплуатационная документация обеспечивает возможность запуска, использования и сопровождения программного продукта.

Практическая ценность выполненной работы заключается в том, что разработанная система позволяет сократить время ручного поиска недвижимости, централизовать хранение клиентских данных и обеспечить удобное формирование подборки объектов для клиента. Тем самым проект решает важную прикладную задачу автоматизации работы риелтора и может рассматриваться как основа для дальнейшего развития программного решения в области сопровождения сделок с недвижимостью.

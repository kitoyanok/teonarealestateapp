## 2.2 Проектирование модели данных

Тут тебе нужна одна нормальная схема БД и одна большая таблица с описанием сущностей и связей.

## Какие таблицы включать в схему

- `users`
- `clients`
- `client_search_profiles`
- `properties`
- `client_found_properties`
- `shortlist_items`
- `share_messages`
- `search_runs`

## Как они связываются

### `users -> clients`

- один ко многим;
- один риелтор ведет много клиентов.

### `clients -> client_search_profiles`

- один к одному;
- у каждого клиента один активный профиль поиска.

### `clients -> client_found_properties`

- один ко многим;
- у клиента много результатов поиска.

### `properties -> client_found_properties`

- один ко многим;
- один объект может участвовать в результатах нескольких клиентов.

### `clients -> shortlist_items`

- один ко многим;
- у клиента может быть много объектов в подборке.

### `properties -> shortlist_items`

- один ко многим;
- один объект может попасть в подборку разных клиентов.

### `clients -> share_messages`

- один ко многим;
- по клиенту могут быть сформированы разные версии сообщений.

### `clients -> search_runs`

- один ко многим;
- один клиент может запускать поиск много раз.

## Что писать текстом в разделе

- почему нужна реляционная модель;
- почему выбран PostgreSQL;
- почему данные разбиты на сущности, а не хранятся одной таблицей;
- как обеспечивается целостность через внешние ключи и ограничения.

## Что должно быть в таблице описания сущностей

По каждой сущности:

- название сущности;
- назначение;
- ключевое поле;
- основные атрибуты;
- связи с другими сущностями.

## 2.3 Разработка алгоритмов обработки данных

Здесь надо описывать только значимые алгоритмы, а не все подряд.

Для твоего проекта оптимально взять четыре алгоритма.

## Алгоритм 1. Поиск объектов по профилю клиента

### Что показывать на блок-схеме

- Начало.
- Риелтор создает клиента.
- API сохраняет клиента и профиль поиска.
- API отправляет запрос в search-service.
- Search-service выбирает источники по типу недвижимости.
- Получает HTML/JSON-LD.
- Извлекает карточки.
- Если карточка невалидна - отбросить.
- Если объект не из Краснодара - отбросить.
- Если объект не подходит по обязательным фильтрам - отбросить.
- Рассчитать процент совпадения.
- Вернуть результат в API.
- API сохраняет объекты и связи.
- Конец.

### Самые важные условия на схеме

- `Есть цена?`
- `Объект относится к Краснодару?`
- `Подходит по бюджету?`
- `Есть ли результаты поиска?`

## Алгоритм 2. Нормализация объявления

### Что показывать на блок-схеме

- Получить сырое объявление.
- Выделить текст, цену, площадь, комнатность, район.
- Проверить заголовок на служебный мусор.
- Если заголовок плохой - собрать короткое название из характеристик.
- Проверить описание.
- Если описание плохое - сформировать краткое служебное описание.
- Проверить изображения.
- Отбросить логотипы, иконки и пустые картинки.
- Сформировать нормализованный объект.

## Алгоритм 3. Фильтрация по городу Краснодар

### Что показывать на блок-схеме

- Получить текст и ссылку объявления.
- Проверить наличие маркеров `г. Краснодар`, `город Краснодар`, районов города.
- Проверить отсутствие других городов Краснодарского края.
- Если найден другой город - отклонить.
- Если найден Краснодар или район Краснодара - принять.

## Алгоритм 4. Расчет процента совпадения

### Для квартиры

- сравнить цену;
- сравнить комнатность;
- сравнить площадь;
- сравнить район;
- сравнить срок сдачи;
- сравнить отделку;
- суммировать баллы.

### Для дома

- сравнить цену;
- сравнить площадь дома;
- сравнить участок;
- сравнить локацию;
- сравнить коммуникации;
- сравнить этажность и спальни;
- суммировать баллы.

### Что еще писать в тексте

- какие критерии считаются обязательными;
- какие критерии повышают релевантность;
- почему результат выражается в процентах.

# EstateFlow — CRM для риелтора по подбору недвижимости

## 1. Краткое описание проекта

**EstateFlow** — веб-приложение для риелтора, где вся работа строится вокруг клиента.

Риелтор добавляет клиента, вводит его параметры поиска, система ищет подходящую недвижимость среди внешних публичных источников, показывает найденные объекты в карточке клиента, а риелтор вручную отбирает лучшие варианты в подборку и отправляет клиенту готовый текст со ссылками.

Приложение не является универсальным каталогом недвижимости. Главная сущность — **клиент и его запрос**.

---

## 2. Цель MVP

Сделать лаконичный рабочий кабинет риелтора:

1. Вход в аккаунт.
2. Главная страница с KPI и клиентами.
3. Добавление клиента с параметрами поиска.
4. Автоматический поиск объектов под клиента.
5. Просмотр найденных объектов в карточке клиента.
6. Детальный просмотр объекта: фото, цена, описание, характеристики.
7. Добавление объекта в подборку клиента.
8. Генерация текста подборки.
9. Подсветка контакта, куда отправить подборку.
10. Копирование текста для ручной отправки или переход к выбранному каналу.

## 3. Основной пользовательский сценарий

```text
Риелтор входит в приложение
↓
Попадает на Dashboard
↓

↓

↓
Формирует текст подборки
↓
Копирует текст или отправляет через выбранный канал
```

## 4. Что не делаем в MVP

Чтобы не перегружать приложение, в MVP не делаем:

- отдельную страницу «Подборки»;
- отдельную страницу «Источники»;
- отдельную страницу «Поиск недвижимости»;
- полноценную аналитику;
- роли пользователей;
- встроенный чат;
- email-рассылки;
- оплату;
- сложный таск-менеджер;
- карту города;
- автоматическую отправку в мессенджеры без подтверждения.

**Почему нет отдельной страницы поиска:** поиск всегда идет под конкретного клиента, поэтому он находится внутри карточки клиента.

**Почему нет отдельной страницы подборок:** подборка всегда принадлежит конкретному клиенту, поэтому она отображается внутри карточки клиента.

**Почему нет отдельной страницы источников:** источники заложены внутри сервиса и не редактируются риелтором. Статус поиска можно показывать локально в карточке клиента.

## 5. Стек проекта

### Frontend

- **JavaScript / TypeScript**
- **React**
- **Vite**
- **React Router**
- **TanStack Query** для запросов к API
- **React Hook Form + Zod** для форм и валидации
- **Recharts** для графика активности на Dashboard
- **CSS Modules / Tailwind CSS** — выбрать один подход

Рекомендация: использовать **React + TypeScript + Vite + Tailwind CSS**.

### Backend Web API

- **Node.js**
- **Express / Fastify**
- **Prisma ORM**
- **PostgreSQL**
- JWT / session auth для авторизации

Node.js отвечает за:

- авторизацию;
- клиентов;
- подборки;
- сохранение найденных объектов;
- работу с PostgreSQL;
- проксирование запросов к Python search service;
- генерацию текста подборки.

### Search Service

- **Python**
- **FastAPI**
- **httpx / aiohttp**
- **BeautifulSoup / selectolax**
- **Pydantic**
- **Playwright** только для источников, где данные грузятся через JavaScript

Python отвечает за:

- поиск объектов во внешних источниках;
- адаптеры под конкретные сайты;
- нормализацию данных;
- расчет совпадения объекта с запросом клиента;
- возврат объектов в едином формате.

### Database

- **PostgreSQL**

В README текущей версии упоминался SQLite, но в новой версии MVP нужно заменить хранение на PostgreSQL.

### DevOps

- Docker
- Docker Compose
- `.env`
- миграции Prisma
- единый запуск через `npm run dev`

## 6. Порты и запуск

### Рекомендуемые порты

```env
WEB_PORT=5002
NODE_API_PORT=5002
PYTHON_SEARCH_PORT=8002
POSTGRES_PORT=5432
```

Рекомендация: единая точка входа для пользователя — **http://localhost:5002**.

Node.js должен отдавать frontend и API на порту `5002`.

Python search service может работать отдельно на `8002`, но пользователю он напрямую не нужен.

### Команды запуска

```bash
npm install
npm run dev
```

или через Docker:

```bash
docker compose up --build
```

### Проверка занятости порта

Если `5002` занят, можно использовать `5003`.

```bash
lsof -i :5002
```

# 24. Источники для парсинга

## Важное правило

Перед реализацией каждого адаптера нужно проверить:

- доступность публичных страниц;
- наличие цен и карточек;
- robots.txt;
- условия использования сайта;
- можно ли получать данные без авторизации;
- не нарушается ли закон и правила сайта;
- не требуется ли официальное API или партнерское соглашение.

Сервис должен работать в режиме **live only**: если источник не отдал публичные карточки с ценой, показываем пустое состояние, а не демо-данные.

# 27. Адаптив

Обязательно продумать адаптив.

## Desktop

- Sidebar всегда виден.
- Dashboard: 4 KPI в ряд.
- Client page: найденные объекты слева, подборка справа.
- Property drawer справа.

## Tablet

- Sidebar схлопывается до иконок.
- KPI: 2 карточки в ряд.
- Client page: 1 колонка или 60/40.
- Drawer шириной 80%.

## Mobile

- Sidebar становится drawer или bottom navigation.
- KPI: 1 карточка в ряд.
- Таблицы заменяются карточками.
- Client page:
  1. карточка клиента;
  2. подборка;
  3. найденные объекты.
- Property drawer открывается на весь экран.
- Кнопка «Отправить подборку» закрепляется снизу, если есть объекты в подборке.

# 30. Definition of Done для MVP

MVP готов, если:

- приложение открывается на `http://localhost:5002`;
- риелтор может войти;
- риелтор может создать клиента;
- можно выбрать тип недвижимости: квартира или дом;
- параметры поиска сохраняются в PostgreSQL;
- после создания клиента запускается поиск;
- найденные объекты отображаются в карточке клиента;
- объект можно открыть в детальном drawer;
- объект можно добавить в подборку;
- подборка отображается внутри карточки клиента;
- можно сформировать текст подборки;
- контакт клиента для отправки подсвечен;
- текст можно скопировать;
- приложение нормально выглядит на desktop, tablet и mobile;
- пустые состояния и ошибки отображаются красиво и понятно.

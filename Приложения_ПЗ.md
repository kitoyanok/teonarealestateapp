# Приложения ПЗ по структуре Петруновского для проекта «Тэона»

Ниже приведена рабочая версия приложений к пояснительной записке. Структура приложений выстроена по логике Петруновского, но содержание полностью адаптировано под проект `Тэона`.

В файле сохранена следующая последовательность:

- Приложение А — техническое задание;
- Приложение Б — руководство по стилю;
- Приложение В — программный код модулей;
- Приложение Г — тест-кейсы;
- Приложение Д — руководство пользователя;
- Приложение Е — руководство программиста.

Основной принцип этого файла: не дробить систему на слишком большое число мелких частей, а выделить несколько крупных функциональных модулей, а внутри каждого модуля описать функции, задачи, входные и выходные данные.

# Приложение А

(обязательное)

**Техническое задание. Требования к программным модулям**

## 1 Назначение разработки

Разрабатываемая программная система `Тэона` предназначена для автоматизации работы риелтора при сопровождении клиентских заявок, связанных с подбором недвижимости в городе Краснодаре. Система должна обеспечивать ведение карточек клиентов, хранение параметров поиска, автоматический поиск объектов по внешним источникам, фильтрацию и ранжирование найденных объявлений, формирование подборки и подготовку текста сообщения для клиента.

## 2 Требования к программным модулям

В составе программной системы должны быть реализованы следующие крупные функциональные модули:

1. модуль авторизации и ведения клиентских заявок;
2. модуль поиска и обработки объявлений;
3. модуль формирования подборки и сообщения;
4. модуль аналитики и контроля состояния заявок.

Указанные модули должны работать совместно и обеспечивать единый пользовательский сценарий: вход в систему, создание карточки клиента, запуск поиска, просмотр найденных объектов, формирование подборки и фиксацию результата работы.

## 3 Требования к входным данным

К входным данным системы относятся:

- логин и пароль риелтора;
- ФИО клиента;
- телефон клиента;
- электронная почта клиента;
- предпочитаемый канал связи;
- тип недвижимости;
- бюджет покупки;
- диапазон площади;
- диапазон комнатности;
- район поиска;
- срок сдачи объекта;
- тип отделки;
- требования к дому или участку;
- дополнительные примечания риелтора;
- HTML-страницы, карточки объектов и текстовые данные, полученные с внешних сайтов недвижимости.

## 4 Требования к выходным данным

К выходным данным системы относятся:

- карточка клиента;
- сохраненный профиль поиска;
- список найденных объектов недвижимости;
- нормализованные характеристики объекта;
- процент совпадения объекта требованиям клиента;
- причины совпадения и несоответствия;
- подборка объектов;
- текст сообщения для клиента;
- история запусков поиска;
- сводные показатели по текущим заявкам.

### Таблица А.1 – Основные входные и выходные данные программных модулей

| Модуль | Входные данные | Выходные данные |
|---|---|---|
| Модуль авторизации и ведения клиентских заявок | Логин, пароль, данные клиента, параметры поиска | Сессия пользователя, карточка клиента, профиль поиска |
| Модуль поиска и обработки объявлений | Параметры поиска, данные внешних источников | Найденные объекты, нормализованные объявления, результаты ранжирования |
| Модуль формирования подборки и сообщения | Найденные объекты, выбранные варианты подборки | Подборка объектов, текст сообщения, отметка об отправке |
| Модуль аналитики и контроля состояния заявок | Данные карточек клиентов, история запусков поиска, статусы заявок | KPI-показатели, активность по заявкам, список клиентов, требующих внимания |

## 5 Требования к надежности

Система должна обеспечивать сохранность данных после перезапуска. Для этого база данных `PostgreSQL` должна использовать постоянное хранилище через `Docker volume`.

При ошибке поискового сервиса карточка клиента не должна теряться. Система должна сохранить данные клиента, зафиксировать запуск с ошибкой и предоставить возможность повторного запуска поиска.

При отсутствии результатов поиска система должна корректно завершать обработку, сохранять историю запуска и переводить заявку в состояние отсутствия результатов.

## 6 Требования к техническим средствам

Разработка должна выполняться с использованием языка `TypeScript`, платформы `Node.js`, библиотеки `React`, серверного фреймворка `Express`, языка `Python`, СУБД `PostgreSQL` и контейнерной платформы `Docker`.

Для запуска и сопровождения должны использоваться:

- `Visual Studio Code`;
- `Docker Desktop`;
- `TablePlus`;
- `PostgreSQL`;
- `npm`;
- встроенные средства терминала.

## 7 Требования к информационной совместимости

Информационная совместимость должна обеспечивать:

- поддержку русского языка при хранении и отображении текстовых данных;
- согласованность структуры данных между frontend, backend API, поисковым сервисом и базой данных;
- хранение сведений о клиентах, параметрах поиска, найденных объектах, подборках и истории сообщений;
- возможность повторного использования данных одной подсистемы в других модулях;
- единый формат представления характеристик объекта недвижимости после нормализации.

Это руководство по стилю определяет визуальную идентичность для программы `Тэона`. Данное руководство описывает использование цветов, шрифтов и основных элементов интерфейса.

# Приложение Д

(обязательное)

**Руководство пользователя. Выполнение программы и сообщения пользователю**

## 1 Выполнение программы

Для начала работы пользователь открывает web-приложение `Тэона`. После запуска открывается страница входа в систему, на которой риелтор вводит логин и пароль одной из преднастроенных учетных записей.

После успешной авторизации открывается главная страница программы. На главной странице отображаются сводные показатели по текущим заявкам: количество клиентов в работе, количество найденных объектов, количество объектов в подборках и число подборок, готовых к отправке.

Для просмотра существующих карточек используется раздел списка клиентов. Пользователь может выполнить поиск по карточкам, выбрать нужный статус и открыть карточку конкретного клиента.

Для создания новой карточки пользователь переходит в раздел добавления клиента. На форме указываются основные сведения о клиенте и параметры поиска недвижимости. После сохранения карточки автоматически запускается поиск объектов.

В карточке клиента отображаются найденные объекты недвижимости, основные характеристики, цена, район и процент совпадения. Пользователь может открывать подробную информацию по объекту и добавлять подходящие варианты в подборку.

После формирования подборки пользователь переходит к блоку сообщения. При наличии хотя бы одного объекта система формирует готовый текст для дальнейшей отправки клиенту. После фактической отправки риелтор отмечает подборку как отправленную.

После завершения работы пользователь закрывает программу стандартным способом средствами браузера и операционной системы.

## 2 Сообщения пользователю

При работе с программой пользователь может получать информационные сообщения и предупреждения. Основные сообщения пользователю следует представить в таблице Д.1.

# Приложение Е

(обязательное)

**Руководство программиста. Характеристика программы. Обращение к программе. Сообщения**

## 1 Характеристика программы

Программная система `Тэона` реализована как web-приложение с разделением на клиентскую часть, backend API, отдельный поисковый сервис и реляционную базу данных `PostgreSQL`.

В программе выделены четыре функциональных направления:

1. модуль авторизации и ведения клиентских заявок;
2. модуль поиска и обработки объявлений;
3. модуль формирования подборки и сообщения;
4. модуль аналитики и контроля состояния заявок.

### 1.1 Модуль авторизации и ведения клиентских заявок

Модуль отвечает за проверку учетных данных риелтора, создание карточки клиента, сохранение параметров поиска и повторный запуск поиска.

Функции модуля:

- формирование исходных учетных записей риелторов;
- обработка входа пользователя;
- подготовка профиля поиска;
- сохранение карточки клиента;
- повторный запуск поиска по существующей карточке.

Задачи функций модуля:

- предоставить риелтору возможность входа в систему;
- исключить доступ при неверных учетных данных;
- обеспечить сохранение клиентской карточки и параметров поиска;
- связать сохранение карточки с запуском поискового процесса.

### 1.2 Модуль поиска и обработки объявлений

Модуль отвечает за обращение к внешним источникам недвижимости, фильтрацию результатов по городу Краснодару, нормализацию характеристик объявлений и расчет процента совпадения.

Функции модуля:

- запуск поиска по карточке клиента;
- загрузка карточек объектов из внешних источников;
- фильтрация объектов по городу Краснодару;
- нормализация названия, района, площади, комнатности и цены;
- расчет процента совпадения объекта требованиям клиента;
- сохранение результатов поиска в базе данных.

Задачи функций модуля:

- исключить нерелевантные объявления;
- привести характеристики объекта к единому формату;
- рассчитать степень соответствия объекта заданным параметрам;
- передать результаты в основную систему для отображения пользователю.

### 1.3 Модуль формирования подборки и сообщения

Модуль отвечает за ручной отбор объектов, формирование подборки, подготовку итогового текста сообщения и фиксацию факта отправки клиенту.

Функции модуля:

- добавление объекта в подборку;
- удаление объекта из подборки;
- формирование текста сообщения;
- сохранение текста сообщения;
- изменение состояния карточки после отправки подборки.

Задачи функций модуля:

- дать риелтору возможность вручную отобрать подходящие объекты;
- обеспечить подготовку готового текста для клиента;
- зафиксировать завершение этапа подбора.

### 1.4 Модуль аналитики и контроля состояния заявок

Модуль отвечает за расчет сводных показателей и контроль состояния клиентских заявок.

Функции модуля:

- расчет числа клиентов в работе;
- расчет числа найденных объектов;
- расчет числа объектов в подборках;
- расчет числа заявок, готовых к отправке;
- формирование статистики активности по заявкам;
- вывод клиентов, требующих внимания.

Задачи функций модуля:

- предоставить риелтору оперативную сводную информацию;
- отразить текущее состояние клиентской базы;
- упростить контроль за заявками и повторными запусками поиска.

| Функциональное направление | Входные данные | Выходные данные |
|---|---|---|

## 2 Обращение к программе

Для обращения к программе необходимо открыть корневой каталог проекта `realestate` в среде `Visual Studio Code`. После открытия проекта следует проверить наличие файла `.env`, доступность `Docker Desktop` и возможность подключения к базе данных `PostgreSQL`.

Подготовка к запуску выполняется в следующем порядке:

1. открыть каталог проекта `realestate`;
2. при необходимости установить зависимости командой `npm install`;
3. запустить `Docker Desktop`;
4. выполнить команду `npm run prod:up`;
5. дождаться запуска контейнеров `postgres`, `search-service`, `api` и `web`;
6. открыть в браузере адрес `http://localhost:5002`.

Для просмотра базы данных рекомендуется использовать `TablePlus`. Для этого необходимо подключиться к локальному экземпляру `PostgreSQL` с параметрами, указанными в `docker-compose.yml`.

Для остановки программного продукта используется команда `npm run prod:down`. Для просмотра журналов контейнеров используется команда `npm run prod:logs`. Для полного сброса базы данных используется команда `npm run prod:reset-db`.

Модульные тесты запускаются командой `cd apps/search-service && .venv/bin/python -m unittest discover -s tests -v`.

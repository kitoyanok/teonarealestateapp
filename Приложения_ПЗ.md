## 1 Требования к функциональным характеристикам

В программную систему `Тэона` должны входить следующие модули:

- модуль авторизации и ведения клиентских заявок;
- модуль поиска и обработки объявлений;
- модуль формирования подборки и сообщения;
- модуль аналитики и контроля состояния заявок.

Каждый модуль должен быть реализован в соответствии с назначением системы и обеспечивать совместную обработку данных о клиентах, параметрах поиска, найденных объектах недвижимости и результатах работы риелтора.

### 1.1 Требования к составу выполняемых функций модуля авторизации и ведения клиентских заявок

Модуль авторизации и ведения клиентских заявок должен быть предназначен для формирования, хранения, изменения и представления данных о риелторах и клиентах.

Основные функции модуля авторизации и ведения клиентских заявок должны состоять из:

- функции формирования исходных учетных записей риелторов;
- функции авторизации пользователя;
- функции формирования карточки клиента;
- функции ведения данных клиента;
- функции управления состоянием клиентской заявки.

Функция формирования исходных учетных записей риелторов должна решать следующие задачи:

- формирование исходного набора учетных записей риелторов;
- хранение логинов, имен и контактных данных пользователей;
- подготовку учетных данных к использованию в системе;
- обеспечение возможности входа в программу без ручного добавления новых пользователей.

Функция авторизации пользователя должна решать следующие задачи:

- проверку логина и пароля риелтора;
- предоставление доступа только существующим пользователям;
- формирование пользовательской сессии;
- предотвращение некорректного входа в систему.

Функция формирования карточки клиента должна решать следующие задачи:

- создание новой карточки клиента;
- ввод и сохранение контактных данных клиента;
- ввод и сохранение параметров поиска недвижимости;
- связывание карточки клиента с конкретным риелтором;
- автоматический запуск поиска после сохранения карточки.

Функция ведения данных клиента должна решать следующие задачи:

- хранение и изменение сведений о клиенте;
- хранение параметров поиска;
- представление сведений о клиенте и найденных объектах;
- сохранение истории действий по карточке клиента.

Функция управления состоянием клиентской заявки должна решать следующие задачи:

- поддержку состояний `new`, `searching`, `found`, `no_results`, `shortlist_ready`, `sent`, `error`;
- обеспечение переходов между состояниями в соответствии с выполненными действиями;
- предотвращение некорректных переходов;
- актуализацию состояния заявки после запуска поиска, формирования подборки и отправки сообщения.

### 1.2 Требования к составу выполняемых функций модуля поиска и обработки объявлений

Модуль поиска и обработки объявлений должен быть предназначен для автоматизированного формирования списка объектов недвижимости на основе параметров клиентской заявки.

Основные функции модуля поиска и обработки объявлений должны состоять из:

- функции анализа параметров поиска клиента;
- функции загрузки и первичного отбора объявлений;
- функции фильтрации объектов по городу Краснодару;
- функции нормализации характеристик объектов;
- функции расчета процента совпадения и ранжирования результатов.

Функция анализа параметров поиска клиента должна решать следующие задачи:

- определение параметров, влияющих на поиск объектов;
- выявление ограничений по бюджету, площади, комнатности и району;
- подготовку входных данных для поискового модуля;
- формирование запроса к внешним источникам.

Функция загрузки и первичного отбора объявлений должна решать следующие задачи:

- получение данных с внешних сайтов недвижимости;
- извлечение карточек объектов;
- отбор объявлений, содержащих обязательные данные;
- исключение неполных и некорректных результатов.

Функция фильтрации объектов по городу Краснодару должна решать следующие задачи:

- определение принадлежности объекта к городу Краснодару;
- исключение объявлений из других городов Краснодарского края;
- учет района, текста объявления и URL-адреса источника;
- предотвращение попадания нерелевантных объектов в выдачу.

Функция нормализации характеристик объектов должна решать следующие задачи:

- приведение названия объекта к единому виду;
- выделение площади, комнатности, цены и района;
- формирование структурированного описания объекта;
- очистку текстового описания и изображений.

Функция расчета процента совпадения и ранжирования результатов должна решать следующие задачи:

- сопоставление характеристик объекта с требованиями клиента;
- начисление баллов по ключевым критериям;
- расчет итогового значения `matchScore`;
- формирование причин совпадения и несоответствия;
- ранжирование найденных результатов.

### 1.3 Требования к составу выполняемых функций модуля формирования подборки и сообщения

Модуль формирования подборки и сообщения должен быть предназначен для отбора подходящих объектов недвижимости и подготовки текста сообщения для клиента.

Основные функции модуля формирования подборки и сообщения должны состоять из:

- функции формирования подборки объектов;
- функции ведения данных подборки;
- функции формирования текста сообщения;
- функции фиксации факта отправки подборки.

Функция формирования подборки объектов должна решать следующие задачи:

- выбор подходящих объектов из списка найденных результатов;
- сохранение выбранных объектов в подборке клиента;
- предотвращение дублирования объектов в подборке;
- обеспечение возможности удаления объекта из подборки.

Функция ведения данных подборки должна решать следующие задачи:

- представление текущего состава подборки;
- хранение связи между клиентом и выбранными объектами;
- предоставление данных для формирования итогового сообщения;
- обеспечение повторного редактирования подборки.

Функция формирования текста сообщения должна решать следующие задачи:

- подготовку краткого описания каждого выбранного объекта;
- объединение данных по объектам в единый текст;
- указание цены, площади, района и ссылки на источник;
- сохранение готового сообщения для последующего использования.

Функция фиксации факта отправки подборки должна решать следующие задачи:

- сохранение признака отправки;
- изменение состояния карточки клиента после отправки;
- исключение неоднозначности в текущем состоянии работы по заявке.

### 1.4 Требования к составу выполняемых функций модуля аналитики и контроля состояния заявок

Модуль аналитики и контроля состояния заявок должен быть предназначен для представления сводной информации о текущей работе риелтора.

Основные функции модуля аналитики и контроля состояния заявок должны состоять из:

- функции расчета сводных показателей;
- функции представления активности по заявкам;
- функции контроля текущих состояний карточек клиентов;
- функции представления аналитической информации.

Функция расчета сводных показателей должна решать следующие задачи:

- определение числа клиентов в работе;
- определение числа найденных объектов;
- определение числа объектов в подборках;
- определение числа подборок, готовых к отправке.

Функция представления активности по заявкам должна решать следующие задачи:

- группировку истории запусков поиска по дням;
- построение сводной активности по текущим карточкам;
- представление данных для визуального графика на dashboard.

Функция контроля текущих состояний карточек клиентов должна решать следующие задачи:

- выявление карточек в состоянии `error`;
- выявление карточек в состоянии `no_results`;
- выявление карточек, требующих повторного действия риелтора;
- обеспечение быстрого перехода к проблемным карточкам.

Функция представления аналитической информации должна решать следующие задачи:

- отображение KPI-показателей на главной странице;
- отображение активности по заявкам;
- представление короткой управленческой информации по текущему состоянию системы.

## 2 Требования к надежности

В системе `Тэона` должны быть обеспечены два уровня надежности: уровень сохранности работоспособности и уровень сохранности информации.

Уровень сохранности работоспособности должен обеспечивать возможность продолжения работы программы после отказа одного из прикладных сервисов, временной недоступности внешнего источника объявлений или повторного запуска контейнеров программного комплекса. При отказе поискового сервиса данные карточки клиента не должны теряться, а состояние клиентской заявки должно переводиться в состояние ошибки с возможностью повторного запуска поиска.

Уровень сохранности информации должен обеспечивать сохранение сведений о риелторах, карточках клиентов, параметрах поиска, найденных объектах, подборках, сообщениях и истории запусков поиска. Хранение информации должно осуществляться в базе данных `PostgreSQL` с использованием постоянного тома хранения, подключенного в контейнерном окружении `Docker`.

Все входные данные должны проверяться на корректность, полноту и целостность с использованием механизмов валидации. Для обязательных полей должны быть обеспечены проверки на пустые значения, для числовых полей — проверки на допустимый диапазон значений, для статусов заявок — проверки на допустимость переходов между состояниями.

В выходных данных должны обеспечиваться точность, своевременность и полнота представления информации. Результаты поиска должны сохраняться только после прохождения фильтрации по городу Краснодару, а данные подборки и сформированного сообщения должны соответствовать текущему составу выбранных объектов.

Сохранность информации, являющейся объектом хранения системы, должна обеспечиваться при следующих аварийных ситуациях:

- сбой или перезапуск контейнеров `postgres`, `api`, `search-service` и `web`;
- временная недоступность поискового сервиса;
- ошибки при обработке внешних страниц объявлений;
- сбой прикладного программного обеспечения вследствие ошибок конфигурации.

При других видах аварийных ситуаций сохранность информации должна быть максимально обеспечена средствами используемой СУБД, системой контейнеризации и механизмами повторного выполнения операций.

## 3 Условия эксплуатации

Для обеспечения надежной работы системы и сохранности данных, а также выполнения заданных характеристик, оборудование и носители информации должны эксплуатироваться в следующих условиях:

- температура окружающего воздуха: от плюс 10 градусов по Цельсию до плюс 35 градусов по Цельсию;
- относительная влажность воздуха: от 30 процентов до 70 процентов при температуре плюс 20 градусов по Цельсию;
- атмосферное давление: в диапазоне от 86 килопаскаль до 106 килопаскаль;
- чистота воздуха: исключение пыли, агрессивных газов и паров, способных повредить оборудование или носители данных;
- защита от вибрации и механических воздействий: оборудование должно устанавливаться в условиях, исключающих вибрацию и удары, способные повредить накопители данных или ухудшить работу системы.

Соблюдение указанных условий обеспечивает надежную работу системы `Тэона` и сохранность базы данных, контейнерного окружения и программных файлов проекта.

## 4 Требования к составу и параметрам технических средств

Условия выполнения программы ограничены следующими параметрами технических средств:

- процессор: не ниже `Intel Core i3` или аналогичного;
- оперативная память: `8 GB` и более;
- свободное место на диске: не менее `2 GB` для программного комплекса и не менее `5 GB` для рабочей среды, контейнеров и данных;
- операционная система: `Windows 10` и выше, `macOS` или современный дистрибутив `Linux`;
- установленное контейнерное окружение `Docker Desktop`;
- установленная среда разработки `Visual Studio Code`;
- установленный `Node.js` для работы с frontend и backend-частью;
- установленный современный web-браузер;
- средство просмотра базы данных `TablePlus` или аналогичное;
- доступ к локальным адресам `localhost:5002`, `localhost:5003` и `localhost:8002`.

Выполнение данных условий обеспечивает корректный запуск пользовательского интерфейса, backend API, поискового сервиса и базы данных `PostgreSQL`.

## 5 Требования к информационной и программной совместимости

Программные модули должны быть совместимы с современными операционными системами пользовательского рабочего места и поддерживать запуск в контейнерной среде `Docker`.

Разработка должна выполняться с использованием языка `TypeScript` для frontend и backend API, языка `Python` для поискового сервиса, библиотеки `React`, серверной платформы `Node.js`, базы данных `PostgreSQL`, контейнеризации `Docker Compose` и сопутствующих средств разработки `Visual Studio Code` и `TablePlus`.

Информационная совместимость должна обеспечивать:

- поддержку русского языка при хранении и представлении текстовых данных;

## 1 Выполнение программы

Для начала работы пользователь открывает приложение `Тэона`. После запуска открывается страница входа в систему. Для входа в программу риелтор вводит логин и пароль одной из преднастроенных учетных записей.

После успешной авторизации открывается главная страница. На главной странице отображаются сводные показатели по текущим клиентским заявкам: количество клиентов в работе, число найденных объектов, число объектов в подборках и число подборок, готовых к отправке. С главной страницы пользователь может перейти к списку клиентов.

Для просмотра текущих заявок пользователь открывает раздел клиентов, где отображаются карточки с основными сведениями: именем клиента, бюджетом, типом недвижимости, выбранными районами, текущим статусом заявки и числом найденных объектов.

Для создания новой карточки пользователь нажимает кнопку добавления клиента. На форме указываются ФИО клиента, номер телефона, при необходимости адрес электронной почты, тип недвижимости, бюджет, диапазон площади, количество комнат, районы поиска и дополнительные параметры. После сохранения карточки система автоматически запускает поиск объектов.

В карточке клиента отображаются основные сведения о клиенте, параметры поиска, текущий статус заявки и найденные объекты недвижимости. Для каждого объекта указываются название, цена, площадь, комнатность, район, процент совпадения и ссылка на источник. Пользователь может открыть подробную информацию об объекте и добавить выбранный объект в подборку.

После формирования подборки пользователь переходит к блоку сообщения. При наличии выбранных объектов система формирует текст сообщения для клиента, включающий сведения о подобранных вариантах. После фактической отправки сообщения риелтор может отметить подборку как отправленную, в результате чего состояние карточки клиента обновляется.

При необходимости пользователь может выполнить повторный запуск поиска по текущим параметрам клиента. В этом случае система повторно обращается к поисковому сервису, обновляет найденные результаты и изменяет состояние заявки в соответствии с итогом поиска.

После завершения работы пользователь закрывает программу стандартным способом средствами web-браузера и операционной системы.

## 1 Характеристика программы

Программная система `Тэона` реализована как web-приложение с разделением на клиентскую часть, backend API, отдельный поисковый сервис и реляционную базу данных `PostgreSQL`. Основной проект построен по модульному принципу и включает каталоги `apps/web`, `apps/api` и `apps/search-service`.

В программе выделены четыре функциональных направления: модуль авторизации и ведения клиентских заявок, модуль поиска и обработки объявлений, модуль формирования подборки и сообщения, модуль аналитики и контроля состояния заявок.

### 1.1 Модуль авторизации и ведения клиентских заявок

Модуль отвечает за проверку учетных данных риелтора, создание карточки клиента, сохранение параметров поиска и повторный запуск поиска. Основными функциями модуля являются формирование исходных учетных записей, обработка входа пользователя, подготовка профиля поиска, сохранение карточки клиента и изменение состояния клиентской заявки.

### 1.2 Модуль поиска и обработки объявлений

Модуль отвечает за обращение к внешним источникам недвижимости, фильтрацию результатов по городу Краснодару, нормализацию характеристик объявлений и расчет процента совпадения. Основными функциями модуля являются запуск поиска, сохранение истории запусков, нормализация названий объектов, определение района, расчет степени совпадения и ранжирование результатов.

### 1.3 Модуль формирования подборки и сообщения

Модуль отвечает за ручной отбор объектов, формирование подборки, генерацию итогового текста сообщения и фиксацию факта отправки. Основными функциями модуля являются добавление и удаление объектов из подборки, формирование текста сообщения и изменение состояния клиентской заявки.

### 1.4 Модуль аналитики и контроля состояния заявок

Модуль отвечает за расчет сводных показателей по текущей работе риелтора. Основными функциями модуля являются расчет числа клиентов в работе, найденных объектов, объектов в подборках, готовых к отправке подборок и формирование активности по заявкам.

## 2 Обращение к программе

Для обращения к программе необходимо открыть корень проекта `realestate` в среде `Visual Studio Code`. После открытия проекта проверяются наличие файла `.env`, доступность `Docker Desktop` и возможность подключения к базе данных `PostgreSQL`.

Подготовка к запуску выполняется в следующем порядке:

- открыть корневой каталог проекта в `Visual Studio Code`;
- при необходимости установить `Node.js`-зависимости командой `npm install`;
- запустить `Docker Desktop`;
- выполнить команду `npm run prod:up`;
- дождаться запуска контейнеров `postgres`, `search-service`, `api` и `web`;
- открыть в браузере адрес `http://localhost:5002`.

Для просмотра содержимого базы данных используется `TablePlus`. Для этого необходимо подключиться к локальному экземпляру `PostgreSQL`, используя параметры из файла `docker-compose.yml`.

Для остановки программного продукта используется команда `npm run prod:down`. Для просмотра журналов контейнеров используется команда `npm run prod:logs`. Для полного сброса базы данных используется команда `npm run prod:reset-db`.

Модульные тесты запускаются командой `cd apps/search-service && .venv/bin/python -m unittest discover -s tests -v`. Указанные тесты проверяют корректность нормализации объявлений, определения принадлежности объекта к городу Краснодару и расчета процента совпадения без необходимости ручного открытия интерфейса.

## 3 Сообщения

В ходе сборки, запуска и сопровождения программы могут возникать сообщения, требующие действий программиста или специалиста сопровождения. Данные сообщения приведены в таблице Е.1.

**Таблица Е.1 – Основные сообщения**

| Сообщение или ситуация | Возможная причина | Рекомендуемое действие |
|---|---|---|
| Контейнер `postgres` не запускается | Порт занят или поврежден том хранения | Проверить занятость порта `5432`, при необходимости выполнить `npm run prod:reset-db` |
| Контейнер `api` не проходит health-check | База данных или search-service недоступны | Проверить логи контейнеров и доступность зависимых сервисов |
| Контейнер `search-service` не проходит health-check | Ошибка Python-зависимостей или проблемы с запуском сервиса | Проверить логи контейнера `search-service` |
| Модульные тесты завершаются ошибкой | Нарушена логика нормализации или сопоставления | Проверить файлы `test_normalization.py` и `test_matcher.py` |
| В приложении не отображаются данные клиентов | Ошибка подключения frontend к API или проблема с авторизацией | Проверить доступность `http://localhost:5003/api/health` и корректность cookie-сессии |
| Повторный запуск контейнеров не сохраняет данные | Ошибка в работе постоянного тома PostgreSQL | Проверить подключение тома `estateflow-postgres` в `docker-compose.yml` |

# Перечень схем, диаграмм, таблиц и скриншотов для ПЗ

Данный файл является подробной рабочей инструкцией по сборке графической и табличной части пояснительной записки по проекту `Тэона`. Он нужен для того, чтобы без догадок понять:

- какие диаграммы нужно рисовать;
- какие актеры, сущности, связи и поля в них включать;
- какие таблицы должны быть в ПЗ;
- как должны выглядеть таблицы по базе данных;
- какие скриншоты нужно снять и что на них обязательно показать.

Файл синхронизирован с текущей структурой `Диплом_структура_ПЗ.md`, с отдельным файлом приложений `Приложения_ПЗ.md` и с реальным кодом проекта.

# 0 Общие правила оформления по образцу Петруновского

## 0.1 Как оформлять рисунки

Для каждого рисунка в основном тексте ПЗ используется одна и та же схема подачи:

1. короткий вводный абзац перед рисунком;
2. подпись рисунка под изображением;
3. поясняющий абзац после рисунка со словами `В соответствии с рисунком ...`.

### Шаблон текста перед рисунком

`На рисунке N представлена ...`

### Шаблон подписи рисунка

`Рисунок N – ...`

### Шаблон текста после рисунка

`В соответствии с рисунком N, ...`

## 0.2 Как оформлять таблицы

Подпись таблицы ставится над таблицей. Формат подписи:

`Таблица N – ...`

Если таблица находится в приложении, используется буквенная нумерация:

- `Таблица А.1`;
- `Таблица В.1`;
- `Таблица Г.1`;
- `Таблица Д.1`;
- `Таблица Е.1`.

## 0.3 Что важно не выдумывать

Диаграммы и схемы должны отражать реальную текущую систему, а не предполагаемую. Для проекта `Тэона` это означает следующее:

- отдельного экрана просмотра истории поисковых запусков сейчас нет;
- история запусков **хранится в БД** и используется в аналитике, но не выделена в отдельный пользовательский экран;
- поэтому на use case диаграммах не нужно рисовать отдельный прецедент `Просмотреть историю запусков поиска` как самостоятельную экранную функцию;
- `search_runs` нужно показывать в ER-диаграмме и в таблицах БД, потому что таблица реально существует и используется backend-логикой.

## 0.4 Где размещать формулы

Формулы уместнее всего размещать в разделе алгоритмов:

- `2.6.2 Разработка алгоритма нормализации объявления`;
- `2.6.4 Разработка алгоритма расчета процента совпадения`.

Для проекта `Тэона` следует использовать:

- формулу расчета стоимости одного квадратного метра;
- формулу итогового процента совпадения;
- формулу коэффициента совпадения по коммуникациям.

## 1.1 Рисунок 1 – Общая диаграмма вариантов использования программных модулей системы «Тэона»

### Что это за диаграмма

Это верхнеуровневая `use case` диаграмма. Она показывает, какие основные действия выполняет риелтор в системе.

### Какого актера рисовать

Основной актер один:

- `Риелтор`.

Внешние сайты недвижимости здесь не нужно делать отдельным актером. Их правильнее показать на диаграмме компонентов.

### Какие прецеденты рисовать

- `Авторизоваться в системе`;
- `Просмотреть dashboard`;
- `Создать карточку клиента`;
- `Изменить карточку клиента`;
- `Запустить поиск объектов`;
- `Просмотреть найденные объекты`;
- `Открыть подробности объекта`;
- `Добавить объект в подборку`;
- `Удалить объект из подборки`;
- `Сформировать текст сообщения`;
- `Отметить подборку как отправленную`.

### Какие связи рисовать

- `Создать карточку клиента` <<include>> `Запустить поиск объектов`;
- `Просмотреть найденные объекты` <<extend>> `Открыть подробности объекта`;
- `Добавить объект в подборку` <<include>> `Сформировать текст сообщения`;
- `Сформировать текст сообщения` <<extend>> `Отметить подборку как отправленную`.

### Как расположить

- риелтор слева;
- внутри границы системы все use case;
- сверху разместить авторизацию и dashboard;
- в центре карточку клиента и запуск поиска;
- справа работу с найденными объектами;
- снизу подборку и отправку.

### Что не надо рисовать

- отдельный прецедент `Просмотреть историю запусков поиска`;
- технические прецеденты уровня `Сохранить в PostgreSQL`, `Обратиться к API`, `Обработать HTML`.

## 1.2 Рисунок 2 – Диаграмма вариантов использования модуля создания и ведения клиентских заявок

### Что должно быть на диаграмме

- `Создать карточку клиента`;
- `Изменить карточку клиента`;
- `Сохранить профиль поиска`;
- `Запустить поиск объектов`;
- `Открыть карточку клиента`;
- `Повторно запустить поиск`.

### Связи

- `Создать карточку клиента` <<include>> `Сохранить профиль поиска`;
- `Создать карточку клиента` <<include>> `Запустить поиск объектов`;
- `Открыть карточку клиента` <<extend>> `Повторно запустить поиск`.

### Важная оговорка

История запусков поиска в текущем интерфейсе не выделена отдельным самостоятельным экраном. Поэтому ее не нужно рисовать как отдельный use case. Если захочешь показать ее косвенно, это можно сделать только как внутренний результат работы карточки клиента, а не как отдельное пользовательское действие.

## 1.3 Рисунок 3 – Диаграмма вариантов использования модуля поиска объектов недвижимости

### Что должно быть на диаграмме

- `Запустить поиск объектов`;
- `Отфильтровать объекты по городу Краснодару`;
- `Нормализовать найденные объявления`;
- `Рассчитать процент совпадения`;
- `Сохранить результаты поиска`;
- `Просмотреть найденные объекты`.

### Как строить

- риелтор связывается только с `Запустить поиск объектов` и `Просмотреть найденные объекты`;
- остальные действия включаются в `Запустить поиск объектов` через <<include>>.

### Что важно подчеркнуть

Это модуль автоматической обработки. Пользователь не управляет вручную фильтрацией по Краснодару и расчетом `matchScore`, он только запускает поиск и получает результат.

## 1.7 Рисунок 7 – Диаграмма состояний клиентской заявки

### Состояния

- `new`;
- `searching`;
- `found`;
- `no_results`;
- `shortlist_ready`;
- `sent`;
- `error`.

### Переходы

- `new -> searching` после сохранения карточки и старта поиска;
- `searching -> found` если объекты найдены;
- `searching -> no_results` если поиск завершился без результатов;
- `searching -> error` если поиск завершился ошибкой;
- `found -> shortlist_ready` после добавления хотя бы одного объекта в подборку;
- `shortlist_ready -> found` если подборка снова стала пустой, но найденные объекты остались;
- `shortlist_ready -> sent` после отметки отправки подборки.

### Что не надо дорисовывать

- `closed`, если такого статуса нет в текущем UI и API;
- произвольные переходы `error -> sent`, `no_results -> sent`.

# Перечень схем, диаграмм, таблиц и скриншотов для ПЗ

Данный файл является рабочей картой по всем графическим и табличным материалам пояснительной записки. Его задача — снять неопределенность при сборке ПЗ: что именно рисовать, как подписывать рисунки и таблицы, какие сущности и связи показывать, из каких файлов брать кодовые фрагменты и какие скриншоты необходимо подготовить.

Файл синхронизирован с текущей структурой основного документа `Диплом_структура_ПЗ.md`.

---

# 0. Общие правила оформления по образцу Петруновского

## 0.1 Правило для рисунков

Для каждого рисунка в основном тексте ПЗ должна использоваться одна и та же схема подачи:

1. короткий вводный абзац перед рисунком;
2. подпись рисунка под изображением;
3. поясняющий абзац после рисунка со словами `В соответствии с рисунком ...`.

### Шаблон текста перед рисунком

`На рисунке N представлена ...`  
или  
`На рисунке N следует представить ...`.

### Шаблон подписи рисунка

`Рисунок N – ...`

### Шаблон текста после рисунка

`В соответствии с рисунком N, ...`

## 0.2 Правило для таблиц

Подпись таблицы ставится над таблицей. Формат подписи:

`Таблица N – ...`

Если таблица находится в приложении, то используется буквенная нумерация:

- `Таблица Г.1`;
- `Таблица Д.1`;
- `Таблица Е.1`.

## 0.3 Правило для приложений

Приложения должны идти в следующем порядке:

- `Приложение А` — техническое задание;
- `Приложение Б` — руководство по стилю;
- `Приложение В` — программный код модулей;
- `Приложение Г` — тест-кейсы;
- `Приложение Д` — руководство пользователя;
- `Приложение Е` — руководство программиста.

## 0.4 Общий принцип отбора материалов

В ПЗ включаются только те материалы, которые подтверждают:

- пользовательские сценарии работы системы;
- архитектуру и модель данных;
- алгоритмы обработки объявлений;
- факт программной реализации;
- факт тестирования;
- возможность запуска и сопровождения системы.

## 0.5 Правило для формул

Формулы уместнее всего размещать в разделе алгоритмов, а не в приложениях. Для текущей ПЗ рекомендуется использовать формулы в подразделах:

- `2.6.2 Разработка алгоритма нормализации объявления`;
- `2.6.4 Разработка алгоритма расчета процента совпадения`.

Формула должна:

- быть расположена отдельной строкой;
- иметь номер в круглых скобках справа;
- после формулы должен идти абзац `где ...`, раскрывающий обозначения.

Для проекта `Тэона` в основной текст уже логично заложены следующие формулы:

- формула расчета стоимости квадратного метра;
- формула итогового процента совпадения;
- формула коэффициента совпадения по коммуникациям.

## 1.4.1 Формулы, которые должны сопровождать алгоритмы

### Формула (1) – Расчет стоимости одного квадратного метра

Формулу следует размещать в подразделе `2.6.2` после описания извлечения цены и площади:

`C_{м²} = C / S`

Под формулой должно быть пояснение:

- `C_{м²}` — стоимость одного квадратного метра;
- `C` — полная стоимость объекта;
- `S` — общая площадь объекта.

### Формула (2) – Итоговый процент совпадения объекта требованиям клиента

Формулу следует размещать в подразделе `2.6.4` после текста о весах критериев:

`M = (Σ(w_i * k_i) / Σw_i) * 100%`

Под формулой должно быть пояснение:

- `M` — итоговый процент совпадения;
- `w_i` — вес `i`-го критерия;
- `k_i` — коэффициент выполнения `i`-го критерия в диапазоне от `0` до `1`.

### Формула (3) – Коэффициент совпадения по коммуникациям

Формулу следует размещать в подразделе `2.6.4` после формулы (2):

`k_{комм} = n_{совп} / n_{треб}`

Под формулой должно быть пояснение:

- `k_{комм}` — коэффициент совпадения по коммуникациям;
- `n_{совп}` — число совпавших коммуникаций;
- `n_{треб}` — число требуемых коммуникаций.

# 4. Что и как снимать для скриншотов

## 4.1 Как запустить систему для интерфейсных скринов

```bash
npm run prod:up
```

После запуска открыть:

- web: `http://localhost:5002`;
- API health: `http://localhost:5003/api/health`;
- search-service health: `http://localhost:8002/health`.

## 4.2 Как снимать скрины интерфейса

Снимать нужно уже на чистом состоянии интерфейса:

- без открытых devtools;
- без лишних всплывающих окон;
- по возможности без панели браузерных вкладок;
- с читаемыми названиями кнопок и карточек.

## 4.3 Как снимать скрины базы данных

1. Запустить систему через Docker.
2. Открыть `TablePlus`.
3. Создать подключение к `PostgreSQL` по параметрам из раздела `1.6`.
4. Сначала снять список таблиц.
5. Затем открыть по очереди `users`, `clients`, `properties`, `client_found_properties`.

## 4.4 Как снимать скрин модульных тестов

```bash
cd apps/search-service && .venv/bin/python -m unittest discover -s tests -v
```

На скрине должно быть видно имена тестовых методов:

- из `test_normalization.py`;
- из `test_matcher.py`;
- финальный статус `OK`.

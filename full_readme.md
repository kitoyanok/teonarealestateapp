# EstateFlow / Teona Real Estate App: Full Technical README

## 1. Что это за проект

Этот проект - CRM и инструмент подбора недвижимости для риелтора. Система позволяет:

- авторизоваться под учетной записью риелтора;
- завести клиента и сохранить его запрос;
- автоматически запустить поиск подходящих объектов;
- получить список найденных квартир или домов;
- сохранить подходящие позиции в подборку;
- сформировать текст для отправки клиенту;
- хранить все данные в PostgreSQL;
- после перезапуска контейнеров не терять данные;
- показывать сводную аналитику по клиентам, поискам и подборкам.

Если описывать систему одной фразой, то это связка из трех частей:

1. `web` - интерфейс для риелтора.
2. `api` - сервер, который хранит клиентов, управляет логикой и пишет в БД.
3. `search-service` - отдельный Python-сервис, который парсит сайты и возвращает нормализованные объекты недвижимости.

---

## 2. Главная бизнес-идея

Пользователь системы - риелтор.

Основной сценарий такой:

1. Риелтор заходит в систему.
2. Создает карточку клиента.
3. Указывает параметры: бюджет, район, площадь, комнатность, тип недвижимости.
4. Система отправляет запрос в сервис поиска.
5. Сервис поиска проходит по списку сайтов-источников, вытаскивает объявления, очищает мусор, фильтрует по Краснодару, нормализует характеристики и считает процент совпадения.
6. Backend сохраняет результаты поиска в PostgreSQL.
7. На странице клиента риелтор видит найденные объекты.
8. Риелтор вручную добавляет часть объектов в подборку.
9. Система формирует готовое сообщение для клиента.
10. После отправки клиент получает подборку, а система фиксирует статус работы.

## 5. Как система работает целиком

## 5.1 Поток от браузера до базы

Пользователь открывает `web`, нажимает кнопки и заполняет формы.  
Frontend делает HTTP-запросы в `api`.  
`api`:

- проверяет авторизацию;
- валидирует входные данные;
- записывает данные в PostgreSQL;
- при необходимости обращается в `search-service`.

`search-service`:

- получает JSON с параметрами поиска;
- обходит сайты-источники;
- вытаскивает объявления;
- отбрасывает мусор и нерелевантные данные;
- считает score;
- возвращает массив нормализованных объектов в `api`.

`api` затем:

- создает запись о запуске поиска;
- upsert-ит объекты в таблицу `properties`;
- связывает их с конкретным клиентом в `client_found_properties`;
- обновляет статус клиента;
- отдает frontend итоговый ответ.

## 5.2 Поток с подборкой

Когда риелтор нажимает `В подборку`:

1. Frontend вызывает endpoint API.
2. API проверяет, что объект действительно был найден для этого клиента.
3. Если все нормально, запись добавляется в `shortlist_items`.
4. Статус клиента меняется на `shortlist_ready`.
5. Dashboard и страница клиента перечитывают свежие данные.

## 5.3 Поток с отправкой клиенту

1. Риелтор нажимает кнопку формирования сообщения.
2. API берет shortlist клиента.
3. Сервис `messageService.ts` собирает текст.
4. Текст сохраняется в `share_messages`.
5. После фактической отправки риелтор может отметить отправку.
6. Статус клиента меняется на `sent`.

## 6. User flow для диплома

Ниже сценарий, который можно почти дословно переносить в ПЗ.

### Сценарий 1. Вход в систему

- Пользователь открывает страницу входа.
- Вводит логин и пароль.
- Система отправляет запрос на `/api/auth/login`.
- Backend проверяет пользователя.
- При успехе backend возвращает cookie с JWT.
- Пользователь попадает в защищенную часть приложения.

### Сценарий 2. Создание клиента

- Пользователь переходит на страницу добавления клиента.
- Заполняет ФИО, телефон, канал связи и параметры поиска.
- Нажимает `Сохранить и найти`.
- Frontend формирует payload.
- API валидирует данные.
- API создает клиента и профиль поиска.
- API запускает поиск через Python-сервис.
- Результаты сохраняются в PostgreSQL.
- Пользователь попадает в карточку клиента.

### Сценарий 3. Просмотр найденных объектов

- Пользователь открывает карточку клиента.
- Frontend запрашивает детали клиента.
- Backend возвращает:
  - данные клиента;
  - профиль поиска;
  - найденные объекты;
  - shortlist;
  - историю запусков поиска.
- Пользователь видит список карточек объектов.

### Сценарий 4. Формирование подборки

- Пользователь просматривает найденные объекты.
- У подходящих объектов нажимает `В подборку`.
- Система сохраняет связь клиент-объект в shortlist.
- Счетчик подборки обновляется.

### Сценарий 5. Отправка подборки клиенту

- Пользователь формирует сообщение.
- Система собирает человекочитаемый текст.
- Пользователь копирует и отправляет сообщение клиенту.
- После отправки отмечает действие в системе.
- Система сохраняет факт отправки.

### Сценарий 6. Повторный поиск

- Если параметры клиента изменились или результаты устарели, риелтор запускает поиск заново.
- Система создает новую запись `search_run`.
- Парсер снова проходит источники.
- Backend обновляет найденные объекты и историю поиска.

## 9. Как работает поиск и парсинг

## 9.1 Общая схема

1. Frontend создает клиента.
2. API вызывает `search-service`.
3. Search-service получает `SearchRequest`.
4. Строит список адаптеров по типу недвижимости.
5. Каждый адаптер обходит свои `start_urls`.
6. Для каждой страницы:
   - проверяется `robots.txt`;
   - скачивается HTML;
   - сначала ищется `JSON-LD`;
   - затем ищутся визуальные карточки из HTML.
7. Из текста извлекаются:
   - цена;
   - площадь;
   - комнатность;
   - район;
   - этаж;
   - отделка;
   - материал дома;
   - коммуникации и так далее.
8. Все записи проходят географическую фильтрацию.
9. Все записи нормализуются.
10. Дубликаты склеиваются.
11. Для каждого объекта считается `matchScore`.
12. Результаты сортируются и возвращаются в API.

## 9.2 Ограничение только по Краснодару

Сейчас логика специально ужесточена:

- если в тексте видны другие города Краснодарского края, объект отбрасывается;
- если в тексте есть район Краснодара, объект считается краснодарским;
- если в URL есть маркеры `krasnodar` или `krd`, объект тоже может пройти;
- города вроде Сочи, Анапы, Новороссийска, Геленджика не должны попадать.

## 9.3 Как считается совпадение

Для квартир учитываются:

- бюджет;
- комнатность;
- площадь;
- район;
- год сдачи;
- отделка.

Для домов учитываются:

- бюджет;
- площадь дома;
- площадь участка;
- локация;
- коммуникации;
- этажность;
- количество спален.

Каждый критерий дает часть баллов. В конце формируется:

- `matchScore`
- `matchReasons`
- `mismatchReasons`

## 10. Как запускать проект

## 10.1 Production-подъем через Docker Compose

Это основной и самый правильный режим для диплома.

### Первый запуск

1. Создать `.env` на основе `.env.example`.
2. Выполнить:

```bash
npm run prod:up
```

### Что произойдет

- поднимется PostgreSQL;
- поднимется Python search-service;
- поднимется Node API;
- соберется и поднимется web;
- frontend будет доступен на `http://localhost:5002`;
- API будет работать на `http://localhost:5003`;
- search-service будет работать на `http://localhost:8002`.

### Остановить

```bash
npm run prod:down
```

### Посмотреть логи

```bash
npm run prod:logs
```

### Полный сброс базы

```bash
npm run prod:reset-db
```

Эта команда удаляет volume PostgreSQL, поэтому применять ее нужно только когда реально нужен полный reset.

## 10.2 Dev-режим

```bash
npm run dev
```

Что делает команда:

- запускает скрипт `scripts/dev.mjs`;
- тот проверяет, свободны ли порты `5002`, `5003`, `8002`;
- если порты заняты, скрипт завершится с понятным сообщением;
- если все нормально, дальше запускаются три процесса:
  - `web`
  - `api`
  - `search-service`

## 10.3 Раздельный запуск

```bash
npm run dev:split
```

или отдельные workspace-команды.

- `start:prod` - поднимает production compose.

- `postgres` использует volume `estateflow-postgres`;
- таблицы инициализируются SQL-файлом;
- `search-service` получает свой порт и healthcheck;
- `api` зависит от `postgres` и `search-service`;
- `web` зависит от `api`;
- между контейнерами используются внутренние имена хостов;
- `DATABASE_URL` внутри контейнеров указывает на `postgres`, а не на `localhost`.

## 13.3 `database/init.sql`

Это схема БД. Файл:

- создает расширение `pgcrypto`;
- создает все таблицы;
- задает `uuid` по умолчанию;
- задает внешние ключи;
- задает индексы.

Практически каждая группа SQL здесь нужна для конкретной сущности:

- блок `users` - учетные записи;
- блок `clients` - карточки клиентов;
- блок `client_search_profiles` - критерии поиска;
- блок `properties` - единый справочник объектов;
- блок `client_found_properties` - найденные объекты для клиента;
- блок `shortlist_items` - подборка;
- блок `share_messages` - готовые сообщения;
- блок `search_runs` - история поисков;
- блок индексов - ускорение выборок по `user_id`, `client_id`, `status`, `source_url` и другим полям.

## 14. Backend API: разбор по файлам

Ниже идет максимально подробный разбор backend-кода сверху вниз.

## 14.1 `apps/api/src/config.ts`

Роль файла: централизованно загрузить переменные окружения и собрать конфиг.

Логика по порядку:

- импортируется `dotenv`, `path` и `fileURLToPath`;
- вычисляется путь к текущей директории ESM-модуля;
- вызывается `dotenv.config()` сначала для локального `.env`, потом для корневого `.env`;
- это сделано, чтобы код работал и из workspace, и из корня монорепы;
- дальше экспортируется объект `config`.

Что в нем:

- `port` - порт API;
- `jwtSecret` - ключ подписи JWT;
- `searchServiceUrl` - базовый адрес Python-сервиса;
- `isProduction` - флаг production;
- `databaseUrl` - строка подключения PostgreSQL;
- `demoLogin`, `demoPassword` - учетные данные seed-пользователя.

Почему файл важен:

- он снимает размазанность настроек по проекту;
- все остальные части backend читают параметры отсюда.

## 14.2 `apps/api/src/db/pool.ts`

Роль файла: создать единое подключение к PostgreSQL.

Что происходит:

- импортируется `Pool` из `pg`;
- импортируется `config`;
- создается `pool` на основе `config.databaseUrl`;
- экспортируется helper `query<T>()`.

Что делает `query<T>()`:

- принимает SQL-строку и массив параметров;
- вызывает `pool.query`;
- возвращает типизированный результат.

Почему это полезно:

- в коде не нужно в каждом файле руками создавать `Pool`;
- все SQL-запросы используют одно место подключения.

## 14.3 `apps/api/src/middleware/auth.ts`

Роль файла: сессия и защита API.

### `signSession(userId)`

- получает `userId`;
- создает JWT с payload `{ userId }`;
- подписывает его `config.jwtSecret`;
- задает срок жизни `7d`.

### `requireAuth`

Это express middleware.

Логика:

- пытается найти токен в cookie `estateflow_token`;
- если cookie нет, смотрит заголовок `Authorization`;
- если токен не найден, сразу отвечает `401`;
- если токен найден, делает `jwt.verify`;
- из payload достает `userId`;
- проверяет, что такой пользователь реально существует через `findUserById`;
- если пользователь найден, кладет `userId` в `req.userId`;
- вызывает `next()`.

Если что-то пошло не так:

- ответ `401`;
- сообщение о неавторизованном доступе.

Смысл файла:

- защищенные маршруты не работают без валидной сессии.

## 14.4 `apps/api/src/repositories/db.ts`

Очень маленький файл.

Он просто переэкспортирует бизнес-функцию `ensureSeedUser`.

Зачем вообще отдельный файл:

- чтобы `server.ts` не знал детали SQL-реализации.

## 14.5 `apps/api/src/services/validation.ts`

Роль файла: валидация и нормализация входных данных.

### Телефон

- `normalizePhone` убирает лишние символы;
- переводит номер в единый вид;
- используется при сохранении клиента.

- `phoneError` содержит текст ошибки.

### Вспомогательные Zod-преобразователи

- `nullableNumber` превращает пустые строки в `null` или число;
- `nullableInt` делает то же самое для целых;
- `list` приводит строковые списки к массивам.

### `loginSchema`

Проверяет:

- что пришел логин;
- что пришел пароль.

### `clientSchema`

Это центральная схема backend.

Она валидирует:

- ФИО;
- телефон;
- канал связи;
- заметки;
- статус;
- вложенный `searchProfile`.

Во вложенном профиле проверяются:

- тип недвижимости;
- бюджеты;
- диапазоны комнат;
- площади;
- районы;
- год сдачи;
- отделка;
- параметры дома;
- коммуникации.

Смысл файла:

- backend никогда не пишет в БД "сырые" данные без схемы.

## 14.6 `apps/api/src/services/serializers.ts`

Роль файла: привести ответ API к JSON-дружелюбному виду.

Что делает:

- рекурсивно проходит по объекту;
- если встречает `Date`, превращает ее в ISO-строку;
- если встречает массив, сериализует каждый элемент;
- если встречает вложенный объект, обходит и его.

Почему это нужно:

- PostgreSQL и JS могут возвращать `Date`;
- frontend ожидает JSON, а не объекты Date.

## 14.7 `apps/api/src/services/messageService.ts`

Роль файла: генерация текста подборки.

Логика:

- принимает клиента и shortlist;
- начинает сообщение с обращения к клиенту;
- дальше по каждому объекту собирает компактное описание;
- для квартиры и дома формат чуть отличается;
- в описание попадают цена, площадь, район, источник, ссылка;
- в конце получается текст, который можно копировать в мессенджер.

Зачем отдельный сервис:

- чтобы формат сообщений жил в одном месте;
- чтобы этот код не размазывать по route-handler.

## 14.8 `apps/api/src/services/searchService.ts`

Это один из самых важных файлов backend.

### Общая роль

Связывает карточку клиента, Python search-service и PostgreSQL.

### Что происходит сверху вниз

- импортируются helpers для БД;
- импортируется `config`;
- формируется payload для Python-сервиса.

### Формирование payload

Из `client_search_profiles` берутся поля, понятные `SearchRequest`:

- `clientId`
- `propertyType`
- бюджеты
- площади
- комнаты
- районы
- finishing
- параметры дома

### Запуск поиска

- создается запись `search_run` со статусом `searching`;
- клиенту тоже ставится статус `searching`;
- создается `AbortController` с таймаутом около 20 секунд;
- выполняется `fetch`/HTTP POST в `search-service`.

### После ответа

Если ответ успешен:

- для каждого объекта выполняется `upsertProperty`;
- затем `upsertClientFoundProperty`;
- считается количество результатов;
- `search_run` закрывается со статусом успеха;
- клиенту ставится:
  - `found`, если что-то найдено;
  - `no_results`, если ничего нет.

Если ответ неуспешен:

- `search_run` помечается ошибкой;
- клиент получает статус `error`.

Смысл файла:

- это оркестратор поиска;
- здесь backend перестает быть просто CRUD и начинает управлять бизнес-процессом.

## 14.9 `apps/api/src/repositories/sql.ts`

Это центральный файл backend. Здесь живет почти вся SQL-логика.

### Верх файла

- импортируется `bcryptjs` и `pool`;
- объявляется массив `profileColumns`.

`profileColumns` нужен затем, чтобы:

- в одном месте хранить список колонок профиля;
- одинаково собирать INSERT и UPDATE;
- не дублировать названия полей вручную много раз.

### `camelize()`

Функция:

- берет объект из PostgreSQL;
- преобразует `snake_case` в `camelCase`;
- делает это рекурсивно.

Зачем:

- frontend и TS-код работают в `camelCase`;
- SQL-таблицы живут в `snake_case`.

### `withClient()`

Это helper для транзакций.

Что делает:

- берет client из пула;
- запускает callback;
- в `finally` освобождает соединение.

### Пользователи

#### `ensureSeedUser(login, password)`

- ищет пользователя по логину;
- если уже есть, ничего не делает;
- если нет, хэширует пароль через `bcrypt.hash`;
- создает запись в `users`.

#### `findUserByLogin`

- делает SELECT по логину.

#### `findUserById`

- делает SELECT по id.

### Доступ к клиентам

#### `getClientAccess`

- проверяет принадлежность клиента конкретному пользователю;
- используется как базовый уровень безопасности.

### Работа со списком клиентов

#### `listClients`

SQL здесь:

- берет клиентов текущего пользователя;
- джойнит профиль поиска;
- подсчитывает найденные объекты;
- подсчитывает shortlist;
- применяет фильтры по строке поиска и статусу;
- сортирует по `updated_at desc`.

На выходе frontend получает готовую таблицу.

### Работа с профилем поиска

#### `profileValues()`

- берет JS-объект профиля;
- в порядке `profileColumns` собирает массив значений;
- нужен для параметризованных SQL-запросов.

### Создание клиента

#### `createClientWithProfile()`

Логика:

- открывает транзакцию;
- вставляет запись в `clients`;
- вставляет запись в `client_search_profiles`;
- коммитит;
- возвращает готовый объект.

### Обновление клиента

#### `updateClientWithProfile()`

Логика:

- в транзакции обновляет `clients`;
- затем обновляет или upsert-ит профиль поиска;
- обновляет `updated_at`.

### Удаление клиента

#### `deleteClient()`

- удаляет клиента;
- каскадные зависимости БД чистят связанные записи.

### Детали клиента

#### `getClientDetails()`

Функция:

- загружает саму карточку клиента;
- загружает профиль поиска;
- загружает найденные объекты;
- загружает shortlist;
- загружает последние `search_runs`.

Именно эта функция кормит страницу клиента почти целиком.

### Найденные объекты

#### `getFoundProperties()`

- получает найденные для клиента объекты;
- подтягивает данные из `properties`;
- сортирует по `match_score`.

#### `getFoundProperty()`

- получает один конкретный объект для клиента.

### Подборка

#### `getShortlist()`

- выбирает shortlist клиента;
- джойнит `properties`.

#### `addToShortlist()`

- вставляет пару `client_id` + `property_id`;
- при дубликате не падает;
- меняет статус клиента на `shortlist_ready`.

#### `removeFromShortlist()`

- удаляет объект из shortlist;
- если shortlist стал пустым, статус клиента пересчитывается:
  - `found`, если найденные объекты еще есть;
  - `no_results`, если найденных вообще нет.

### Сообщения и отправка

#### `createShareMessage()`

- сохраняет текст сообщения в `share_messages`.

#### `markSent()`

- в транзакции проставляет `sent_at`;
- обновляет статус клиента на `sent`.

### Dashboard

#### `dashboardSummary()`

Возвращает агрегаты:

- сколько клиентов в работе;
- сколько найдено объектов;
- сколько объектов в подборках;
- сколько подборок готово к отправке.

#### `dashboardActivity()`

Это более сложный SQL.

Он:

- генерирует последние 7 дней через `generate_series`;
- считает накопительные/дневные метрики;
- отдает набор точек для графика.

#### `attentionClients()`

- возвращает клиентов, которым стоит уделить внимание.

### Работа с объектами

#### `getPropertyById()`

- достает объект по id.

#### `getClientForSearch()`

- берет клиента вместе с профилем для повторного поиска.

### Журнал поиска

#### `createSearchRun()`

- создает запись о старте поиска.

#### `updateClientStatus()`

- меняет статус клиента.

#### `upsertProperty()`

Одна из крупнейших функций.

Она:

- ищет объект по `external_id` или `source_url`;
- если объект уже есть, обновляет его поля;
- если объекта нет, вставляет новый;
- сохраняет все нормализованные характеристики и `raw_data`.

#### `upsertClientFoundProperty()`

- связывает клиента и объект;
- хранит score и причины совпадения;
- связывает запись с конкретным `search_run`.

#### `finishSearchRun()`

- пишет статус успеха;
- сохраняет `results_count`;
- проставляет `finished_at`.

#### `failSearchRun()`

- пишет статус ошибки;
- сохраняет текст ошибки;
- проставляет `finished_at`.

Итог по файлу:

- это не просто репозиторий;
- это фактически основной data layer всей системы.

## 14.15 `apps/api/src/server.ts`

Это один из самых насыщенных методов.

- объединяет текст;
- проверяет, что объект относится к Краснодару;
- извлекает:
  - площадь;
  - комнатность;
  - площадь участка;
  - ЖК;
  - район;
  - адрес;
  - поселок;
- нормализует title;
- нормализует картинки;
- нормализует description;
- собирает `PropertyItem`.

### `_parse_visible_cards()`

Этот путь нужен на случай, когда у сайта нет адекватного JSON-LD.

Метод:

- берет все ссылки `a[href]`;
- поднимается на несколько родителей вверх, чтобы найти "карточку";
- собирает текст карточки;
- требует минимальную длину текста и наличие ценового маркера;
- извлекает href, title, image, description, price, area, rooms;
- снова проверяет Краснодар;
- нормализует title и description;
- создает `PropertyItem`.

### `_normalize_image_url()`

- склеивает относительный URL с базовым;
- отсекает `svg`;
- отсекает `logo`, `icon`, `placeholder`, `telegram`, `vk`, `whatsapp`;
- принимает только `.jpg`, `.jpeg`, `.png`, `.webp`.

### `_normalize_title()`

Очень важная бизнес-логика UI-подготовки.

Для квартиры:

- если исходный title плохой, он игнорируется;
- строится короткий title вида:
  - `Студия, 28 м²`
  - `1-к квартира, 40.6 м²`
  - `2-к квартира, 65 м²`

Для дома:

- строится `Дом 128 м²`.

### `_normalize_description()`

Если удалось очистить исходное описание, используется оно.  
Если нет - делается fallback.

Для дома fallback такой:

- title;
- участок;
- цена;
- источник.

Для квартиры fallback такой:

- title;
- локация;
- площадь;
- комнатность;
- цена;
- источник.

### `_clean_description()`

Отбрасывает описание, если:

- там служебные слова;
- там телефоны;
- там слишком много цифрового шума;
- там короткие бессмысленные куски.

Потом:

- чистит пробелы;
- приводит символы к нормальному виду;
- режет на предложения;
- оставляет только достаточно длинные и внятные предложения;
- ограничивает длину.

- `_external_id()` - создает стабильный внешний id через SHA1 от URL.

Итог по файлу:

- это основная интеллектуальная очистка объявлений;

## 16.4 `apps/search-service/app/adapters/sources.py`

Роль файла: реестр источников.

Что внутри:

- список `APARTMENT_SOURCES`;
- список `HOUSE_SOURCES`;
- несколько пустых классов-оберток адаптеров;

Почему классы почти пустые:

### `build_adapters(property_type)`

## 16.5 `apps/search-service/app/services/matcher.py`

### `score_property()`

Для квартиры:

## 16.6 `apps/search-service/app/services/search.py`

Это оркестратор Python-поиска.

### `_passes_required_filters()`

Дополнительный обязательный фильтр:

- объект должен иметь цену;
- объект должен вписываться в бюджет;
- допускается небольшой люфт около 8% сверху.

### `run_search()`

Логика:

- создается `httpx.AsyncClient`;
- через `build_adapters()` создается набор источников;
- все поиски запускаются параллельно;
- исключения отдельных адаптеров не валят весь поиск;
- результаты собираются в один массив;
- удаляются дубликаты по `sourceUrl`;
- каждому объекту считается score;
- объекты сортируются по убыванию score;
- берутся первые 80.

Итог:

- сервис возвращает компактную ранжированную выдачу.

## 18. Почему данные не пропадают после перезапуска

Потому что PostgreSQL в Docker Compose использует volume:

- `estateflow-postgres`

Это значит:

- контейнер можно остановить;
- контейнер можно пересоздать;
- данные останутся в volume.

Потеря данных произойдет только если:

- вручную удалить volume;
- выполнить `npm run prod:reset-db`;
- удалить Docker data.

Короткая правильная формулировка:

> Система построена по трехзвенной схеме. Клиентская часть реализована на React и отвечает за интерфейс риелтора.

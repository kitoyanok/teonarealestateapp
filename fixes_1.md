# EstateFlow / Тэона — файл правок для Codex

Этот файл описывает точечные правки текущей версии интерфейса и логики. Его нужно положить в корень проекта, например как `fixes_for_codex.md`, и использовать вместе с уже существующим `redesign.md`.

Визуальные референсы лежат в корне проекта в папке `views`. Codex должен ориентироваться на эти изображения:

- `views/logo.png` — новый логотип приложения;
- `views/photo_1` — изображение для страницы входа;
- остальные скриншоты в `views` — UI-референсы для светлой темы, карточек, таблиц, меню, модальных окон и страницы клиента.

Перед внесением изменений Codex должен просмотреть структуру проекта, найти текущие компоненты layout, sidebar, topbar, страницы входа, клиентов, карточки клиента, карточки объекта и модальные окна.

Codex должен использовать свои доступные команды, плагины и скиллы через `$` и `@`, если они доступны в среде разработки. Перед изменениями нужно сделать быстрый анализ проекта, после изменений запустить проверки, линтеры и локальную сборку.

## 1. Главная проблема: фиксированный layout

Сейчас скроллится всё окно браузера, из-за этого боковое меню и верхняя шапка уезжают вместе с контентом. Нужно переделать layout так, чтобы:

- левое меню всегда было закреплено на экране;
- верхняя шапка всегда была закреплена на экране;
- скроллился только основной контент страницы;
- внутри крупных блоков, например таблицы клиентов или списка найденных объектов, скроллился сам блок, если данных много;
- на мобильных устройствах layout должен оставаться удобным и не ломаться.

## 2. Валидация и маска телефона

Нужно добавить строгую проверку номера телефона при создании и редактировании клиента.

### Требования

Номер должен начинаться только с:

- `+7`
- `8`

Поддерживаемые форматы ввода:

```text
+7 (999) 999-99-99
8 (928) 999-99-99
+79999999999
89999999999
```

В интерфейсе желательно автоматически приводить номер к читаемой маске:

```text
+7 (999) 999-99-99
```

или если пользователь начал с `8`:

```text
8 (928) 999-99-99
```

### Правила валидации

1. Удалить из значения все символы, кроме цифр и плюса.
2. Если номер начинается с `+7`, после `+7` должно быть ровно 10 цифр.
3. Если номер начинается с `8`, после `8` должно быть ровно 10 цифр.
4. Номера с другим кодом страны отклонять.
5. Пустой телефон запрещен.
6. Ошибка должна показываться под полем.

Текст ошибки:

```text
Введите номер в формате +7 (999) 999-99-99 или 8 (999) 999-99-99.
```

### Где проверить

Валидация должна быть и на frontend, и на backend.

Frontend нужен для удобства пользователя.
Backend нужен, чтобы в базу не попадали некорректные данные.

### Кнопка сохранения

Кнопка `Создать и найти объекты` должна быть disabled, если:

- не заполнено имя клиента;
- телефон невалидный;
- не выбран тип недвижимости;
- не указан бюджет;
- для квартиры не выбрана комнатность или площадь;
- для дома не указана площадь дома или участок.

## 4. Account popup по клику на фото профиля

Сейчас в верхней панели есть фото аккаунта. Нужно сделать выпадающее меню при клике на фото.

### Поведение

- клик по фото открывает popup;
- повторный клик закрывает popup;
- клик вне popup закрывает popup;
- Escape закрывает popup;
- popup должен быть поверх контента;
- на мобильной версии popup может открываться как bottom sheet.

### Содержимое popup

```text
[Фото]
Риелтор
realtor@example.com

Профиль
Помощь
Настройки

Выйти
```

### Пункты

#### Профиль

Открывает страницу `/profile` или модальное окно профиля.

Страница профиля должна содержать:

```text
Профиль
Данные риелтора

Имя
Телефон
Email
Фото профиля

[Сохранить]
```

Для MVP можно оставить простую форму без загрузки фото, но пункт должен быть.

#### Помощь

Открывает страницу `/help`.

Страница помощи должна быть коротким гайдом для риелтора.

Текст страницы:

```text
Помощь
Как работать с сервисом

1. Добавьте клиента и укажите параметры поиска.
2. Выберите тип недвижимости: квартира или дом.
3. Дождитесь, пока система найдет объекты.
4. Откройте карточку клиента и просмотрите найденные варианты.
5. Добавьте подходящие объекты в подборку.
6. Нажмите «Отправить подборку».
7. Скопируйте номер и текст сообщения.
8. Отправьте подборку клиенту вручную в удобном мессенджере.

Важно:
Сервис работает в live-режиме и не подмешивает демо-объявления.
Если источник не передал фото или цену, такие данные могут отсутствовать.
```

#### Настройки

Открывает текущую страницу настроек.

#### Выйти

Вызывает logout и возвращает пользователя на страницу входа.

### Что убрать из popup

Не нужно добавлять пункт `Клиенты`, потому что он уже есть в sidebar.
Вместо него должны быть `Профиль`, `Помощь`, `Настройки`, `Выйти`.

## 12. Поиск и фотографии объектов

Очень важная правка: сейчас в карточки иногда попадают логотипы, иконки, favicon или случайные изображения сайта. Так делать нельзя.

Нужно доработать парсер и нормализацию изображений.

### Запрещено использовать как фото объекта

- favicon сайта;
- логотип застройщика;
- логотип сайта;
- sprite/icons;
- изображения с маленьким разрешением;
- svg-иконки;
- картинки кнопок, соцсетей, меню;
- изображения, в URL которых есть признаки `logo`, `favicon`, `icon`, `sprite`, `placeholder`, `avatar`, `brand`, `social`.

### Фильтр изображений

При выборе изображения объекта нужно проверять:

1. URL изображения есть.
2. URL абсолютный или корректно приводится к абсолютному.
3. Расширение или Content-Type похожи на изображение: jpg, jpeg, png, webp.
4. Не SVG.
5. URL не содержит запрещенных слов:

```text
logo
favicon
icon
sprite
placeholder
avatar
brand
social
```

6. Если можно получить размеры изображения, брать только изображения примерно от:

```text
min-width: 300px
min-height: 180px
```

Если размеры получить нельзя, использовать эвристику по URL и месту в DOM.

### Приоритеты выбора фото

1. Изображение внутри карточки объекта.
2. Изображение рядом с названием ЖК/объекта.
3. Изображение из галереи объекта.
4. OpenGraph image только если оно не логотип и похоже на объект.
5. Если ничего нет — локальная заглушка.

### Заглушка

Если нормального фото нет:

```text
Фото не найдено
```

Не использовать чужие логотипы как заглушку.

- повторяющиеся слова;
- меню сайта;
- тексты `Скоро в продаже`, `На карте`, `По алфавиту`, если они не относятся к конкретному объекту;
- огромные полотна текста без структуры.

### Правило формирования описания

Описание должно собираться из нормализованных данных объекта.

Для квартиры:

```text
Квартира в жилом комплексе «Название». Площадь — 40.2 м², комнатность — 1-комн., срок сдачи — 2026. Цена — 4 639 080 ₽. Источник — Домострой Краснодар.
```

Для дома:

```text
Дом площадью 128 м² в КП Мелодия. Участок — 5 сот., коммуникации: газ, электричество, вода. Цена — 9 350 000 ₽. Источник — КП Краснодар.
```

Если каких-то данных нет, не писать странные фразы. Писать только то, что действительно известно.

### Дополнительное исходное описание

Если сайт дает нормальное описание объекта, можно добавить его ниже, но предварительно очистить:

- убрать лишние пробелы;
- убрать повторяющиеся слова;
- ограничить длину до 500–700 символов;
- убрать навигационный мусор.

max-width: 1000px

## 17. Страница клиентов

Текущий общий стиль хороший, но нужно обеспечить правильный скролл внутри таблицы.

### Структура

```text
Все клиенты
Все заявки и параметры поиска в одном списке                     [Все статусы]

Клиент | Тип | Телефон | Запрос | Найдено | В подборке | Статус | Действие
```

### Поведение

- topbar и sidebar не двигаются;
- если клиентов много, скроллится таблица;
- шапка таблицы должна оставаться видимой, если это несложно реализовать;
- кнопка `Открыть` всегда справа;
- строка также может открывать клиента по клику.

## 18. Главная страница

Главная должна оставаться в светлой дизайн-системе и иметь график риелтора.

### Верхние KPI

```text
12  Клиенты в работе
107 Найдено объектов
6   В подборках
2   Готово к отправке / копированию
```

Использовать оранжевые мини-графики как в референсе.

### График риелтора

Блок:

```text
Активность за 7 дней
С кем продолжить работу
```

Если данных нет, не оставлять огромную пустую область. Сделать аккуратный empty-state:

```text
Пока нет истории активности
После создания клиентов и запуска поиска здесь появится график найденных объектов.
```

Если данные есть, отрисовать простой график:

- найдено объектов по дням;
- добавлено в подборки по дням.

Можно использовать canvas/svg/простые div-bar. Не обязательно подключать тяжелую библиотеку.

### Правая карточка

```text
Клиенты требуют внимания
С кем продолжить работу
```

Таблица или компактный список.

## 19. Статусы

Использовать такие статусы:

```text
В работе
Готово
Нет объектов
Отправлено
```

Цвета:

```text
В работе: #FFF8C7 / текст #9A8700
Готово: #CFFAD6 / текст #3C9A4B
Нет объектов: #FFD1D1 / текст #B94A4A
Отправлено: #CFFAD6 / текст #3C9A4B
```

Не использовать слишком яркие цвета для бейджей, стиль должен быть мягкий.

## 20. Адаптив

### Desktop от 1200px

- sidebar фиксированный слева;
- topbar фиксированный сверху;
- контент скроллится внутри;
- карточки объектов 3–4 в ряд;
- подборка справа.

### Tablet 768–1199px

- sidebar можно сделать уже;
- текст логотипа можно скрыть, оставить только логотип;
- карточки объектов 2 в ряд;
- подборка уходит под найденные объекты;
- topbar сохраняется.

### Mobile до 767px

- sidebar скрыть или заменить нижним меню;
- topbar фиксирован сверху;
- кнопка добавления клиента доступна в topbar или floating button;
- таблицы клиентов превратить в карточки;
- карточки объектов 1 в ряд;
- модалка объекта открывается почти на весь экран;
- account popup открывается как bottom sheet.

## 22. Ручные сценарии проверки

После исправлений проверить вручную:

1. Вход на страницу логина.
2. На странице логина виден `views/logo.png`.
3. Справа на странице логина есть фото `views/photo_1` за формой.
4. Логин `test / test` работает.
5. Sidebar не скроллится вместе с контентом.
6. Topbar не скроллится вместе с контентом.
7. На странице клиентов скроллится только таблица/контент.
8. Создание клиента не пропускает неправильный телефон.
9. Номер `+7 (999) 999-99-99` принимается.
10. Номер `8 (928) 999-99-99` принимается.
11. Номер с другим кодом отклоняется.
12. Фото профиля открывает account popup.
13. В popup есть `Профиль`, `Помощь`, `Настройки`, `Выйти`.
14. Карточки объектов выглядят как в референсе.
15. Кнопка `В подборку` не пустая.
16. В карточках не используются логотипы или favicon как фото объекта.
17. Названия объектов не состоят из слов `Кабинет`, `Избранное`, `Отзывы`, `Наверх`.
18. Описание объекта информативное, а не набор SEO-текста.
19. Модальное окно объекта открывается и не ломает страницу.
20. Добавление объекта в подборку работает.
21. Отправка подборки открывает модальное окно копирования.
22. Копирование телефона работает.
23. Копирование текста подборки работает.
24. На мобильной ширине интерфейс не ломается.

## 23. Definition of Done

Правка считается выполненной, если:

- sidebar и topbar фиксированы;
- body не скроллится целиком;
- основной контент скроллится внутри рабочей области;
- телефон валидируется и форматируется;
- используется `views/logo.png`;
- на странице входа используется `views/photo_1`;
- account popup работает;
- страницы `Профиль` и `Помощь` доступны;
- карточки объектов приведены к новому виду;
- странная широкая статистика на странице клиента убрана или заменена компактным блоком;
- изображения объектов фильтруются, логотипы не попадают в карточки;
- названия объектов нормализованы;
- описания объектов очищены и стали информативными;
- модалка объекта соответствует светлой дизайн-системе;
- интерфейс адаптирован под desktop, tablet и mobile;
- проект запускается на текущем порту приложения;
- нет критических ошибок в консоли браузера и терминале.

# EstateFlow Redesign Guide

Файл предназначен для разработчика / Codex-агента. Его нужно положить в корень проекта как `redesign.md`.

Цель редизайна — полностью перевести текущий темный интерфейс EstateFlow на светлую минималистичную дизайн-систему из референсов. Интерфейс должен быть лаконичным, чистым, с большим количеством воздуха, тонкими границами, мягкими скруглениями и ярким оранжевым акцентом.

## 1. Важное указание для Codex / AI-агента

Перед началом работы изучи этот файл полностью и посмотри изображения-референсы в папке со скриншотами в корне проекта.

Ожидаемые файлы-референсы:

```text
/screenshots/главная.png
/screenshots/клиенты.png
/screenshots/карточка клиента.png
/screenshots/отправить подборку.png
```

Если папка называется иначе, найди изображения по названию. Нужно использовать их как визуальный эталон.

Используй доступные команды, плагины и скиллы через синтаксис `$` и `@`, если они доступны в твоей среде. Например:

```text
$ npm install
$ npm run dev
$ npm run build
$ npm run lint
$ npm run test
$ psql
$ docker compose up
```

И плагины / скиллы:

```text
@workspace
@terminal
@browser
@figma
@docs
@skills
```

Не придумывай несуществующие команды проекта. Сначала посмотри `package.json`, структуру папок, `.env.example`, README и текущие компоненты.

Перед изменениями:

```text
1. Найди текущие страницы и компоненты.
2. Найди глобальные стили.
3. Найди layout/sidebar/header.
4. Найди компоненты карточек, таблиц, модалок и форм.
5. Сохрани рабочий функционал поиска недвижимости, клиентов и подборок.
```

Запрещено:

```text
- ломать текущий пользовательский флоу;
- удалять рабочий поиск недвижимости;
- возвращать темную тему;
- добавлять лишние страницы "Источники", "Сообщения", "Подборки";
- добавлять автоматическую отправку сообщений;
- использовать Prisma/in-memory store, если проект переводится на упрощенный стек;
- делать свободный ввод там, где нужны фиксированные варианты;
- оставлять элементы темной дизайн-системы.
```

Нужно:

```text
- переписать UI под светлую дизайн-систему;
- сохранить основной функционал;
- сделать аккуратный адаптив;
- привести формы к выпадающим спискам, chips, segmented controls;
- убрать поле "канал отправки";
- оставить только телефон клиента и копирование;
- сделать окно подготовки сообщения с копированием текста и телефона;
- добавить / улучшить окно просмотра объявления;
- добавить полноценный график активности на главной или качественную заглушку;
- сделать все тексты короткими, понятными и в едином стиле.
```

## 2. Финальный пользовательский флоу

```text
Риелтор входит в систему
↓
Открывает главную
↓
Видит KPI, график и клиентов, требующих внимания
↓
Добавляет клиента
↓
Выбирает тип недвижимости: квартира или дом
↓
Заполняет параметры поиска через списки, chips и select
↓
Система ищет объекты под параметры клиента
↓
Открывается карточка клиента
↓
Риелтор смотрит найденные объекты
↓
Открывает объявление в модальном окне / drawer
↓
Добавляет подходящие объекты в подборку
↓
Открывает окно "Отправка подборки"
↓
Копирует телефон клиента и текст подборки
↓
Отправляет вручную в любом мессенджере
```

Приложение не отправляет сообщения само. Оно только готовит текст и дает скопировать телефон.

## 3. Стек проекта после упрощения

Целевой стек:

```text
Frontend: React
Сборка frontend: Vite допустим, если уже используется
Backend: Node.js
HTTP API: Express допустим, если уже используется
Search service: Python
Database: PostgreSQL
UI: CSS / CSS Modules / обычные компоненты без тяжелой UI-библиотеки
```

Важно: Vite и Express не нужно выпячивать в дипломе. Их можно оставить как технические инструменты, если они уже есть и не мешают.

В дипломной документации формулировать проще:

```text
React отвечает за интерфейс.
Node.js отвечает за API и работу с базой.
Python отвечает за поиск и парсинг недвижимости.
PostgreSQL хранит клиентов, параметры поиска, найденные объекты и подборки.
```

Что убрать из архитектуры, если сейчас присутствует:

```text
- Prisma;
- in-memory store;
- demo-хранилище в памяти;
- лишние ORM-слои;
- канал отправки;
- автоматическую отправку сообщений.
```

## 4. Общий визуальный стиль

Новый стиль — светлый, чистый, почти монохромный, с оранжевым акцентом.

Ключевые признаки:

```text
- светлый фон приложения;
- белые карточки;
- тонкие серые границы;
- оранжевые кнопки и progress-bar;
- мягкие скругления;
- минимум декоративных эффектов;
- таблицы с легкими разделителями;
- компактный sidebar;
- topbar с поиском;
- тексты короткие и спокойные;
- интерфейс похож на CRM / SaaS dashboard.
```

Не использовать:

```text
- темный фон;
- зеленый акцент;
- тяжелые тени;
- стеклянные карточки;
- neon glow;
- крупные черные CTA;
- перегруженные графики.
```

## 5. Design Tokens

### 5.1. Цвета

Основные цвета из референсов:

```css
:root {
  --color-text: #3D3D3D;
  --color-muted: #898989;
  --color-accent: #FD6000;
  --color-bg: #FAFAFA;
  --color-surface-soft: #F1F1F1;
  --color-white: #FFFFFF;

```

Дополнительные цвета, которые нужно добавить для полноценного UI:

```css
:root {
  --color-border: #E7E7E7;
  --color-border-strong: #DCDCDC;

  --color-hover: #F6F6F6;
  --color-input: #FAFAFA;

  --color-accent-hover: #E95700;
  --color-accent-soft: #FFF1E8;
  --color-accent-border: #FFD2B8;

  --color-success-bg: #CFF8D2;
  --color-success-text: #3C9F4B;

  --color-warning-bg: #FFF7C9;
  --color-warning-text: #A99524;

  --color-danger-bg: #FFD4D4;
  --color-danger-text: #C45A5A;

  --color-overlay: rgba(0, 0, 0, 0.22);
}
```

### 5.2. Шрифт

Использовать только Inter.

```css
font-family: Inter, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif;
```

Основные размеры:

```text
10px regular — подписи, мелкие описания, subtitle в карточках.
12px regular — table header, маленькие бейджи, метаданные.
14px regular — основной текст интерфейса, строки таблиц, кнопки.
16px medium — заголовки карточек, важные значения.
20px medium/semibold — заголовки страниц при необходимости.
48px regular/medium — крупные KPI-цифры на главной.
```

Начертания:

```text
400 Regular — основной текст.
500 Medium — кнопки, nav active, table values.
600 SemiBold — заголовки карточек и модалок.
```

Не использовать слишком жирный текст везде. В референсе интерфейс легкий.

### 5.3. Радиусы

```text
Sidebar logo circle: 36px круг.
Nav item: 9-10px.
Cards: 8-10px.
Buttons: 9-12px.
Inputs: 9-10px.
Badges: 8-10px / pill.
Property cards: 8-10px.
Modal: 12px.
```

### 5.4. Отступы

```text
Sidebar width: 230px.
Topbar height: 58px.
Page horizontal padding: 10px от sidebar boundary до контента, дальше card-grid.
Main content padding: 10px 10px 12px 10px.
Card padding: 14px 15px.

Table row height: 56px.

```

### 5.5. Границы и тени

Основной стиль — границы, не тени.

```css
border: 1px solid var(--color-border);
box-shadow: none;
```

Легкая тень допустима только для выпадающих списков и модалок:

```css
box-shadow: 0 12px 40px rgba(0, 0, 0, 0.08);
```

## 7. Страница "Главная"

Референс: `главная.png`.

### 7.1. Цель страницы

Главная — рабочая сводка риелтора. Она должна показывать:

```text

- график активности риелтора;
- клиентов, с которыми нужно продолжить работу;
- последних клиентов.
```


### 7.3. KPI-карточки

Высота: `158px`.

4 карточки в ряд.

Фон: `#FFFFFF`.

Граница: `#E7E7E7`.

Радиус: `8px`.

Внутри:

```text
Крупное число слева сверху.
Маленький тренд справа сверху.
Мини-график / bar chart внутри.
Название KPI снизу.
```

Карточка 1:

```text
12
↗ +8.4% активные заявки
Клиенты в работе
```

Карточка 2:

```text
107
↘ -3.7% по текущим клиентам
Найдено объектов
```

Карточка 3:

```text
6
↗ +8.4% отобрано вручную
В подборках
```

Карточка 4:

```text
2
↗ +8.4% подборки можно отправить
Готово к отправке
```

Можно оставить текст "Готово к отправке", но в логике приложения понимать, что это ручная отправка через копирование.

Для мини-графиков:

```text
- bar mini-chart оранжевый для карточек 1 и 4;
- line mini-chart розовый/оранжевый для найденных объектов;
- маленький bar chart с одним оранжевым столбиком для подборок.
```

Если нет библиотеки графиков, сделать простые div-бары.

Цвета мини-графиков:

```text
Оранжевый: #FD6000
Светло-серый: #F1F1F1
Розовый line допустим: #FF1F70, но лучше использовать #FD6000 для единства.
Зеленый тренд: #2AA86B
Розовый отрицательный тренд: #FF2F7D
```

## 20. Definition of Done

Работа считается завершенной, если:

```text
1. Весь интерфейс переведен на светлую тему.
2. Цветовой акцент везде #FD6000.
3. Sidebar, topbar, таблицы, карточки и модалки соответствуют референсам.
4. На всех страницах используется Inter.
5. Темные элементы старого UI удалены.
6. Форма клиента использует select/chips/segmented control.
7. Поле "канал отправки" удалено.
8. Остался только телефон клиента.
9. Есть кнопка "Скопировать телефон".
10. Окно отправки подборки содержит телефон и textarea.
11. Есть drawer просмотра объявления.
12. Карточки объектов выглядят как в референсе.
13. Главная содержит KPI, график/заглушку, клиентов внимания и последних клиентов.
14. Страница клиентов содержит таблицу и фильтры.
15. Карточка клиента содержит инфо, подборку и найденные объекты.
16. Интерфейс адаптивен под desktop, tablet и mobile.
17. Проект запускается на localhost:5002.
18. `npm run build` проходит без ошибок.
19. Поиск недвижимости и подборки работают после редизайна.
20. README или docs обновлены при необходимости.
```

## 22. Примечание по диплому

В дипломной работе описывать приложение так:

```text
Тэона — веб-приложение для риелтора, которое помогает вести клиентов, фиксировать параметры поиска недвижимости, автоматически получать подходящие объекты из внешних источников и формировать текст подборки для ручной отправки клиенту.
```

Не писать, что приложение само отправляет в мессенджеры. Правильная формулировка:

```text
Система формирует готовый текст подборки и позволяет скопировать телефон клиента для ручной отправки.
```
